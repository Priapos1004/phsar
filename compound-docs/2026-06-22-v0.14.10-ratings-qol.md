# v0.14.10 — Ratings quality-of-life upgrades

**Date:** 2026-06-22 | **Branch:** v0.14.10-ratings-quality-of-life-upgrades

## Summary

Three independent ratings features: (1) a `WatchStatus` enum (`completed`/`on_hold`/`dropped`) replacing the binary `dropped` boolean, so a future "continue watching" surface can tell a paused show from an abandoned one; (2) rewatch tracking via an append-only `watch_events` log with a **derived** `watched_count`, never stored; (3) an admin-curated "story complete" flag on an `anime_completion` 1:1 sidecar, surfaced as `is_finished` badges on the anime page + search cards and managed from a new admin Completion tab.

## Key Decisions

- **`watched_count` is derived (`COUNT(events)`), never denormalized onto `Ratings`.** Watch events key to `(user_id, media_id)`, NOT to a rating row, so the history survives a rating delete + re-add. The whole point is a clean time series for future watch-behaviour forecasting — a stored counter would drift on every edit/delete path.
- **Watch history delete is opt-in, and the trigger is server-derived.** `delete_watch_history` defaults False on `upsert`/`delete`/`bulk_delete`; history is kept by default so a delete-then-re-add (or a completed→on_hold downgrade) doesn't strand or pollute the series. On upsert, the flag only fires when the server sees a genuine `completed → non-completed` transition (`is_downgrade`), so a stale/forged flag can't wipe history on an upgrade or no-op edit.
- **First-watch event logs exactly once.** `_maybe_log_first_watch` writes an event only when a rating lands `completed` AND the user has no prior events for that media. So first completion and an on_hold/dropped→completed transition each log once, but re-rating media that already has history never duplicates. Rewatches go through the explicit `log_rewatch` path instead.
- **Story-complete is row-presence, not a boolean.** `anime_completion` row exists ⇒ finished; marking inserts, unmarking deletes. Keeps it a plain `SELECT *` with no "row exists but false" ambiguity, and keeps the admin flag off the canonical `anime` row so it can't leak into the anime Pydantic schemas — surfaced explicitly as `is_finished`. (Same sidecar convention as `anime_freshness`.) `marked_by_user_id` is `SET NULL` (not CASCADE) so the completion fact outlives the admin who set it.
- **Story-complete is manual — no detector, no dismissal history.** Unlike merge/split candidates, there's no signal to infer narrative conclusion from. The Completion tab reuses the public `/search/anime` to find anime to mark; there is deliberately **no** search method on `AnimeCompletionDAO`.

## Gotchas & learnings

- **"Finished Airing" ≠ story complete.** MAL's `airing_status` only means broadcast stopped; an announced sequel or ongoing manga adaptation is "Finished Airing" yet narratively unfinished. The whole feature exists to express that gap — hence the badge tooltip on the detail page spelling it out. No tooltip on search cards (hover popups while scrolling are intrusive).
- **`not_applicable` ending-quality is auto-set on non-completed ratings.** A new `isAttrRated()` frontend helper (value !== null && !== 'not_applicable') is the single source of truth for "count this attribute as rated" — used by the per-rating card, the distribution bars, and the overview gate so radar/bars/badges all agree and an on_hold rating can't trigger an empty Attribute Summary.
- **Migration backfill must run exactly once.** `c3e8a1f9b2d4` seeds one event per existing `completed` rating at its `created_at`. There's no `(user, media)` uniqueness guard (rewatches need multiple rows), so re-running just the data statement would silently double every `watched_count`. Safe on the normal single-upgrade path (table created empty in the same migration); flagged here because the corruption would be invisible (derived count, no constraint to catch it).
- **`watchstatus` enum created with `create_type=False` + explicit `.create(checkfirst=True)`** to avoid the double-create trap when `add_column` would otherwise auto-create the type. Downgrade drops the type so a re-upgrade doesn't collide.
- **Test-harness commits persist; global-list endpoints need membership assertions.** The router test harness binds the app session to a connection without savepoint-join, so API `commit()`s actually land in the dev DB. Most router tests don't notice because they scope queries to a fresh per-test user — but `list_finished` is a *global* query (admin sees every marked anime), so an exact-list assertion collided with genuine dev marks (AoT, DxD, …). Assert membership of the specific uuid (`uuid in [...]`, `.count(uuid) == 1`), never the whole list, for any global admin-list endpoint.

## Future work / technical debt

- **`ix_watch_events_watched_at` is for a reader that doesn't exist yet.** The time-series forecasting use is inherently per-user (`WHERE user_id ORDER BY watched_at`); when that lands, replace the standalone `watched_at` index with composite `(user_id, watched_at)`. Until then it's unused weight on every insert.
- `on_hold` is currently only set by new writes — unreachable from legacy data (migration maps old `dropped` to `dropped`/`completed` only). The "continue watching" surface that consumes it is not yet built.
