First user-facing ratings experience: media detail page with hero card, inline rating card, and related media carousel.

## Key Decisions

- **Inline RatingCard instead of modal** — The original plan (issue #16) called for a dialog/modal. An inline card proved better for the single-page context: no overlay z-index issues, form state is always visible, and the card doubles as the read-only rating display. Reusable on the future anime detail page without needing dialog infrastructure.
- **`PUT` upsert for ratings** — A single `PUT /ratings/media/{uuid}` creates or updates. Avoids the frontend needing to track whether a rating exists before choosing POST vs PUT. Matches the idempotent intent.
- **Query-param routing (`/media?uuid=`)** — Chose this over `/media/[uuid]` path params. Trade-off: bypasses SvelteKit's `load` functions but keeps the `q` search token as a sibling param for back-navigation. Worth revisiting before v0.11.0 anime pages establish a pattern.
- **Score step hardcoded to 0.5** — Will come from UserSettings in v0.12.0. `clampAndSnapScore(val, step)` already accepts step as a parameter, so the migration path is clean.

## Failed Approaches

- **`AnimeDAO.get_with_media_by_id`** — Initially built an anime-centric query that loaded all media with genres/studios. Abandoned because the page navigates by media UUID, not anime ID, making the extra lookup wasteful. The media-centric `MediaDAO.get_by_uuid_with_relations` was the right entry point. Dead method was removed during review.

## Gotchas

- **Restricted users get 403 on rating endpoints** — `require_user_or_admin` rejects restricted users. The media detail page must treat 403 the same as 404 on the rating fetch, otherwise the entire page breaks behind an error message even though the media loaded fine.
- **Rapid navigation race condition** — Clicking through the related media carousel quickly fires overlapping `loadMedia` calls. Without a request generation counter, stale responses can overwrite fresh data. Fixed with a simple `requestId` pattern.
- **`lazy="raise"` on sibling media** — Sibling `Media` objects loaded via `Anime.media` don't have genres/studios eagerly loaded. Accessing those relationships throws at runtime. The `MediaSibling` schema intentionally uses only direct columns to avoid this.

## Future Work

- Delete confirmation UX could evolve to an undo-toast pattern instead of two-click
- `media_to_dict` manual mapping could be replaced with `model_validate(from_attributes=True)` to reduce field sync burden
- Image-with-fallback pattern is repeated in 3 components — extraction candidate when a 4th appears
