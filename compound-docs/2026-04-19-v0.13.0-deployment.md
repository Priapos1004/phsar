# v0.13.0 — Deployment & Infrastructure

**Date:** 2026-04-19 | **Branch:** v0.13.0-deployment

## Summary

Shipped Phsar to a Coolify-managed VM: backend + frontend Dockerfiles (bun build → node runtime for FE, CPU-only torch + baked sentence-transformers for BE), tag-triggered GitHub Actions image builds to ghcr.io (the 2 vCPU / 4 GB VM can't survive a SvelteKit build), `/health` + version footer, npm → bun migration, and a full pg_dump/pg_restore admin system with maintenance-mode gate, content-hash dedupe, 14-daily + 8-Sunday retention, pre-restore auto-snapshot, and an off-host rsync script.

## Key Decisions

- **Build images in CI, not on the VM.** Vite's client-bundle pass peaks at 2–3 GB and OOMs bun after ~42 min of swap thrash on a 4 GB VM. Moved to `build-images.yml` on tag push (`v*`); Coolify just pulls.
- **Read frontend env vars at runtime via `$env/dynamic/public`.** One image targets any backend by swapping container env — no rebuild. Matches `VersionFooter`'s existing pattern.
- **`:latest` only on stable tags (no hyphen).** Preview tags like `v0.13.0-rc1` publish only their explicit tag so a preview build can't silently become prod on next redeploy.
- **Content-level backup dedupe via `pg_restore -f -` hash, not raw bytes.** `pg_dump -Fc` headers embed a creation timestamp and pg16+ randomizes `\restrict` tokens per run, so raw-byte hashing never dedupes. Stripping those lines before hashing makes download→upload round-trips idempotent.
- **Per-source dedupe policy.** Manual raises 409 (admin wants explicit "identical to X" feedback); cron + pre_restore silently return the existing dump so schedulers don't accumulate no-op snapshots. Implemented as `_finalize_partial_dump → (metadata, was_duplicate)` tuple so callers decide — flag-driven API was ambiguous under same-second timestamp collisions.
- **`.current_db.json` pointer follows DB state, not restore history.** Re-pointed whenever a create/upload dedupe-matches an existing dump, so the UI "Current" badge tracks reality even when the current DB came from a recent re-confirmation, not the last restore.
- **Prime connection pool BEFORE lifting maintenance gate after restore.** Warm-up-in-`finally`-after-`set_maintenance(False)` defeats its own purpose — requests are already unblocked when warm-up hits a hang. Warm-up now runs first, bounded with 5s `asyncio.wait_for` so a broken DB can't hang the gate lift.
- **`-manual` source suffix on manual dumps.** Filename was ambiguous when a manual backup's label collided with another source token (`phsar-<ts>-cron.dump` for a manual labeled "cron" vs a cron dump). Explicit `-manual` segment + order-sensitive `_guess_source_from_name` lookup fixes it without a filename-compat break.
- **Per-user try/commit in spoiler-cache recompute paths.** Both `save_service._refresh_spoiler_cache_for_all_users` (post-save trigger) and `backfill_spoiler_visibility` (startup mop-up) now isolate each user — one poisoned row doesn't abort the rest, and the mop-up path must itself be resilient since save-service relies on it for failures.

## Failed Approaches

- **`$SOURCE_COMMIT` as `APP_VERSION` in Coolify.** Disappeared once builds moved to CI. Replaced with `${{ github.ref_name }}` → Dockerfile ARG → runtime ENV.
- **Relying on Tailwind v4's `@property initial-value` for the default theme.** Dev worked; production CSS pipeline silently dropped it. Default theme rendered transparent buttons on white cards. Fix: explicit `:root { --primary: ...; --ring: ...; }` block alongside `@property`.
- **Preview deploys via force-moving a tag.** Rejected — cut `-rc2`, `-rc3` instead of rewriting `-rc1`.
- **`raise_on_duplicate` boolean flag on `_finalize_partial_dump`.** Cleaner-looking filename-compare variant broke when two calls share a same-second timestamp (filenames equal, check missed the dedupe). Reverted to explicit `(metadata, was_duplicate)` tuple return.

## Gotchas & Learnings

- **Tailwind v4 production build does not reliably honor `@property initial-value`.** `getComputedStyle(html).getPropertyValue('--primary')` returned empty string in prod despite the `@property` declaration. Any future Tailwind upgrade needs a prod smoke-test on both themed and un-themed states.
- **Frontend hardcoded `http://localhost:8000` in `config.ts`** despite CLAUDE.md claiming the URL was baked in at build time. Never verified. Prod bundles shipped with localhost. Lesson: when docs say "baked in," grep for the reader.
- **`.gitignore: **/versions/*` silently excluded all Alembic migrations.** Prod deploy fell over on `relation "genre" does not exist` because the image had no migration files.
- **ghcr.io repo paths must be lowercase.** `github.repository_owner` returns the user's actual casing. Normalize with `${GITHUB_REPOSITORY_OWNER,,}`.
- **Coolify build secrets use `--mount=type=secret,env=VAR`** — Dockerfile syntax 1.10+. Pinned `1.7` failed with `unexpected key 'env' in 'env=COOLIFY_URL'`. Bumped to `# syntax=docker/dockerfile:1`.
- **`bun install` resolves transitive deps differently from npm.** Swapping lockfiles bumped bits-ui 2.16.5 → 2.18.0, which broke `@testing-library/svelte`'s ResizeObserver expectations. Fix: `cp package-lock.json` first, then `bun install` imports versions instead of re-solving.
- **Coolify doesn't allow changing a resource's source type.** Switching "Git repository" → "Docker image" requires delete + recreate. Screenshot env vars, domains, and ports first — recreation loses them all. Never touch the database service (persistent volume).
- **Single-worker assumption is load-bearing.** `core/maintenance.py`'s flag and `backup_service._BACKUP_WRITE_LOCK` are both in-memory process-global. Scaling horizontally needs a file sentinel + flock; documented in CLAUDE.md.
- **`/health` endpoint must be time-boxed.** FastAPI's default `get_db` dependency rides asyncpg's 60s connect timeout on unreachable DB, so Coolify's liveness probe slow-503s instead of fast-503s. `/health` now bypasses `get_db` and wraps the ping in `asyncio.wait_for(..., 2.0)`.
- **`pg_restore --clean` hangs on ACCESS SHARE locks** held by any live session. `engine.dispose()` handles our pool, but stragglers (psql shells, notebook sessions, requests that slipped the maintenance gate by microseconds) need a `pg_terminate_backend` sweep. Double-sweep with a 100 ms sleep between passes catches requests that were mid-connect when the first sweep ran.
- **`integrity` in sidecar meta is snapshot-at-create-time, never re-checked.** A dump that corrupts on disk later still lists as `ok`. Retention relies on this for "most recent known-good"; edge case flagged as follow-up.

## Future Work / Technical Debt

- **Auto-rollback to pre-restore snapshot on `pg_restore` timeout.** Currently just raises with a recovery hint — the next restore needs admin hands. Bigger design decision (rollback could itself time out); deferred.
- **Retention fallback when all dumps are corrupt.** Current invariant is "most-recent *known-good*"; if integrity is bad across the board, retention still evicts per window and might drop a salvageable older dump.
- **Subprocess test seam.** `backup_service` shells out to real `pg_dump`/`pg_restore` — CI needs the binaries installed, and failure-branch tests (timeout, corrupt archive) are expensive to stage. Extract `_run_subprocess(args, env, timeout) → (returncode, stderr)` to enable cheap monkeypatch-based unit tests.
- **2 GB upload cap is hardcoded** at `backup_service._UPLOAD_MAX_BYTES`. Good enough for an app whose DB is <100 MB; bump the constant if deployments ever need more.
- **`require_backup_cron_token` placement.** Currently in `core/dependencies.py` with the generic auth primitives. Too backup-specific long-term — factor into parameterized `require_bearer_token(settings_attr, error)` when v0.14.0 adds more scheduled-job endpoints.
