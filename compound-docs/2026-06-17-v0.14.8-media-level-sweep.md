# v0.14.8 — Media-level update sweep + v5 result_summary

**Date:** 2026-06-17 | **Branch:** v0.14.8-update-sweep-media-level | **Status:** ready to merge. Live-validated end-to-end on dev DB (sweep ran v5, items 500/500 media-grained, media_refreshed=500, anime_touched=48, media_skipped_fresh=73, ~8 min, zero failures).

## Summary

The nightly `update_sweep` selected work at **anime** granularity (`select_due_for_sweep` returned whole anime; `_refresh_one_anime` re-`/full`'d every child). A long-running airing show like One Piece (~68 stable, years-old media) re-paid a MAL call per member every night — pure waste against the 1 req/s budget. This converts refresh **selection** to media granularity (`select_due_media_for_sweep`): only individually-due media are refreshed, grouped by parent anime for the per-anime commit/probe loop. Adds symmetric `probe_failures[]` logging and bumps `update_sweep` `result_summary` to v5 (counters go media-grained). Admin Overview's SweepTiersCard gains an anime/media toggle.

## Key Decisions

- **Two-clock model is the load-bearing idea.** `MediaFreshness` (`last_checked_at` + new `stable_check_count`) is the per-media **refresh** selection clock; `AnimeFreshness` stays the per-anime **probe** clock. The step-1 loop already iterated per-media and already wrote per-media `last_checked_at` — the only anime-grained thing was the *selection gate*, so the fix was a new query + a media stability counter, not a dispatcher rewrite.
- **Probe needs no new cadence concept.** A recent-main anime was already selected+probed ~weekly; media-level selection re-surfaces it on essentially the same cadence. The only new wrinkle is an anime's media draining across consecutive nights — bounded by a single **7-day floor** (`PROBE_CADENCE_DAYS`) on `AnimeFreshness.last_checked_at` in `_qualifies_for_relations_probe`. Reuses today's weekly cutoff; faithful to prior behavior. Considered and rejected: a separate probe-tier query (more surface, no benefit).
- **Anchor is NOT force-refreshed into the due set.** When a Main anchor isn't due, `raw_payloads` lacks its `/full` payload, so `search_title(seed_payload=None)` fetches `/relations` itself (+1 MAL call per probed anime, at most weekly). Forcing the anchor into `media_to_refresh` would re-introduce the exact inefficiency this release removes.
- **Single thresholds across both grains.** `SWEEP_STABILIZE_THRESHOLD=5` and `SWEEP_LONG_TAIL_DAYS=90` are one constant each, shared by `_media_sweep_atoms` and `_sweep_atoms` so the media due-query, the anime/media tier cards, and the probe gate can't drift. (Anime previously used 3-stable / 180-day; unified to 5 / 90.)
- **Per-anime commit/savepoint kept; NOT pushed to per-media.** Step 1 commits durably per anime; step-1 counters accumulate *after* that commit but *before* the probe, so a `ProbeFailure` can't undercount already-committed refresh work (those media won't re-refresh — `MediaFreshness` already advanced). Per-media commit/rollback was discussed and rejected: sustained-outage cost + circuit-breaker complexity outweighed a modest isolation gain.
- **Progress is media-grained.** `items_total = len(due_media)`, `items_done = media_refreshed`. The bar advances in per-anime chunks but measures the true MAL-call/time unit; the end gap is exactly the media of step-1-failed anime (probe failures don't widen it — their media committed).
- **`JOBS_SWEEP_MAX_PER_RUN` semantics changed (200 anime → 500 media).** Same env var, new unit (media = the real 1 req/s cost unit), default raised to a ~6–8 min window. Called out in `config.py`, `.env.example`, and CLAUDE.md since the *meaning* changed, not just the default.

## Gotchas & Learnings

- **Identity-map sharing makes the eager-load chain work and lets it stay lean.** `select_due_media_for_sweep` eager-loads the parent `Anime` + its *full* `.media` set (reclassify + airing detection read the whole franchise, and `lazy="raise"` is global). The due-media rows are a subset of that set and resolve to the **same** identity-map instances, so loading their child relations under `selectinload(Anime.media)` covers them — a top-level `*media_child_loads` would double the M2M/edge hydration for the due subset during the herd (caught in review; removed). Conversely, do NOT strip the nested `Anime.media` child loads — `reclassify_anime` reads non-due siblings' `relation_edges`.
- **First-sweep herd is by design.** The migration server-defaults every `stable_check_count` to 0 and existing `last_checked_at` == `created_at`, so the whole long-tail catalog is "stabilizing"/due at once on first run — drains over `catalog/JOBS_SWEEP_MAX_PER_RUN` nights, ordered oldest-first (`NULLS FIRST, created_at, id`), no starvation. Same one-time dynamic the anime sweep had at launch.
- **v5 bump is forced by renames + removals, not additions.** `anime_refreshed → media_refreshed`; added `anime_touched` + `media_skipped_fresh`; dropped the `anime_with_dynamic/static_changes` pair (`media_with_*` carries the signal). Frontend keeps v2–v4 parsers; `JobDetailCounters` splits a version-gated HEAD from a shared TAIL so only the rows that changed branch on version.
- **`media_skipped_fresh` is observability-only, free to compute** (`len(anime.media) - len(media_to_refresh)`) — it's the proof the conversion avoids work, not a control signal. At the cap boundary it can include due-but-deferred media; the tooltip discloses this.

## Future Work / Technical Debt

- `count_media_by_sweep_tier_priority` + `count_by_sweep_tier_priority` are full GROUP-BY scans on every admin Overview load — fine at ~800 media / sub-ms, but cache the breakdown (changes only on sweep/scrape) if the catalog grows toward tens of thousands.
- No recent-main media index was added — the freshness/staleness path dominates selection and is served by `ix_media_freshness_last_checked_at`. EXPLAIN ANALYZE on prod-scale data before adding one.
