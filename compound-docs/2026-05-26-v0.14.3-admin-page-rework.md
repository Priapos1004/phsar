# v0.14.3 — Admin page rework + jobs audit + navigation polish

**Date:** 2026-05-26 | **Branch:** v0.14.3-admin-page-rework | **Status:** ready to merge.

## Summary

The admin page becomes a `?tab=` URL-driven shell (Overview / Curation / Jobs Log / Tokens / Backups) so each surface gets full real estate and adding a tab is a one-place change. New backend surfaces — `GET /admin/stats/overview`, `GET /admin/jobs` (paginated all-jobs), `GET /admin/curation/pending-counts` — plus a self-referential `parent_job_id` FK so seasonal-sweep children cluster under their parent in the audit log. Companion polish: bell shows unresolved curation count, daily-50-cap on `user_scrape`, dismissed split candidates stay dismissed, back-to-library navigation propagates across detail pages.

## Key Decisions

- **Daily cap counts every status, including failed.** `count_user_scrapes_in_window` deliberately omits a status filter so a fast-failing client (auth misuse, hitting a perma-banned MAL id, retrying a `PermanentPhsarError`) can't cycle through the limit. The trade-off is acknowledged: a transient MAL outage that burns 50 jobs locks the user for 24 h. The bell shows the 429 as permanent (`DailyJobLimitExceededError` extends `PermanentPhsarError`) so users don't waste retries on an unrecoverable surface — better than silent retry-storms against a still-down upstream. Revisit if real prod traffic surfaces outage-lockout pain.
- **Self-referential FK with `ON DELETE SET NULL`, not a separate `jobs_lineage` table.** Parent/child is single-edge audit metadata, not a graph; a sidecar table would have meant joins on every Jobs Log row. SET NULL preserves audit history when an admin deletes a sweep parent — orphaned children become indistinguishable from real roots, which is the documented intent. Partial index `WHERE parent_job_id IS NOT NULL` keeps the index small (>99% of rows are roots).
- **Default Jobs Log view hides children (`roots_only=True`).** Sunday seasonal sweeps enqueue O(hundreds) of `user_scrape` rows; without the filter, the default 50-row page would be 50 children deep before the next un-clustered job showed up. Expander fetches `?parent_uuid=<UUID>&limit=500` so an entire flock loads in one go without paging; if a sweep ever exceeds 500 children the expanded view surfaces an amber "Showing X of Y" notice — silent truncation isn't an option for an audit surface.
- **Job Health card excludes seasonal-sweep children from `user_scrape`.** "Are users' submissions doing OK?" shouldn't drag down on Sunday's burst of MAL-filtered Music/PV entries. SQL filter on `requested_by_user_id IS NOT NULL`; aligns Job Health with the Activity card's "Scrapes submitted" rule. Child health remains visible in the Jobs Log expander under the parent.
- **`AdminJobResponse` is a subclass of `JobResponse`, not a sibling.** Adds `requested_by_username` as a new field; keeping `/jobs/mine` on the base class means username can't accidentally leak to owner-scoped surfaces — a compile-time guarantee, not a code-review one.
- **Eager-render tabs with `class:hidden`, not `{#if}` unmount.** Admin sessions touch several tabs in a row; one-time parallel-fetch cost on first paint buys instant subsequent switches. No card polls, so persistent mount adds zero ongoing traffic.
- **Sticky split-candidate dismissal via DAO-side cluster-signature compare.** Pre-fix, `upsert_pending` checked only `status='pending'` for a matching signature — a dismissed row was invisible to the supersede path, so the next detection sweep inserted a fresh pending. Now `_candidates_for_anime` loads all statuses (pending/dismissed/split); matching signature on any historical row returns False and inserts nothing. Genuinely new contamination (different signature than every historical row) still supersedes and inserts. The `USER_FLOWS` doc already described this behaviour — spec was right, code was wrong.
- **`current_position` is a required schema field, not optional with a 0 default.** Position 0 is a legitimate slot (current media is oldest); a defaulted-missing field would silently render the "you are here" marker at the front of the row.
- **`chronological_media_key()` is the single sort source for three services.** Spoiler frontier, anime-detail media table, related-media carousel all import the same key. Prior state had three near-identical sort lambdas drifting in subtle ways; the new helper means "if the order looks wrong on one surface, it's wrong on all three" — easier to find, harder to silently diverge.
- **`DetailOrigin` union narrowing on the back-button origin.** Adding a future entry point (watchlist, recommendations) is a compile error in both the type AND `BackLink`'s switch until both are updated — same shape as the `Record<JobKind, string>` narrowing on `JOB_KIND_LABELS`.

## Gotchas & Learnings

- **`result_summary["retryable"]` reads coalesce to `True` for legacy rows.** Pre-marker rows have no `retryable` key in their JSONB; the Job Health "retryable subset" query uses `coalesce(..., True)` so legacy failures retain their pre-marker semantic (treated as transient-retryable). Without this, every old failure would render as a permanent failure in the new card.
- **Bell `seenCount` clamps below `totalPending` on candidate resolution.** Otherwise: resolve all candidates, then a new one lands — `seenCount` (stored at last dropdown open) is now higher than `totalPending` so `max(0, total − seen)` returns 0 and the badge silently misses the new work. Same pattern as `BELL_SEEN_KEY` for finished jobs.
- **`SeasonType(str, enum.Enum)` hashes equivalently as enum-or-string.** Two earlier callsites used `.value` to extract the raw string before lookup; that crashed when the column returned a plain string (test fixtures, hand-inserted rows). Dropped — `SEASON_ORDER.get(name)` works on either shape because `SeasonType` is a str subclass.
- **Backfill script is safe because seasonal-sweep dispatcher is the only source of NULL-user `user_scrape` rows.** `backfill_seasonal_sweep_parents.py` attributes any unparented NULL-user `user_scrape` to the closest prior `seasonal_sweep` by `created_at`. A stray manual NULL-user job would mis-attribute — but no production code path creates one. Documented + idempotent (re-runs filter `parent_job_id IS NULL`).
- **Three existing migration-only indexes (`ix_jobs_user_scrape_recent`, `ix_jobs_scrape_query`) were brought back into the model `__table_args__`.** Alembic `--autogenerate` was flagging drift on every run because the indexes existed in DB but not in metadata. Single source of truth restored.
- **`PARENTING_KINDS` constant gates the Jobs Log expander chevron.** Set lives next to `JOB_KIND_LABELS` in `formatString.ts`; today only `seasonal_sweep`, but a second parent kind drops in by adding one entry. Backend's `?parent_uuid=` accepts any uuid, so this is purely the frontend's "does this row deserve a chevron" guard.

## Future Work / Known Follow-ups

- **Daily-cap exempt for transient-outage failures.** Reviewer-flagged HIGH: a MAL outage burning 50 failed jobs locks a user out for 24h. Option: exclude `result_summary.error_category = 'upstream_outage'` (or `retryable=True`) from the window count, matching the dedupe path's "failures don't lock the query" semantic. Deferred — needs real outage signal before tuning.
- **Admin Overview has no graceful degradation on partial sub-query failure.** Reviewer-flagged MEDIUM: a statement-timeout on `_jobs_stats`'s GROUP BY 500s the whole tab. Wrap each sub-aggregator in try/except with a sentinel "unavailable" shape, OR add a frontend skeleton fallback per card. Deferred until the queries cross ~10ms in EXPLAIN against a year-old catalog.
- **`list_admin_paginated` COUNT is unbounded.** Scale-tomorrow concern; at 50k+ jobs the COUNT dominates page latency. Options when it bites: cap with `LIMIT 1001` subquery for "1000+" semantics, or per-filter cache. No change today.
- **Backfiller bell visibility.** `backfill_split_candidates` at lifespan startup still runs silently (carried over from v0.14.2). At many-thousand-anime catalog the few-second pass may matter for cold-start latency — wrap in a job kind?
