# v0.14.7 — Sweep failure logging, not-yet-aired relations, spoiler-cache scoping

**Date:** 2026-06-17 | **Branch:** v0.14.7-little-fixes | **Status:** ready to merge. All three fixes validated against a restored prod dump + the dev catalog.

## Summary

Three independent fixes, all surfaced while investigating why a nightly `update_sweep` reported only **64 "Anime touched"** when it had selected 113.

1. **Silent step-1 sweep failures (v4).** Investigating the 06-17 sweep against the prod dump showed `items_total=113`, `items_done=64` — 49 anime were selected but never committed. Each hit an MAL outage during step-1 refresh, was swallowed by `_try_step1_refresh` (`logger.exception` + `return None`), and counted in **no** `result_summary` field. The sweep ran 47 min vs the usual ~30 (retry-backoff signature) yet rendered a clean green `succeeded`. Fix: `_try_step1_refresh` returns a typed `Step1Failure`; the dispatcher counts it (`counters.step1_failed`) and records `{anime_uuid, title, error_category, error_message}` in top-level `step1_failures[]`. Bumped `update_sweep` v3 → v4. Frontend: a "Failed refresh" counter cell (amber, "—" pre-v4), an `items_done`/`items_total` divergence warning, and a per-anime failure list card.

2. **Not-yet-aired sequels mis-classified as `side_story`.** "Hell Mode" S2 (mal 63817, TV, Not yet aired, NULL episodes + NULL duration) was correctly reached via the sequel edge into the main chain, then demoted by `passes_substance` at its first line (`if duration is None: return False`) — before the NULL-episodes exemption. The classifier had no `airing_status`, so it couldn't tell "metadata not published yet" from a genuine anomaly. Fix: add `airing_status` to `ClassifierNode`; NULL duration/episodes is a **provisional pass for Not-yet-aired only**.

3. **Spoiler-cache scoping + restricted-user exclusion.** The four post-mutation call sites recomputed the *whole catalog* for *every* user (`refresh_spoiler_cache_for_all_users`, O(users × all_media)). Replaced with `refresh_spoiler_cache_for_anime_ids` (O(users × changed)). Restricted (guest) users — who can't rate — are pinned to `spoiler=off` and dropped from the cache entirely.

## Key Decisions

- **Bump update_sweep to v4 even though the new keys are net-new with safe defaults.** The registry's own rule says net-new + safe-default = no bump. We bumped anyway so the frontend can distinguish a v4 `step1_failed: 0` (genuinely zero failures) from a v3 row that never tracked it (rendered "—", not a misleading `0`). The rationale is encoded inline in `job_versions.py` so the next shape-change author sees why.

- **Typed `Step1Failure` return, not a bare counter.** `_try_step1_refresh` returns `RefreshResult | Step1Failure`. The `isinstance(step1, Step1Failure)` branch in the loop is clearer than `dict`-sniffing, and capturing the anime identifiers happens *before* the savepoint (same MissingGreenlet trap that hit `relation_backfiller` — reading `anime.uuid` in the `except` after a nested rollback can fault on an expired instance). `error_category` reuses `job_worker.classify_error` (promoted from private `_classify_error`) so a sweep-time MAL outage tags the same `upstream_outage` as a job-level failure.

- **Not-yet-aired leniency is "Not yet aired" ONLY, never "Currently Airing".** Verified against the prod catalog (185 anime): 34 media flip fail→pass with a duration-NULL leniency. On the date axis none can steal an anchor (not-yet-aired sorts last). On the **tier** axis exactly one anime was at risk — Fei Ren Zai, whose whole ONA franchise is 5-6 min shorts (so the Movie is the anchor): its *currently-airing* "3rd Season Part 2" (NULL duration) would provisionally pass and, as ONA (tier 2) vs Movie (tier 3), steal the anchor onto a mid-franchise part. Restricting leniency to Not-yet-aired drops that case. Currently-airing long-runners (One Piece) already pass via their real published duration + the existing episodes-NULL exemption.

- **Not-yet-aired is main-chain-eligible but NOT anchor-eligible.** A follow-up fix after the first landed: Hyakushou Kizoku is a short-form franchise (240s/episode) whose aired seasons all fail the duration gate. With the leniency, the announced 4th season (NULL duration, not yet aired) became the *only* substance-passing node → the sole anchor candidate → stole the umbrella onto an unaired season (`anchor_changed=True` on the backfiller). Split the two uses: `passes_substance` still passes pending nodes (so a sequel isn't demoted), but `_pick_anchor` excludes them via `_is_metadata_pending`. When nothing else is anchor-eligible the fallback picks the most main-like of all nodes (the oldest aired TV), restoring the pre-leniency anchor. **Lesson: a substance-gate change has two independent consumers (demotion + anchor selection); test both axes.**

- **`AIRING_STATUS_*` sentinels moved into `relation_classifier` (the pure, DB-less module).** The substance gate now needs the "Not yet aired" string, and `relation_classifier` can't import from `jikan_scraper` (that would cycle — jikan_scraper already imports the classifier). So the constants live in the classifier as the single source of truth; the four consumers (jikan_scraper, scrape_dispatcher, anime_search_service, search_filters, anime_dao) import directly from there.

- **Scoped spoiler recompute is sufficient because the frontier is per-anime and the cache keys on media_id.** Media ids only move *between the named anime* on merge/split, so recomputing exactly the affected anime covers every moved row. The whole-catalog `refresh_spoiler_cache_for_all_users` had no callers left after repointing the four sites, so it was deleted — `backfill_spoiler_visibility` (startup) remains the sole whole-catalog path. Each call site passes the affected set it already tracks (save → `new_anime_ids`, sweep → `probe_attached_anime_ids`, merge → survivor, split → source + new). Ratings were *already* scoped per-(user, anime) via `recompute_visibility_for_anime`.

- **Restricted users: pin spoiler=off server-side, exclude from the cache, show settings disabled (not hidden).** Guests can't rate, so a frontier freezes at episode 1 of every anime and "hide" would blank the catalogue. The pin is enforced in `user_settings_service.update_settings` (drops `spoiler_level` for `RestrictedUser`); they're skipped in both registration recompute paths, the startup backfill query, and the scoped recompute. The read path is safe — `get_visible_media_ids` is only consulted when `spoiler_level == hide`, which guests can never reach. Per the user's call, the settings UI shows all controls **disabled** (rating step, spoiler pinned off, export) rather than hidden, so guests see what a full account offers.

## Failed Approaches

- **First substance-gate fix included "Currently Airing" in the pending-statuses set.** Looked symmetric (both airing and not-yet-aired can have NULL metadata), but the Fei Ren Zai anchor-steal made it wrong: an airing show normally *has* a published duration, so a NULL there is a transient gap or a genuinely short entry — not "pending". Caught by querying the prod catalog for tier-steal candidates before shipping.

- **First not-yet-aired fix made pending nodes fully substance-passing (anchor-eligible included).** Passed the Hell Mode test but broke Hyakushou Kizoku (short-form franchise → unaired season stole the anchor). Surfaced by the user running the relation backfiller on the dev DB and reading `anchor_changed=True` in the log. The prod tier-steal query had missed it because it only checked *better-tier* steals, not the "all real media short-form → lone pending node becomes sole anchor" case.

- **Skipping the delete when a user has no ratings on the affected media (efficiency micro-opt suggested by review).** Would be *incorrect* — with zero ratings the first anchor of each affected anime is still visible, so `visible_ids` is non-empty and the delete+insert is required. Skipped the suggestion.
