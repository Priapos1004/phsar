# Jikan / MAL Scraper Quirks & Field Notes

**Date:** 2026-05-11 (initial) | **Owners:** `phsar/app/services/jikan_scraper.py`, `phsar/app/services/search_service.py`, `phsar/app/services/scrape_dispatcher.py`

## Summary

Field notes on Jikan v4 (and the MAL data underneath it) accumulated through v0.13.x → v0.14.0. This is the place to record MAL data quirks, rate-limit lessons, BFS edge cases, and failure modes — anything where the *implementation choice* is non-obvious from the code alone. Append findings here as they're discovered; cross-link to the change/commit that surfaced them.

The corresponding production code lives in three layers:

- **`jikan_scraper.py`** — HTTP client + tenacity retry + class-level rate-limit gate + the BFS algorithm itself (`search_title`).
- **`search_service.py`** — orchestrates BFS, normalizes outputs, handles attach-to-existing / promote-root fallbacks, runs DB-side dedupe.
- **`scrape_dispatcher.py`** — invokes the above as a background job and translates failures into user-visible error categories.

## v0.14.1 — Two-pass relation classifier redesign

The v0.14.0 scrape-time classification was path-dependent. Scrape-from-Movie-1 vs scrape-from-TV produced different relation_type labels on the same franchise. Most visible case: Evangelion seeded from a Rebuild Movie (the typical search-title hit) anchored the umbrella on Movie 1 and marked the 1995 TV as `side_story`. Overlord had two side issues: a 60-second "Manner Movie" came back as `main` because `is_main_story` only checked TV/Movie type + no parent_story upstream; a standalone 2024 "Manner Movie" was its own anime row.

v0.14.1 splits the work into a deterministic two-pass design:
- **Pass 1** (`jikan_scraper.search_title`) captures relation **edges**, not classifications. Returns `(graph, edges, cross_link_mal_ids)` per BFS-result.
- **Pass 2** (`relation_classifier.classify_anime_relations`) picks a canonical anchor by substance gate + tier + age, builds the main chain via sequel/prequel transitive closure, classifies alt-chain via `alternative_version` edges, defaults the rest to `side_story`. Substance gate demotes weak `main` (60-sec Manner Movie inside a healthy chain).

Why two passes: anchor choice no longer depends on which media the user searched for. Same helper runs at three sites: scrape (per BFS-result), merge survivor (consolidated `A.media ∪ B.media`), backfiller (per catalog row).

**Entries below superseded by v0.14.1** (kept for the failure-mode rationale but the code paths are gone):
- "`is_main_story` gated only on TV/Movie" — the gate is removed entirely. Pass 2 handles donghua via tier 2 (ONA) anchor selection.
- "`_pick_root_for_promotion` fallback in search_service" — folded into `relation_classifier.pick_anchor`'s built-in fallback.
- "Post-loop seed-recovery net" / "Re-enqueue check at the bottom of BFS loop body" — BFS no longer drops the seed.
- "Relation string format" gotcha — `_normalize_relation` is now `normalize_relation` (public) and `parse_relation_edges` does the filter + emit.

## Failed Approaches

- **Rate limit at 0.35s (~2.85 req/s).** Within Jikan's documented 3 req/s burst but observed 429s in practice on `/seasons/now` pagination and seasonal-sweep child BFS runs. The binding constraint is the **per-minute** ceiling (60 req/min), not the per-second burst. Never do: set spacing based on the burst ceiling for continuous-request workloads.
- **Rate limit at 0.5s (~2 req/s).** Same outcome. Even 2 req/s exceeds 60/min sustained over a minute window. Never do: assume "well under 3 req/s" is safe without computing the rolling-minute total.
- **Placeholder titles in MediaUnwanted** (e.g., `<mal_id:NNNN>`) for entries where MAL returned `title=null`. Permanently blacklisting a transient null-title pollutes the admin-only table with garbage strings AND blocks rediscovery once MAL fills in the title. Never do: blacklist on transient MAL data states — distinguish "MAL hasn't finished populating this row yet" from "we deliberately don't want this kind of media."
- **Walking `Alternative Setting` relations.** Treating it like Sequel/Prequel/Side Story conflates separate franchises that just share themes (Zhe Tian ↔ Wanmei Shijie, Madoka Magica ↔ Magia Record). The BFS walks both worlds into one Anime row and the merge detector then flags false positives on every sweep. Never do: expand `alternative_setting`; treat it as a graph boundary (like `crossover`, but without recording the node either — alt-settings are explicit "different story" labels).
- **`is_main_story` gated only on `TV`/`Movie`.** Brand-new donghua sub-universes whose canonical main is an ONA produce a graph with no main → `MainMediaNotFoundError` → seasonal-sweep child fails for the whole franchise. Never do: assume `TV`/`Movie` covers the canonical-main case for non-Japanese MAL entries. (We didn't fix the gate itself — instead added a `_pick_root_for_promotion` fallback. See Key Decisions.)
- **`AnimeNotFoundError` for empty 200 responses.** When MAL returns `{"data": null}` or `{"data": {}}` (transient hiccup; observed for valid mal_ids), the old code raised `AnimeNotFoundError`, a `PermanentPhsarError` → `retryable=False` → bell hides retry → user locked out for the 72h dedupe window. A real 404 is `HTTPStatusError` from `_get`, which is correctly permanent. The empty-data case is transient and deserves `TransientUpstreamError` (retryable). Never do: conflate "MAL returned no row" with "MAL returned no data this time."
- **Not retrying 429.** At a sustained 1 req/s rate with no headroom under MAL's 60 req/min, a brief per-minute-window overrun produces a 429 even though the average rate is correct. Not retrying = a single 429 kills the whole job. Never do: treat 429 like other 4xx; 429 is the one 4xx with a "wait and retry" semantic baked into HTTP. (Counterpoint: at *higher* request rates, retrying 429 masks sustained abuse — see Key Decisions for how we cap.)
- **Re-enqueue check at the bottom of the BFS loop body.** The original BFS had `if not left_mal_ids and mal_id not in visited_ids: re-enqueue` as the last statement inside the `while` body. When the last popped node hit a `continue` path (visited / excluded / null-title / filtered), the check was bypassed. For franchises whose only relation is `Other` and no node points back (Rilakkuma → Pocket no Naka), the dropped seed was never recovered and the graph ended without a main story. Never do: put a recovery check below a `continue` boundary without an explicit post-loop net.
- **MediaUnwanted entries as cross-link signal.** When the BFS reached an excluded mal_id and added it to `cross_link_mal_ids` for the merge detector, it didn't distinguish "this lives in Media under some anime" (real cross-link) from "this lives in MediaUnwanted" (filtered, no parent). The attach-to-existing fallback then tried to attach a graph to a Music/PV/CM with no parent Anime; the attach silently no-op'd; the dispatcher then raised `MainMediaNotFoundError` instead of trying the promote-root fallback. Never do: trust raw `excluded_mal_ids` membership as a franchise-overlap signal.
- **Substance gate rejecting NULL episodes for TV/ONA.** First pass of the gate required `episodes IS NOT NULL AND episodes >= 8` for all TV-like types. Currently-airing or long-running shows (Detective Conan, Anpanman) have `episodes=None` because MAL has no terminal count — they failed substance and lost their anchor slot to a substance-passing sub-show (Magic Kaito 1412 for Conan, a 90-min Movie for Anpanman). Never do: treat NULL on a bounded field as automatic failure when the field is legitimately unbounded for a class of rows. Fix: NULL episodes is acceptable for TV/ONA (the duration check carries the substance bar). TVSpecial stays strict — that type is bounded by definition.
- **Filtering "dangling" edges at scrape time.** The BFS's `graph_edges = [(a, b, r) for a, b, r in edges if a in graph_ids and b in graph_ids]` filter at storage dropped any edge whose target wasn't in the current anime's graph. Silently lost the bridges needed when two anime later got merged: a split Dr. Stone fixture (A holds S1+TVSpecial bridge, B holds New World+) loses Stone Wars→New World at scrape time. After merge, the consolidated classifier's main-chain closure stops at the TVSpecial bridge (which fails substance and gets demoted) and New World+ default to `side_story`. Never do: filter at storage when the filter's premise (graph_ids) can change later. Fix: persist all observed edges; defensive filter at classifier input.

## Key Decisions

- **Rate limit `_MIN_REQUEST_INTERVAL_S = 1.0`** (1 req/s) for continuous-request flows. Matches Jikan's *sustained* 60 req/min, not the *burst* 3 req/s. Trade-off: a 200-anime sweep takes ~3-5 min instead of ~1 min, but no 429s and no IP-ban risk. Class-level lock so multiple `JikanScraper()` instances share the gate.
- **Asymmetric retry budget** via `_stop_strategy`. 429: 3 total attempts (2 retries — enough to bridge a misaligned per-minute window, not enough to mask sustained throttling). 5xx / timeout / network: 5 attempts. Other 4xx (404, etc.): not retried (deterministic). The 429 cap is deliberate — if MAL is sustained-rate-limiting us, retrying harder doesn't fix it; the right response is to slow the source rate.
- **Null titles → silent skip** (same sentinel pattern as `Not yet aired` + `media_type=None`). MAL routinely leaves `title=null` on entries it's still populating (romanization pending for Chinese/Korean shows, brand-new PV stubs); the field reliably fills within hours. We skip the entry from the BFS without adding to MediaUnwanted, so the next sweep gets another chance.
- **`alternative setting` as a graph boundary cut.** The relation is *enqueued* alongside `adaptation` — we never walk it. (Note: the literal in the code uses a SPACE (`"alternative setting"`), not the underscore form some other checks use. See Gotchas.)
- **`_pick_root_for_promotion` fallback** in `search_service.search_mal_api`. When `get_first_main_relation` raises AND `seed_mal_id` is set AND there's no single cross-link, pick the most main-like entry by `(type_tier, aired_from)` where TV/Movie=0, ONA=1, OVA/TVSpecial=2, Special=3 (nulls last on aired_from). Promotes that entry to `relation_type="main"` and saves the graph as a new Anime. Handles donghua sub-universes (whose canonical main is ONA) cleanly; also handles the pilot-aired-before-the-real-show edge because tier > date.
- **Post-loop seed-recovery net** after the BFS `while` loop. When the seed was dropped via the `is_main_story + has_relations` branch AND no organic rediscovery happened AND the inline bottom-of-loop re-enqueue was bypassed by a `continue`, re-add the seed directly as `relation_type="main"` (the drop branch only fires when `is_main_story=True`, so this classification matches).
- **`unwanted_mal_ids` filter on cross-link signals.** `search_mal_api` subtracts MediaUnwanted entries from `cross_link_mal_ids` before the attach-vs-promote decision. MediaUnwanted entries aren't real franchise overlaps — they're filtered media with no parent Anime to attach to.
- **`AttachToExistingAction` schema for orphan-side-story graphs with a single cross-link to existing Media.** The dispatcher routes these through `attach_search_result_to_anime` (the same primitive 7c's freshness probe uses for tier-3 sweeps), attaching new media under the existing parent instead of saving as a duplicate Anime. Closes the tier-1/tier-2 parent gap where the probe doesn't run.
- **`TransientUpstreamError(PhsarBaseError)`** distinct from `PermanentPhsarError`. Empty-200 MAL responses surface as transient so the bell offers retry. `_classify_error` in `job_worker` also tags it as `upstream_outage` for friendly bell copy.
- **`AnimeFilteredOutError(PermanentPhsarError)`** distinct from `AnimeNotFoundError`. When the dispatcher sees `results == []` after a seeded BFS AND `MediaUnwantedDAO.get_by_mal_id(seed)` returns a row, the seed was filtered as Music/PV/CM/Hentai — surface that explicitly. Admin sees `"'X' was filtered out as PV"` in `jobs.failed` instead of the misleading `"Anime titled X not found"`.
- **`MainMediaNotFoundError` surfaced honestly** when a seeded BFS finds the entry but can't anchor it (no main in graph AND no single cross-link AND no MediaUnwanted match). Previously fell through to `AnimeNotFoundError`, which implied MAL had no record at all.
- **Crossover is kept, alt-setting is cut.** Both label franchise boundaries, but crossover anime *are* legitimately part of multiple franchises (a Fate × Tsukihime crossover belongs to both worlds). Record crossover nodes in the graph with `relation_type="crossover"`; stop expanding from them. Alt-setting anime are *not* in the same franchise — drop them entirely.
- **`RelationType.AlternativeVersion` enum value** for retellings (Evangelion TV ↔ Rebuild Movies, Hokuto no Ken alts, Pretty Rhythm alternates). Pass 2 classifier builds the alt-chain via `alternative_version` edges from the main chain. Spoiler frontier treats alt-version as an anchor (gates the next in the alt-chain — rating Eva TV reveals Rebuild Movie 1 but not Movie 4, which has a wildly different ending). FMA 2003 ↔ Brotherhood is a known-not-fixed limitation: those are arguably distinct productions that should be separate anime rows, but the classifier currently collapses them. Inverse-merge admin tool is v0.15+ work.
- **Substance gate**: TV/ONA needs `episodes >= 8 AND duration_seconds >= 600` (10min); Movie needs `duration_seconds >= 1800` (30min). NULL episodes accepted for TV/ONA (currently-airing / long-running); TVSpecial stays strict. Gate fires twice in classifier: anchor eligibility, then post-classification demotion of weak Mains absorbed during merges (the Overlord standalone-Manner-Movie absorption case).
- **Anchor tier ordering**: TV(1) > ONA(2) > Movie(3) > anything else(4). Within tier: oldest aired_from, then highest scored_by, then lowest mal_id (final tiebreak for determinism). Donghua canonicals are usually ONA → tier 2 anchors them correctly even without TV.
- **Edges persisted unfiltered per Media**: `MediaRelationEdges` sidecar holds the full MAL relation list (`[[target_mal_id, normalized_rel], ...]`) for each media, including targets outside the current anime's graph. Classifier's `_build_adjacency` filters dangling endpoints at adjacency-build time. Benefit: when two anime get merged, previously-dangling bridge edges activate and the consolidated main chain reconnects through substance-failing intermediates (Dr. Stone S1→Stone Wars→Ryuusui TVSpecial→New World stays connected). Cost: ~50-500 bytes per Media sidecar; ~250-500 KB total for an 800-media catalog.
- **7-field umbrella drift detection** in `anime_relation_service.reclassify_anime`: the helper rewrites `mal_id`, `title`, `name_eng`, `name_jap`, `other_names`, `description`, `cover_image` on ANY field drift (not just anchor change). Embedding regen is gated separately on title-affecting drift (skip the ~50-100ms encode for cover-image-only changes). Same helper used by relation backfiller (per-catalog-row) and merge survivor refresh.

## Gotchas & Learnings

### MAL data quirks

- **`title=null` is transient.** Sampled the 100 newest MAL entries during validation and found zero null titles. The failures we observed earlier hit nulls that MAL populated within hours (likely the romanization pipeline lagging behind the entry creation). When the BFS sees null title, silent-skip and trust the next sweep.
- **Empty 200 OK ≠ 404.** MAL occasionally returns `{"data": null}` or `{"data": {}}` for a valid mal_id (a known case: `/anime/64060` in our validation). This is distinct from a real 404 (which raises `HTTPStatusError`). Treat empty data as transient (`TransientUpstreamError`), not permanent.
- **`alternative_setting` ≠ same franchise.** This relation labels separate stories that share themes/setting (e.g., Madoka Magica ↔ Magia Record share the magical-girl framework but are different shows). Always treat as a graph boundary.
- **`Other` relations aren't reciprocal.** Many franchises link only forward via `Other` (Rilakkuma → "Rilakkuma to Kaoru-san") with no back-pointer. Combined with the seed-drop branch, this means the BFS can lose the seed entirely without the post-loop net.
- **Crossover anime exist as standalone MAL entries.** They link to multiple franchises via `Crossover` relations. Don't traverse the crossover boundary, but DO record the node in the graph.
- **Per-minute window is enforced tightly.** Even at exactly 60 req/min average, a brief micro-burst (e.g., variability in the request scheduler) puts you over for a few hundred ms and triggers 429.
- **Jikan upstream is MAL.** The Jikan docs explicitly state "It's still possible to get rate limited from MyAnimeList.net instead" — even when our rate is fine for Jikan, MAL might throttle us independently. The 429 retry policy hedges against both.

### Relation string format

- **MAL emits relation strings with SPACES and CamelCase**: `"Side Story"`, `"Alternative Setting"`, `"Parent story"`, `"Full story"`, `"Spin-Off"`. The scraper normalizes them once on entry via `_normalize_relation(rel) = rel.lower().replace(" ", "_")`, so every downstream comparison works against the underscore form.
- **Why the normalization matters (regression from v0.13.x → fixed in v0.14.0).** The `is_main_story` gate excludes any popped TV/Movie node whose relations list contains `"parent_story"` or `"full_story"`. Before the fix the list was built from raw MAL strings (`"Parent story"`), so both `not in` checks always passed and any TV/Movie with a `Parent story` upward link silently classified as `main`. In practice every Naruto/Bleach/Demon Slayer movie ended up as a separate `main` row instead of `side_story` under its parent series. The Naruto-shaped fixture in `test_search_title_normalizes_relation_strings_so_franchise_movies_are_side_story` is the regression guard.
- **Fix is for new scrapes only.** The sweep's relations probe goes through `attach_search_result_to_anime`, which doesn't reclassify existing `relation_type` values. Catalog rows scraped pre-fix keep their stale `main` labels until the parent anime is deleted and re-scraped. Document this in the upgrade notes for the next deploy; the alternative (data migration that re-runs classification) is its own follow-up.

### BFS algorithm specifics

- **`is_main_story` only applies to the seed.** The check requires `current_relation is None`, which is true only for the first popped node. Other nodes get their relation_type from the `Sequel`/`Side Story`/etc. label that put them in the queue.
- **Drop-and-rediscover dance.** When the seed is_main_story AND has relations, the BFS deliberately removes it from `visited_ids`, `all_info`, and the graph, expecting BFS to walk back through a relation pointing TO the seed (Sequel → seed, Side Story → seed, etc.). When that works, the seed gets a fresh classification in the second pass. When it doesn't (Rilakkuma case — only outgoing `Other`), the post-loop net catches it.
- **The seed re-enqueue check inside the loop is bypassed by `continue` paths.** Pre-fix it sat at the bottom of the loop body and only fired when the iteration ran to completion. The post-loop net at the end of the BFS is the safety belt.
- **`Character` relations are skipped entirely.** Not even recorded — they typically link unrelated anime via shared characters (e.g., a "No More Eiga Dorobou" anti-piracy spot linked from every show that played the spot). Useful for fan navigation, useless for franchise graphing.
- **`Adaptation` is skipped** because adaptations are usually manga/LN (the BFS already filters by `type=anime`, so this is belt-and-braces).

### Failure-mode taxonomy

The dispatcher distinguishes four post-empty-result paths:

| Cause | Detected via | Surface |
|---|---|---|
| Seed was Music/PV/CM/Hentai | `MediaUnwantedDAO.get_by_mal_id(seed_mal_id)` returns row | `AnimeFilteredOutError("'X' was filtered out as Music")` |
| MAL returned empty 200 | Caught earlier in `search_title` | `TransientUpstreamError("MAL returned no data for mal_id=N")` |
| Graph has no main + no single cross-link | Promote-root fallback didn't pick up | `MainMediaNotFoundError("Couldn't identify a main story for 'X'")` |
| No new match (no seed_mal_id mode, fuzzy query) | `results == []` from title search after catalog + MediaUnwanted filtering | `AnimeNotFoundError("No new anime matched 'X' on MAL")` |

This taxonomy was learned empirically — earlier validation runs surfaced jobs in `failed` state with misleading copy that made it hard to tell *which* failure mode hit. The dispatcher's `if not results and not attached_count:` ladder is the lookup table.

### Concurrency / isolation

- **Per-anime commit in `update_sweep_dispatcher`** but **bulk insert in `seasonal_sweep_dispatcher`**. The freshness sweep does MAL I/O per anime; per-anime commit preserves work on crash. The seasonal sweep's enqueue loop does no MAL I/O (already drained in the `fetch_current_season` step); a single `add_all` + commit shortens the maintenance window and the crash-safety argument doesn't apply.
- **Class-level rate lock means concurrent `JikanScraper()` instances share the gate.** If user_scrape and update_sweep ever ran concurrently (currently serialized by single-worker), they'd correctly throttle together at 1 req/s combined. Don't naively parallelize.

### v0.14.3 notes

- **Anchor discovery via structural-upward walk fixes entry-point dependence introduced by v0.14.2 strict BFS.** Strict alone made graph membership depend on which fuzzy MAL search result we entered from: a `q=overlor` typo returns top-3 Movies (no TV root), so WALK never propagates through the sequel chain and the franchise fragments. Fix: before the main BFS, walk only `{prequel, parent_story, full_story}` from each search root to find the canonical chain start (Movie 1 → `full_story` → S1; Pleiades → `parent_story` → S1), then prepend that anchor to the BFS root set. See [jikan_scraper.py:_discover_anchors_upward](../phsar/app/services/jikan_scraper.py).
- **Why exactly `{prequel, parent_story, full_story}`.** These are the relations MAL uses to point a derivative work at its canonical ancestor. `other` is intentionally absent — that's where cross-franchise bridges live (Pleiades x Kagejitsu, Eva ↔ Ultraman). `sequel`/`side_story`/`spin-off`/`summary` go downward (toward derivatives) or laterally, not toward an anchor. `alternative_version` is also out: it's lateral within a franchise (Eva TV ↔ Rebuilds), and the main BFS already propagates WALK across it as an identity-preserving relation, so the anchor pass doesn't need to fetch via it.
- **Only true chain starts get promoted as anchors.** A node is added to the discovered set only if it has NO outgoing structural-upward edges at all. Intermediate nodes whose upward targets are already-visited or in catalog are not anchors. Caught a bug in the first implementation where Movie 1 (intermediate, prequel-chain → Movie 2; full_story → S1) was getting promoted ahead of S1, then walking S1 via `full_story` (identity-breaking in main BFS) demoted S1 to TERMINAL — losing the entire sequel chain. The fix is the "not all_upward → discovered" check.
- **Catalog-aware termination.** If anchor discovery reaches an `excluded_ids` member, the walk stops there and the catalog member is NOT promoted as a fresh root. The main BFS surfaces it as a cross_link instead, so `save_service` routes new media via attach-action to the existing anime. Test: `test_search_title_anchor_discovery_stops_at_catalog`.
- **`relation_cache` shared between discovery and main BFS.** Anchor discovery's upward walk fetches `/relations` for intermediate nodes; the main BFS would otherwise re-fetch when it reaches them. A `dict[int, list[dict]]` cache keyed on mal_id is passed through both phases, so each node's `/relations` is fetched at most once per `search_title` call. The seed path also pre-populates the cache with the dispatcher's bundled `/full` relations.
- **`_ANCHOR_DISCOVERY_MAX_HOPS = 10`.** Defensive cap against pathological MAL data (cycles, absurdly long chains). Typical franchises walk 2-5 hops upward. If the cap is hit with a non-empty frontier, those nodes are added as fallback anchors so we never produce zero output (test: `test_search_title_anchor_discovery_respects_max_hops`).
- **Entry-point invariance is the new invariant test surface.** [tests/services/test_jikan_scraper.py::test_search_title_overlord_entry_point_invariance](../phsar/tests/services/test_jikan_scraper.py) is parametrized over 9 different "MAL search returns" sets that all anchor to Overlord S1 — `[S1]`, `[S2]`, `[S3]`, `[S4]`, `[Movie 1]`, `[Movie 2]`, `[Pleiades]`, `[S1, S4]`, `[61345, Movie 1, Movie 2]` (the real `q=overlor` case). Each must produce a graph with the full S1-S4 main chain AND no Eminence members. The single exception is `[61345]` alone (Manner Movie 2 has no structural-upward edges) — locked in as the explicit "user has to make a more precise query" tradeoff.
- **Failure-mode of anchor discovery.** Fetch errors during the upward walk propagate naturally: tenacity-retry handles transient 5xx/429/timeouts; persistent failures bubble up to the worker as the normal retryable path. `AnimeNotFoundError` during discovery is unlikely (we're following MAL's own edges) but would propagate identically to today's BFS-time `AnimeNotFoundError`.
- **Roots sorted by anchor tier before the main BFS loop.** Anchor discovery only fires when a search root has structural-upward edges. When none of the search roots has any (Eva case: `q=Evangelion` returns `[Chao Xianshi ONA, 3.0 (-46h) Special, Eva TV]` — none of those three has prequel/parent_story/full_story), discovery returns ∅ and we're back to entry-point dependence within the original results. Without sorting, iter 1 (root=Chao Xianshi ONA) walks `other → Eva TV`, demoting Eva TV to TERMINAL in `visited_ids`. Iter 3 (root=Eva TV) then skips → whole Eva chain lost. Fix: sort `results` by `(_ANCHOR_TIER, aired_from)` mirroring the classifier's anchor logic, so the highest-tier root processes first and its WALK propagates through the canonical chain before lower-tier results have a chance to mark anything TERMINAL. `_ANCHOR_TIER` is imported from `relation_classifier` to keep the two in sync.

### v0.14.2 notes

- **Identity-breaking relations bound the BFS at TERMINAL.** Restores the pre-v0.14.0 graph membership: a node arriving at the seed graph via `side_story`, `parent_story`, `summary`, `full_story`, `other`, or `spin-off` is recorded as TERMINAL (info only — its relations are NOT fetched, its neighbors aren't queued). Only `sequel`, `prequel`, and `alternative_version` propagate WALK identity from a WALK parent. See [jikan_scraper.py:_next_expand_status](../phsar/app/services/jikan_scraper.py). Crossover stays the existing hard boundary; `alternative_setting` / `adaptation` / `character` stay filtered before edge capture.
- **The Overlord → Eminence in Shadow regression** triggered the fix: scraping Overlord pulled in the entire Eminence franchise via `Pleiades (31138) → other → Ple Ple Pleiades x Kagejitsu! (57034) → other → Kagejitsu! Second (56842) → sequel chain → Eminence S1/S2/etc`. MAL labels the real-world collab special `Ple Ple Pleiades x Kagejitsu!` with relation `Other` (not `Crossover`), so the pre-v0.14.2 BFS walked through it. Under v0.14.2 strict, Pleiades is TERMINAL → its `other` edge to 57034 isn't followed → the entire chain stays out.
- **Strict was preferred over one-hop after observing the Evangelion case.** An intermediate `ONE_HOP` design (the originally-proposed `WALK → ONE_HOP → TERMINAL` ladder) preserved legitimate deep side-stories (Pleiades Special Edition 51168, Sei Oukoku-hen Manner Movie 61345) but still allowed a direct `Main → other → Main` edge to drag the second franchise's full sequel chain into the first anime row. Evangelion's prod scrape was fetching 100+ mal_ids because Eva Main has direct `other` edges to Ultraman / Gunbuster mains under one-hop. Strict caps the leakage at exactly one node per direct `other` edge (the other franchise's Main as a single TERMINAL `side_story`-classified entry), which the admin can promote out via the future split-anime tool. Aligns with the "easier to merge than split" cost asymmetry: fewer false-positive members in the graph is strictly better than more.
- **What we lose under strict.** Nodes reachable only via a chain of two identity-breaking edges (Pleiades Special Edition 51168 via `Overlord → side_story → Pleiades → other → 51168`; Sei Oukoku-hen Manner Movie 61345 via `Overlord IV → side_story → Movie 3 → other → 61345`). These are deep, low-value franchise members (1-ep ONA compilations, 60-second PSAs). The nightly `update_sweep` step 1's `/full` refresh repopulates sidecar edges from MAL's full relation list regardless, so merge-time bridge restoration is unaffected.
- **Why `alternative_version` stays WALK.** [relation_classifier.py:225-236](../phsar/app/services/relation_classifier.py#L225-L236) builds the alt-chain by walking sequel/prequel/alternative_version edges from main-chain alt-neighbors. Demoting alt-version to TERMINAL would orphan downstream Rebuild Movies in Eva-shaped franchises (TV → alt_version → Movie 1 → sequel → Movie 2 → sequel → Movie 3 → ...).
- **Why `summary` is identity-breaking despite being intra-franchise.** Summaries are recaps, never the canonical chain; the classifier already labels them `summary` not `main`. They land in the graph from their parent's direct edge and the recap-chain stays contained — that's enough.
- **Status promotion: same parent, two edges.** When the same target is reachable from one WALK parent via both an identity-breaking and an identity-preserving relation, the most-permissive status wins via `max(existing, new_status)` in the expand dict. Without this a side-story-then-sequel arrival from the same walker would lock the target into TERMINAL and miss its sequel chain. Test: `test_search_title_status_promoted_when_two_edges_from_same_walker`. The cross-parent case (target reached via side_story from parent X before sequel-walked from parent Y) is NOT handled — once popped, the target stays at its processed level. Acceptable because MAL's reciprocal edge structure means structural parents typically queue their targets before side-story branches do.

### v0.14.1 notes

- **Pre-v0.14.1 catalog rows have FILTERED sidecar edges initially.** Old rows scraped under v0.14.0's dangling-edge filter store only in-graph edges. The nightly `update_sweep`'s step-1 refresh rewrites the sidecar from MAL's `/full` payload (unfiltered), so bridges are restored over the sweep cycle. The backfiller's lazy MAL fetch is the prime mover for empty sidecars; the sweep is the steady-state refresher. Force-refresh on the backfiller would speed this up for known split-franchise merge candidates — tracked in Future Work.
- **`media_to_classifier_node` (ORM projection) co-located with `build_classifier_nodes` (scraper-dict projection) in `relation_classifier.py`.** Two projections into the same `ClassifierNode` shape but from different sources (Media row vs scraper `all_info` dict). When adding a new field to `ClassifierNode`, update BOTH projections; co-location makes the omission obvious in review.
- **`update_sweep` step 1 also refreshes `MediaRelationEdges`** alongside the canonical-field diff. The `/anime/{id}/full` payload already includes a `relations` block (no extra MAL hit). The dispatcher writes the refreshed edge list to the sidecar; edge churn is NOT counted as `anime_changed` for stable-counter purposes (structural metadata, not the volatile canonical fields the counter tracks). Effect: pre-v0.14.1 catalog rows gain bridge edges over the nightly sweep cycle without needing a one-shot force-refresh, and MAL additions (new sequels, new alt-versions) flow in naturally.
- **Spoiler frontier expanded its anchor set** from `{Main}` to `{Main, AlternativeVersion}` in `_ANCHOR_TYPES`. Watching the canonical TV reveals Rebuild Movie 1 (first alt-anchor after the TV) but not Movies 2-4 — each Rebuild gates the next. Mirror in `frontend/src/lib/utils/spoilerFrontier.ts::ANCHOR_TYPES`.

## Future Work

- **Backfill relation_type for catalog rows scraped pre-fix.** The normalization in v0.14.0 corrects `relation_type` for new scrapes only — the sweep's relations probe doesn't rewrite the field. Existing franchise movies/OVAs stay labeled `main` until their parent anime is deleted and re-scraped. A one-shot reclassification migration would re-run `get_relation_type` against existing media graphs. Out of scope for v0.14.0 because it needs its own validation pass against the live catalog.
- **`is_main_story` should include ONA when no parent_story/full_story.** Today, ONA seeds fall through to the promote-root fallback. Making the gate accept ONA + no-parent-story would let donghua land in the regular `save_search_results` flow without needing the fallback. Same outcome, cleaner code path.
- **Track null-title skip frequency per mal_id.** If a mal_id keeps coming back null over months, MAL likely won't ever populate it. After N silent-skips, add to MediaUnwanted with reason=`"NullTitle"` to prevent infinite re-fetch. Threshold to be determined empirically.
- **Distinguish "Music PV linked from main TV show" from standalone Music/PV/CM.** The former is part of a franchise (worth attaching as media under the main anime); the latter is unwanted. Currently both go to MediaUnwanted. Heuristic: a Music/PV/CM reached via a Parent Story or Side Story relation FROM a non-Music main anime *might* be franchise media. Needs careful design — false positives pollute the catalog with promotional clips.
- **Surface `attached_count` in the bell.** Today `result_summary.attached_count` is only visible by querying the `jobs` table directly. For user-initiated scrapes that end up attaching to an existing anime (rare but happens), the bell shows the cryptic `{anime_count: 0, media_count: 0}` instead of `"Attached 1 media to existing anime"`.
- **MAL might support 429 `Retry-After` headers.** Currently we ignore the header and use tenacity's exponential backoff. Check if MAL/Jikan emits a header; if so, honor it for tighter recovery on a true sustained-throttle event.
- **Force-refresh mode for `relation_backfiller`.** The lazy MAL fetch only fills empty sidecars (`if sidecar.edges: return`). Catalog rows scraped under v0.14.0's dangling-edge filter retain incomplete sidecars (no bridge edges across previously-different-anime boundaries). A force-refresh flag would re-hit `/anime/{mal_id}/relations` for every media regardless of sidecar state, populating bridges retroactively. Cost: ~14min cold start at 1 req/s for an 800-media catalog. Worth running once after merging a known split-franchise case (e.g. before processing a Higurashi 2006 ↔ Gou pending candidate).
- **FMA 2003 ↔ Brotherhood inverse-merge (v0.15+).** `RelationType.AlternativeVersion` correctly handles within-franchise retellings (Evangelion) but collapses arguable-separate-production reboots (FMA, Higurashi 2006 ↔ Gou, Hunter × Hunter 1999 ↔ 2011) into one anime row. Admin tool to split an anime by picking a media → promote to its own row would handle these cases. Not in scope for v0.14.1.
