## Summary

Made `update_sweep` jobs auditable. Every refreshed media row now records its
field-level before/after diff in `result_summary`, queryable via a per-job
detail page at `/admin/jobs/[uuid]`. The audit log is the rollback path for
a new "apply everything" drift policy: genre + studio diffs (additions AND
removals) now land automatically, so the same MAL refinement no longer
re-surfaces on every nightly sweep. Unknown genre tags still skip (the
genre seed table is the deliberate source of truth) but are flagged on the
Jobs Log row so admin can seed them. Also: per-kind schema versioning on
`jobs` so future result_summary shape changes don't break historical rows;
JobWorker hardened to never strand a job in `running` after a downstream
crash; admin Overview gains a sweep-tier breakdown card.

## Failed approaches

- **Read-time enrichment of result_summary titles** (rejected mid-session). When the first post-v0.14.5 sweep showed romaji titles, I drafted backend code to overlay `Anime.name_eng/jap` + `Media.name_eng/jap` into the detail-page response by JOINing against current rows. The user redirected: it's a workaround. The dispatcher already writes the snapshot correctly (slice 6 commit `a238a22`); the existing sweep data was just from before that code landed. Triggering a new sweep was the honest fix. Lesson: don't pile on a presentation-layer JOIN when the underlying snapshot will self-correct on the next pipeline run.

- **Anime-name fallback cascade for media titles** (introduced in slice 6a, reverted in slice 6c). The cascade was `media.name_eng ?? anime.name_eng ?? media.title (romaji)`. Goal: render in viewer's language even when MAL doesn't supply media-level name_eng. Caught silently stripping season suffixes — "Dr. Stone: New World" rendered as "Dr. Stone" because the parent anime's `name_eng` covered only the franchise root. Reverted to `resolveTitle(media.title, media.name_eng, media.name_jap, lang)` which already falls back to romaji `title` (which carries the season suffix). Lesson: borrowing the parent's localized name discards specificity that the romaji string preserves by accident.

- **Conservative drift apply policy** (pre-v0.14.5, removed in slice 6c). Originally `_apply_genre_diff` only auto-applied pure addition of all-known tags; everything else (removals, mixed unknowns) was log-only. Outcome: new airing anime spammed the Jobs Log with the same drift on every nightly sweep because nothing was actually written. With the v2 audit log capturing every field change with provenance, silent rewrites became safe — rollback is "look at the relevant job's per-media diff." Unknown genre tags are the one carve-out (seed table is the curated taxonomy).

- **Single bundled commit for slice 1's `make_job` plumbing + slice 4's payload change** (caught + reset). My slice-6 commit accidentally bundled all the slice-6 frontend work because `git add` left files staged from a previous rejected commit. User reset; I re-staged with the explicit two-file scope for the bugfix. Lesson: when an earlier commit attempt is rejected, `git reset HEAD` the staged files before continuing.

## Key decisions

- **jobs.version is a per-kind integer, not a global semver.** Each JobKind has its own counter in `core/job_versions.py`. Update_sweep bumped 1 → 2 → 3 across this release; other kinds stayed at 1. Frontend dispatches on `(kind, version)`. Rationale: the result_summary shape change is per-kind; coupling all kinds to a shared semver would force unrelated bumps. Hardcoded dict, not a registry — KeyError surfaces a forgotten registration loudly.

- **Server_default=1 on the jobs.version column.** Migration backfills every existing row in the same DDL statement via `ALTER TABLE ... DEFAULT 1` (PostgreSQL 11+ fast path, no table rewrite). The default also guards against a future Job() construction site that forgets to use `make_job()` — the row still inserts cleanly with the safe legacy value. The runtime registry is the real source of truth; server_default is defence-in-depth.

- **5-tone visual scheme in MediaChangeCard.** `rating` (zinc/muted, sorts last on every card) splits out score/scored_by from the other dynamic fields because vote-count churn would otherwise drown the actionable bits (episodes/airing_status/aired_to). `static` (sky) for description/title/cover changes that move rarely but always matter. `genre` (fuchsia) + `studio` (indigo) for M2M drift. Tone-badge column was dropped after visual review — colored left border + field name carry enough signal. Rationale: a single column is cheaper than a row of badges and aligns better across cards.

- **Click-through restricted to update_sweep v2+ rows only.** The detail page is the v0.14.5 observability surface — it has nothing to add for backup/restore/seasonal_sweep/user_scrape rows whose payload summary is the full story. v1 update_sweep rows fall back to "predates v0.14.5" since their counters/diffs were never captured. Failed jobs also stay non-clickable (header's error banner is the actionable info, not a stale notice).

- **Pre-capture `str(job.uuid)` immediately after claim in JobWorker.** The original except path touched `job.uuid` in the failure logger — that triggers an expired-attribute refresh, which re-raises on a poisoned session. Captured identity as a plain string before the work session opens, so the failure path never reads ORM attrs again. Matches the pattern in `relation_backfiller.py:121-122` (`anime_id_for_log = anime.id` for the same reason).

- **`_jsonable` coercion at the diff-sink append site, not at JSONB-serializer level.** SQLAlchemy lets you override `json_serializer` on the engine, but that would apply project-wide and force every JSONB write to think about datetime. The crash was localized to one site (`aired_to` field in `_apply_media_diff`); fixing at the source keeps the rest of the codebase honest. Matches the project's pattern (`backup_service.py`, `export_service.py` both `.isoformat()` at the source).

- **Auto-create Studio rows on first sight; don't auto-create Genre rows.** Studios are a discovered taxonomy — co-production credits and outsourced animation studios genuinely surface after airing. Genres are a curated user-facing taxonomy with a deliberate seed list; silently growing it would shift search semantics under users without admin sign-off. Unknown genre tags fall into a new top-level `unknown_genre_tags` list so admin can decide whether to seed them.

- **Outer catch-all around the work_session block in JobWorker.** Three exception handlers now (inner dispatcher except + inner rollback try/except + outer catch-all). Each layer has distinct cleanup logic. Without the outer, an exception leaking out of `dispatch_one` would only get cleaned up by `reap_orphans` on next startup — meaning the bell shows a stuck `running` row indefinitely. Worth the nesting depth.

- **Frontend's `UpdateSweepM2MDrift['kind']` is a 6-value union spanning v2 + v3 vocabularies.** Old kinds (`additions_applied`, `additions_unknown`, `removal_or_replacement`, `any_change`) stay so historical v2 rows render with accurate "Not auto-applied" context. New v3 kinds (`applied`, `applied_with_unknowns`) reflect the relaxed policy. `NOT_APPLIED_REASON` Record keys all 6.

- **Workflow split for backend vs UI slices.** Backend: implement → /simplify → commit (tests cover correctness). UI: implement → user reviews in browser → fix visual feedback → /simplify → commit (/simplify doesn't catch contrast, spacing, broken hrefs, missing user-setting respect — the user does). Codified in `feedback_simplify_before_commit.md` memory.

## Gotchas & learnings

- **`<td class="truncate">` collapses every child to one line.** `truncate` is `overflow:hidden; text-overflow:ellipsis; white-space:nowrap`. A multi-line warning subline below the payload summary got silently clipped. Fix: move `truncate` to a specific `<div>` inside the cell, not the cell itself.

- **Drift detection runs on every sweep regardless of last sweep's result.** The diff is "MAL says X, our DB says Y". If both stay the same, the diff re-surfaces every refresh forever. Conservative log-only policy left admins with no resolution path — the same Yomi-no-Tsugai Fantasy→Supernatural drift would land in every Jobs Log row forever. Apply-by-default + audit-log-as-rollback breaks that loop.

- **Test pollution flake in full pytest run (did not reproduce at release).** Mid-development, backup-service tests passed in isolation but 7 failed inside the full suite — pre-existing ordering brittleness, never caused by v0.14.5 (slice 4's diff didn't touch backup code). The full `pytest` run at release green (623 passed), so the flake is ordering/environment-dependent rather than deterministic; left as a watch-item, not a fix.

- **`{:else if X}` block containing only an HTML comment is technically valid Svelte but fragile.** Slice 6 used this to "skip" the predates-v0.14.5 notice for failed jobs:
  ```svelte
  {:else if job.status === 'failed'}
    <!-- no notice; header shows error -->
  {:else if job.version < 2}
    <Notice>predates ...</Notice>
  ```
  /simplify flagged it; restructured to one boolean: `{:else if job.status !== 'failed' && job.version < 2}`. The comment-only branch silently shadowed the later branch in a way that wasn't obvious from the diff.

- **`dict() copy of TypedDict` adds nothing.** `dict(umbrella_diff) if umbrella_diff else None` in `_refresh_one_anime`'s `RefreshResult` builder was meant to be defensive. ReclassifyDiff is already a plain dict at runtime; the copy didn't protect anything and breaks `is None` identity checks. Removed.

- **Score / scored_by changes generate visual noise.** Popular anime accumulate vote-count drift every sweep — for One Piece et al, score=8.5 → 8.5 + 200 votes nightly. Folded into a separate `rating` tone (zinc, sorts last on each MediaChangeCard) and split out into its own filter chip ("Rating only" / "Dynamic w/o rating") so admins inspecting structural changes don't have to scroll past 50 vote-count rows.

- **`dispatch_one`'s prior failure path tripped a cascade.** Sequence: dispatcher returns OK → `mark_succeeded` flush fails (the JSONB datetime crash) → except block tries `logger.exception("Job %s ...", job.uuid)` → ORM refresh on the poisoned session re-raises → outer loop catches "JobWorker dispatch loop crashed" → job stays `running` forever. Three guards now: pre-capture identity, wrapped rollback, outer catch-all.

- **TypeScript narrows discriminated unions inside `{#each}` blocks correctly.** `DiffRow = ScalarRow | TagsetRow` with `kind: 'scalar' | 'tagset'` discriminator works inside Svelte each-blocks via `{#if row.kind === 'scalar'}`. The narrowing held for both branches accessing branch-specific fields.

- **`RefreshResult.genre_drifts` / `studio_drifts` were dead once per-media diffs landed.** Slice 4 moved the genre/studio drift report into each `media_changes` entry (`genre_drift` / `studio_drift` keys), but the two top-level accumulator lists kept being populated and returned — never read by the dispatcher. Caught in the PR review pass and dropped. Same shape as the `dict()`-copy nit above: a field that survived a refactor only because nothing errored on it.

## Trade-offs / open work

- **`unknown_genre_tags` is a sweep-level aggregate; no per-tag link to seeder yet.** Admin sees "⚠ New genre tags need seeding: Survival Game, Dark Fantasy, Mecha-Sci-Fi" on the Jobs Log row but has to manually edit the seeder file + redeploy. Future slice could add an admin endpoint to insert genres into the seed table from the UI.

- **DELETE M2M via `Genre.name.in_(removed)` runs one roundtrip per drift.** Per-media lookup of known genres / studios still incurs N roundtrips per sweep (one per drifted media). Batching across all media at sweep start would cut ~10-50ms; flagged as future optimization but not blocking.

- **No drift-rollback UI yet.** "Audit log is the rollback path" means admin can READ what changed but has to manually revert via DB script. If a real MAL data bug ever auto-applies wrong genres at scale, we need a one-click "revert this sweep's changes" action.

- **The 5-bucket sweep-tier breakdown is mutually exclusive by priority cascade, which is the user's preferred mental model.** The 4 OR'd tier predicates in `select_due_for_sweep` overlap (an airing anime is in tier 1 AND tier 2 if stabilizing). The Overview card uses a CASE WHEN cascade that picks each anime's highest-priority bucket, so the buckets sum to total anime count.

- **Failed-state transition coverage is asymmetric.** `JobWorker.dispatch_one` now guarantees every dispatch attempt ends in `succeeded` or `failed`, never stuck `running`. But the OUTER `_run` loop's catch-all still doesn't re-mark the in-flight job — if `dispatch_one` somehow raises despite the three guards, the row sits until `reap_orphans` at next startup. The defensive guards make this practically unreachable; flagged for awareness.
