# v0.14.3 — Third-pass split candidates (admin-reviewed BFS-leak repair)

**Date:** 2026-05-18 | **Branch:** v0.14.2-bug-fixes (v0.14.3 work landed on the v0.14.2 branch) | **Status:** ready to merge. Validated end-to-end on dev DB (see "Validation" section below).

## Summary

The v0.14.2 strict-BFS state machine (WALK / TERMINAL) capped Overlord+Eminence transitive contamination, but it does NOT prevent single-step franchise leaks where MAL labels a separate sub-franchise as `side_story` / `spin-off` / `parent_story` directly from a Main. Phase B B1 (Toaru Index → Railgun/Accelerator/Anbu no Item) and B7 (BNHA → Vigilante) made the leak shape concrete: TERMINAL nodes get walked-to-but-not-walked-from, so the discovered node ends up inside the parent's anime row.

This adds a third pass (`find_disjoint_franchises`) alongside the existing two-pass classifier — a pure, DB-less, structural detection function that flags clusters of substance-passing media which form their own sequel chains under the same anime row. Detection fires at three call sites (scrape, relation-backfiller per-anime loop, merge-survivor) and queues a `SplitCandidate` row for admin review. The admin UI exposes the queue + a `Split` button that re-parents the cluster's media into a fresh Anime row, preserving Media UUIDs so existing user ratings stay attached. The pre-prod-reset audit ([phsar/scripts/audit_cross_franchise.py](phsar/scripts/audit_cross_franchise.py)) refactors to use the same function, so audit = production spec.

Companion BFS fix: TERMINAL nodes now fetch `/relations` (previously skipped). The state machine still doesn't recurse from TERMINAL — only the discovered node's outgoing edges are captured into the sidecar. Without this, split-detection couldn't see e.g. Vigilante S1's sequel to S2 from a BNHA scrape, because Vigilante's sidecar landed empty.

**Companion cleanups** (post-simplify pass): extracted `summarize_anime` shared helper into `anime_summary.py` (was duplicated `_summarize` / `_summarize_source` byte-for-byte across the two candidate services); promoted `_build_classifier_graph` → `build_classifier_graph` since two non-defining-module callers now exist (audit script + split backfiller); folded `media_studio.studio` selectinload into `SplitCandidateDAO.list_pending_with_anime` so `list_pending` runs one query instead of two.

## Key Decisions

- **Admin-reviewed split, NOT auto-split.** The v0.14.0 compound docs explicitly defer "the split-anime tool" to a later milestone — splitting is destructive (reassigns which anime a rating contributes to in aggregations), so the new logic surfaces clusters and lets admin act. Mirrors the merge-candidate philosophy: detector flags, human resolves. The "easier to merge than split" invariant holds.
- **Pure post-classifier function with same (nodes, edges) signature.** `find_disjoint_franchises(nodes, edges, anchor) -> list[DisjointFranchise]` takes the classifier's anchor and runs structural cluster detection. No scrape-seed knowledge, no DB I/O — so the answer is identical at scrape, merge, and backfill time. Same anti-path-dependence discipline that motivated v0.14.0's two-pass refactor (Evangelion umbrella anchoring on Rebuild Movie 1 was the canonical bug).
- **TERMINAL nodes now fetch `/relations`, BFS still doesn't recurse.** State transitions unchanged: a `side_story` / `spin-off` / `parent_story` / `other` / `summary` / `full_story` edge from a WALK parent demotes the target to TERMINAL. The change is the gate at jikan_scraper.py L656-671 — the old "TERMINAL → all_related_media = []" branch is gone. The for-loop that queues targets adds an explicit `if current_status is TERMINAL: continue` so the graph stays bounded to nodes reachable via WALK propagation from the seed. Targets of TERMINAL edges become dangling refs in the sidecar (same pattern as cross-graph bridges, defensively filtered at `_build_adjacency` time).
- **Conan-exception rule: movie-only cluster + parent_story/summary bridge = legitimate side-story.** Phase A audit flagged 8 Conan movie pairs (Movie 20/22, Movie 24+summary, Movie 26+summary, Movie 27+summary) because consecutive Conan movies are sequel-chained AND each passes substance via duration. They're not contamination — they're legitimate side-story movies. The function tightens the rule to skip any cluster where (a) every member is a Movie AND (b) at least one bridge edge to the anchored set is `parent_story` or `summary`. Vigilante (TV-tier members) and Railgun (TV-tier members) are unaffected — they're not movies-only. Mononoke 2024 trilogy (`alternative_setting` bridge, dropped at BFS) has no `parent_story` / `summary` bridge so it still flags as a separate franchise (which is the correct call per the audit).
- **One pending row per anime, signature-based supersede.** `SplitCandidateDAO.upsert_pending` hashes the sorted-cluster-mal_id-lists as the natural key. Re-running detection on an unchanged catalog inserts zero rows. If new media absorbs into the contaminated row between detection runs, the existing pending row gets auto-dismissed and a fresh one is inserted with the updated payload — admin always sees the latest detection. Avoids needing a partial-unique-constraint migration (Alembic doesn't handle those cleanly on enum-status partial filters).
- **Detection fires at scrape, backfill, AND merge-survivor.** Same three-site discipline as the classifier. Scrape-time runs in `save_service.save_search_results` after `create_anime_from_media` lands the new anime row. Backfill runs as a per-anime call inside `relation_backfiller`'s existing SAVEPOINT loop AND as a standalone `backfill_split_candidates` pass at lifespan startup (after `relation_backfiller` so the TERMINAL sidecars are populated, and after `backfill_merge_candidates` so resolved merges don't surface duplicate candidates). Merge-survivor fires inside `merge_candidate_service.merge` after `reclassify_anime` — a consolidated A∪B graph may surface previously-dangling bridges that form a disjoint chain.
- **Rating safety via Media UUID stability.** `execute_split` re-parents Media rows (`UPDATE media SET anime_id = new.id WHERE id IN (...)`). Media UUIDs are unchanged. `Ratings.media_id` points at Media rows by PK, not by anime. A rating on Vigilante S1 stays on that exact media row after split — only the anime aggregation shifts. Locked in by `test_execute_split_preserves_media_uuids`.
- **Fail-loud on stale candidate at execute time.** The cluster's `suggested_anchor_mal_id` is recomputed from the cluster subset at execute time. If MAL data shifted between detection and execution (a new sequel landed, an existing entry got demoted), `classify_anime_relations(sub_nodes, sub_edges)` may pick a different anchor — raise `SplitCandidateStaleError(409)` so admin re-runs detection rather than landing under a different anchor than what the preview promised.
- **`db.expire(source_anime, ["media"])` after bulk-re-parent UPDATE.** The bulk UPDATE bypasses ORM relationship tracking; without the explicit expire, the source anime's cached `.media` collection still contains the moved rows. Subsequent `reclassify_anime` runs against stale data and the source's umbrella may not reflect its now-smaller media set. Same hazard exists in `merge_candidate_service.merge` where the surviving anime's `.media` is invalidated post-re-parent — handled there by deleting anime_b entirely.
- **Shared `summarize_anime` helper, public `build_classifier_graph`.** The merge + split admin cards render the same Anime → MergeCandidateAnimeSummary projection; extracted into `app/services/anime_summary.py` so a future schema addition (e.g. `airing_status` on the card) lands in one place. `build_classifier_graph` was private (`_build_classifier_graph`) until the audit script + split-candidate backfiller needed it from outside the defining module — three callers is the signal to drop the underscore.

## Failed Approaches

- **Auto-splitting at scrape time.** Initial sketch: `search_service` expands one BFS graph into N `SearchResultDB` instances when disjoint chains are detected, save_service iterates as usual, multiple anime rows land in one save call. Rejected because (a) splits are disruptive to user ratings; (b) cost-asymmetry: merging is cheap (re-parent + reclassify), splitting needs new anime rows + reclassify + spoiler-cache + embedding regen + post-detect re-runs; (c) re-introduces a scrape-time decision the user can't audit before it lands. Surfacing as a SplitCandidate keeps the human in the loop.
- **Single signature column on SplitCandidate with partial-unique-constraint.** Hashing the cluster payload into a stored column was tempting for idempotency, but partial-unique-constraints with enum-typed predicates (`WHERE status = 'pending'`) don't autogenerate cleanly in Alembic, and a generated column for the hash would mean a separate trigger or computed-column migration. Cheaper to enforce uniqueness in DAO logic (`get_pending_for_anime` + supersede), trading a single SQL query for a migration that pgvector + Postgres 15 do support but our dev/prod stack would need explicit testing on.
- **Returning the full DisjointFranchise dict shape through Pydantic.** Pydantic doesn't smoothly serialize TypedDict-typed fields; the SearchResultDB schema would need a parallel BaseModel wrapper. Skipped — the JSONB column stores raw dicts, and the admin response uses explicit `SplitClusterPreview` schemas built in the service layer.
- **Calling `find_disjoint_franchises` inside `search_service` AND passing it through to `save_service` only to call `upsert_pending`.** The wiring is awkward — detection result rides on `SearchResultDB.disjoint_franchises` as `list[dict]` through the schema → save loop → DAO call. An alternative was running detection in `save_service` after the new Anime row lands, re-deriving (nodes, edges) from the just-persisted media. Considered cleaner, but it'd re-walk the media + sidecars one more time. Carrying the precomputed result through the schema is cheaper and keeps detection co-located with classification.
- **Dangling-edge heuristic for scrape-time singleton orphans.** When a re-scrape of BNHA failed to surface a split candidate (Vigilante S1 sits alone in BNHA's anime row because TERMINAL doesn't recurse, so S2 is never fetched), the proposed fix was to augment `find_disjoint_franchises` to count "substance-passing orphan with outgoing sequel/prequel edge to a dangling target" as chain evidence. Almost shipped. Rejected after empirical validation: the natural workflow handles this — scrape Vigilante separately → MAL detects relation_link merge candidate → admin merges Vigilante into BNHA → `merge_candidate_service.merge` post-reclassify hook fires `detect_split_candidates_for_anime` on the survivor → SplitCandidate emerges with both S1+S2 now in-graph → admin executes split → 2 clean anime rows. The architecture self-heals through the merge-then-split loop, no specialized rule needed. See "Validation" section.

**Failure modes verified:**
- Stale-candidate fail-loud: not triggered in practice (no MAL data shift during the test window) — `test_execute_split_creates_new_anime_and_reparents_media` covers the assertion path.
- Idempotent re-runs: `backfill_split_candidates` re-run on unchanged catalog returned `inserted=0`. Supersede semantics not exercised (no payload changed between runs).
- Rating safety: locked in by `test_execute_split_preserves_media_uuids` — Media UUIDs stable, Ratings stay attached.

## Gotchas & Learnings

- **TERMINAL fetch increases per-scrape MAL cost by ~15-25 calls (~15-25 s at 1 req/s).** Multi-franchise scrapes (Pretty Rhythm, Toaru, BNHA) hit the upper end; lean franchises (Re:Zero, donghua sub-universes) hit the lower. Still well inside the bell-tracked job window. Update_sweep step 1 was already doing the equivalent via bundled `/full` relations, so steady state was already paying this — only scrape-time was new cost.
- **The natural workflow handles dangling-singleton orphans without specialized rules.** When admin scrapes a sibling franchise of a contaminated row (Vigilante for BNHA, Railgun for Toaru if not absorbed via Index's side_story edges), MAL's relation_link triggers a merge candidate. Admin merges → the merge-survivor reclassify path fires `detect_split_candidates_for_anime` on the consolidated graph → SplitCandidate emerges with both partners now in-graph. The lesson: when a detection rule misses a shape, check whether an adjacent workflow already surfaces it before adding a special case. This kept the function signature pure and avoided re-introducing seed-relative reasoning at scrape time.
- **Post-split BNHA ↔ Vigilante title_studio merge candidate is expected noise.** "Vigilante: Boku no Hero Academia Illegals" contains "Boku no Hero Academia" → SequenceMatcher containment fires, studio overlap holds. Admin dismisses once; `seen_pairs` keeps it dismissed. A future "do not re-flag pairs that were just split apart" rule could suppress this — deferred, low-priority polish.
- **The audit script is now the test spec.** Refactoring `phsar/scripts/audit_cross_franchise.py` to call `find_disjoint_franchises` directly means any regression in the production rule (substance-gate tweak, Conan-exclusion drift) surfaces immediately in the audit output before manifesting as actual bad data. Future tightening of the rule should always be paired with re-running the audit + diffing against the locked-in JSON snapshot.
- **`alternative_setting` lives on as a graph boundary even post-v0.14.0.** It's dropped at BFS parse time (`_EXCLUDED_EDGE_RELS`), so a fresh scrape never sees those edges. But pre-v0.14.0 sidecars persisted them, so old contaminated rows (Pretty Rhythm via PriPara, Love Live μ's via Nijigasaki) still show the edges in their sidecars. Splitter operates on the post-classifier graph; `_build_adjacency` filters `alternative_setting` out via the `_EXCLUDED_EDGE_RELS` set in `parse_relation_edges`, so the splitter sees the cluster as bridge-free. Mononoke 2024 trilogy under Ayakashi follows this shape — `alternative_setting` bridge invisible at detect time, no `parent_story` bridge → flagged correctly.
- **`expire_on_commit=False` in test fixtures bites bulk-UPDATE patterns.** The test session's identity map serves cached `.media` collections after a `db.commit()` even though prod sessions would expire them. The explicit `db.expire(anime, ["media"])` in `execute_split` is load-bearing in both test and prod — but test-failure-on-first-write is what surfaced the issue. Mirrors the same pattern in `merge_candidate_service.merge` (which sidesteps by deleting anime_b entirely after re-parent).
- **Conan false positive was the most informative test case.** Before the rule tightening, the audit flagged 6 contamination cases including 4 false-positive Conan movie-summary pairs. The exclusion logic (`_is_legitimate_side_story_chain`) is small (~10 LOC) but lock-in via `test_conan_movie_chain_attached_via_parent_story_not_flagged` ensures a future "let's also flag movie-only chains" tweak triggers a clear failure.

## Future Work / Known Follow-ups

- **Per-detector cluster-payload schema versioning.** `SplitCandidate.clusters` is JSONB with the current DisjointFranchise shape. If the shape evolves (e.g. adding `aired_from` for ordering display), old pending rows still hold the v1 shape — `list_pending` reads via dict.get so missing fields default cleanly, but a future schema migration may need a version tag.
- **Backfill bell visibility.** `backfill_split_candidates` at lifespan startup runs silently. If the catalog grows to many thousands of anime, the few-second pass may matter for cold-start latency. Wrap in a job kind + bell entry?
- **Frontend optimistic-update on split.** Currently the SplitCandidatesCard does a full `fetchCandidates()` after a successful split — refetches the whole list to reflect the now-resolved candidate dropping out plus any post-split fresh detections. Acceptable at current row counts (audit shows 4 contaminated dev rows); at scale, the bell-style optimistic-pruning + reconcile pattern in [phsar/frontend/src/lib/stores/jobs.ts](phsar/frontend/src/lib/stores/jobs.ts) would smooth the UI.
- **Bridge-edge visualization in the admin UI.** The current card shows the count + type of bridge edges per cluster. A small graph viz (e.g., the [phsar/frontend/src/lib/components/RelatedMediaCarousel.svelte](phsar/frontend/src/lib/components/RelatedMediaCarousel.svelte) shape) would make "which MAL relation caused this contamination" visible at a glance. Deferred until the cluster-resolution flow is exercised against more shapes.
- **Audit-as-spec test.** The plan called for `tests/scripts/test_audit_cross_franchise.py` that imports the script's `audit()` function and runs it against an in-memory test DB. Deferred — the regression test corpus in `test_relation_classifier.py` already pins the underlying function, and the audit script is a thin wrapper.
