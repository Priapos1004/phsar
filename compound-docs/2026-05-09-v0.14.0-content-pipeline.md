# v0.14.0 — Content Pipeline (in progress)

**Date:** 2026-05-09 → 2026-05-10 | **Branches:** v0.14.0-content-pipeline (commits 1–6 + F1–F4 + docs) → v0.14.0-update-sweep (commits 7a + 7b) | **Status:** 7a + 7b landed; 7c next

## Summary

User-triggered anime scraping moves from a synchronous in-request BFS to a background-job model. A single asyncio worker drains the new `jobs` table FIFO; each `user_scrape` job runs the existing `handle_search_mal_api_results` + `save_search_results` pipeline behind the scenes and reports incremental progress to the row. The frontend gets a navbar bell that surfaces session-scoped active + unseen-completed counts, plus a `/library/add` page with a query input and a recent-additions grid. Commit 6 adds the duplicate-anime detector (three signals: title containment + studio gate, title hint + description-embedding cosine, BFS relation cross-link), an admin merge-candidate review surface, a startup backfiller, the crossover BFS boundary, and a `PermanentPhsarError` marker that gates the bell's retry button on permanent failures. Commit 7a (the maintenance-window foundation: freshness sidecars, banner countdown, worker maintenance bracket) and 7b (the `update_sweep` dispatcher + tier query + threshold + CORS bug fix + UX hardening) follow on the `v0.14.0-update-sweep` child branch. 7c (relations probe + coalesced spoiler recompute) and 8 (seasonal sweep) close out v0.14.0.

## Key Decisions

- **Single global FIFO worker, not per-user queues.** MAL has a global ~3 req/s rate limit so concurrent scrape jobs would just fragment bandwidth and fight over the dedupe set in `search_title.visited_ids`. Per-user submission cap (`JOBS_PER_USER_LIMIT=4`) keeps one user from monopolizing the queue. The worker uses an `asyncio.Event` for sub-second pickup with a 60s wall-clock fallback in case a `notify()` is lost to a race — short enough to self-heal within a minute.
- **Sequential per-job sessions, not nested.** The dispatcher's failure path used to open a `fail_session` while `work_session` was still in scope. asyncpg's small default pool deadlocked under that pattern in tests (`InterfaceError: another operation is in progress`). Now: try in `work_session`, capture exception, exit that session, open `fail_session` afterward. Sequential sessions also avoid identity-map pollution when a save spans minutes.
- **`engine.dispose()` per test, not just at suite teardown.** Each pytest-asyncio test gets a fresh event loop. Connections pooled in the global engine are bound to whichever loop opened them — once that loop is gone, asyncpg refuses to use them with the same `another operation is in progress` error. An autouse `_reset_engine_pool` fixture in the worker tests disposes pre-yield (post-yield is redundant — the next test will dispose again).
- **Partial index on `(created_at) WHERE status='queued'`.** Instead of a 3-column composite. The worker's claim query walks this index in FIFO order; the `not_before_at` filter is a recheck on the (small) result set. The index excludes finished jobs entirely so it stays tiny no matter how much history accumulates. `not_before_at` isn't in the index because at our queue depth (~6 rows max in production) the recheck cost is irrelevant.
- **UUID-based "seen" tracking on the bell, not timestamp.** Originally compared `finished_at` (server time) to `lastOpenedAt` (client `Date.now()`) — any client clock skew above zero kept finished jobs permanently flagged as unseen. Switched to `phsar.bellSeenJobs` set in sessionStorage. Server-time-independent.
- **Session-scoped bell + permanent recent-additions list on `/library/add`.** The bell only surfaces jobs created since the user's tab loaded (`phsar.bellLoginAt` in sessionStorage). Long-term history lives on `/library/add` instead. Justification: the bell is a notification surface for "things happening now", not an audit log.
- **`onBump(store, fn)` helper for cross-component refresh.** Two stores in `lib/stores/jobs.ts`:
  - `jobsRefresh` — bumped by `/library/add` after a successful POST so the bell refetches in tens of milliseconds instead of waiting for the 30s idle poll.
  - `librarySaved` — bumped by the bell when it observes a NEW succeeded `user_scrape` (UUID-tracked via `announcedSavedUuids` so it fires exactly once per job) so `/library/add` refreshes its recent-additions panel without a manual reload.
  Both consumers use `onBump` which subscribes-and-skips-the-initial-fire so the mount-time fetch isn't double-fired.
- **`ProgressReporter` in its own module, not inside the dispatcher.** Lives at `app/services/progress_reporter.py` so `jikan_scraper` and `save_service` can import it without depending on the dispatcher (which would invert the layering). Each `update()` opens a fresh short-lived session so the bell sees mid-flight progress before the dispatcher's main work tx commits, with **0.5s throttling** + `force=True` bypass for stage transitions — without throttling a tight BFS loop opens hundreds of sessions per second.
- **No anime picker on `/library/add`.** Initially built one (form sent `mal_id`) before realizing connected MAL search results dedupe naturally through the BFS visited-set in `search_title`. The 3-candidate search exists for query imprecision, not user choice — connected matches collapse to one anime row, unconnected matches each become their own. Saving everything matches the original `/save/search-results` flow and removes the picker UI entirely.
- **Recent additions are global, bell is per-user.** Family-shared catalog: when one member scrapes Naruto, others see "Naruto was added" on `/library/add`. The bell stays per-user because it tracks YOUR actions/notifications.
- **Dedupe at enqueue, 72h window, failed-status excluded.** Re-running the same query (case-insensitive, trim-normalized) within `JOBS_DEDUPE_HOURS` returns 409 — the BFS would just see every mal_id in the excluded set and fail with `AnimeNotFoundError`. Failed jobs DON'T count toward the dedupe so a transient MAL outage doesn't lock a user out for 3 days. The 72h window matches the seasonal-update cadence (commit 8): after 3 days new releases may have landed, so a re-scrape can produce new results.
- **Restricted users see the bell + `/library/add` page, just disabled.** Mirrors how `/ratings` handles the read-only role. The page itself reads `userRole` from context; `POST /jobs/scrape` rejects via `require_user_or_admin` regardless. Drove the navbar to drop its `isRestricted` prop entirely — no role-gated nav items.
- **`MainMediaNotFoundError` shortened + caught per relation graph.** Was leaking a Python list repr (`Main media not found in the list: [('Goulart Knights...`) into the bell UI. Now: `"Couldn't identify a main story for '{first_title}'. Try a more specific search term."`. Full title list still on `self.title_relation_tuple` for server logs. `search_mal_api` catches per graph and skips, so one weird MAL graph alongside good ones doesn't fail the whole scrape — partial results land.
- **Tenacity tuned to `wait_exponential(2, 1, 30)` × 5 attempts.** Better tail behavior on transient MAL outages, especially during overnight unattended runs (relevant for commits 7–8). Previous fixed-1s × 3 surfaced flaps as failures within seconds.
- **Min query length 4 chars** at both backend (`Field(min_length=4)`) and frontend (`minlength` attr + `disabled` until 4 chars typed). Shorter queries are ambiguous on MAL ("fma" hits Fullmetal Alchemist + dozens of unrelated entries) and waste a job slot.
- **`AnimeRecentItem` carries `name_eng` + `name_jap`** so the frontend can call `resolveTitle(...)` with the user's `name_language` setting without a second fetch. Recent additions previously displayed only the romaji `title` regardless of preference.
- **`/library/add` route over empty-state on `/search`.** The plan originally proposed an empty-state CTA on `/search` ("Can't find this? Add via MAL") — vetoed because vector search rarely returns truly empty even on misses.
- **Stage labels capitalized + new "Done" terminal stage.** "Fetching" → "Saving" → "Done" with `force=True` writes at each transition. Without "Done", the bell briefly showed "Saving — 7/7" before the row flipped to `succeeded` — read as stuck. "Done — 7/7" reads correctly even if it's only visible for a poll cycle.
- **Test queries use `_q()` with UUID suffix.** The dedupe check is global so any pre-existing job in the dev/CI DB with the same query collides. `_q()` returns `test-{uuid.hex[:8]}` so tests are robust to leftover data from manual app usage.
- **`onBump` subscribes via plain `store.subscribe()`, not `$effect`.** Runes only work in `.svelte` files. The helper lives in `lib/stores/jobs.ts` (.ts) and returns the unsubscribe function for the caller to wire into `onDestroy` or an `$effect` cleanup. Same code reads cleanly at every call site as `$effect(() => onBump(store, fn))`.

### Commit 6 — Duplicate detection + admin merge UI + crossover BFS boundary

- **Three detectors, not one.** Original sketch was just `title_studio` (SequenceMatcher ratio ≥ 0.85 + studio overlap). Real-world testing on the local catalog showed Dr. Stone vs Dr. Stone: New World scoring ratio ≈ 0.62 (length-symmetric ratio over-penalizes the longer subtitle), so it was missed entirely. Two changes followed: (a) **title score becomes `max(ratio, containment)`** where `containment = longest_match_size / min(len)` — catches the `X ⊂ X: Subtitle` family at 1.0 without needing aggressive normalization. (b) Add a second rule **`title_desc`**: weaker title floor (≥ 0.5) plus description-embedding cosine ≥ 0.85, also studio-gated. Catches Dr. Stone reliably (containment 1.0, descriptions both mention Senku/Stone World). Description embeddings already exist on `AnimeSearch` so this is free data.
- **Relation-link signal trumps title heuristics.** When BFS during a `user_scrape` traverses to a media that already exists in the catalog under a *different* anime via a non-crossover relation, MAL itself is asserting the connection — flag with `detected_by="relation_link"` regardless of title or studio. To produce this signal the BFS had to **split `excluded_ids` (frozen pre-existing catalog) from `visited_ids` (this run)** — previously they were the same set, so cross-links were silently swallowed by the visited-skip. The signal is captured per-graph (`search_title` returns `list[(graph, set[mal_id])]` instead of `list[graph]`), then resolved to owning anime ids in `save_service` while the new anime row is in scope.
- **Crossover BFS boundary skip.** When `current_relation == "crossover"`, record the anime in the graph but skip its `fetch_relations` call entirely — naturally separates franchises like Fate × Tsukihime that share a single crossover OVA. Also a small MAL request budget win.
- **Same-tx detection in save_service.** Detection runs before the final `db.commit()`, not in a separate session. Atomicity matters: if detection has a bug and rolls back, the save rolls back too. Acceptable trade — detection is a few SELECTs + INSERTs, not minutes of work.
- **Backfiller covers the existing × existing pair sweep at startup.** `merge_detection_service.backfill_merge_candidates` runs in lifespan after `backfill_embeddings`. First restart after upgrade flags real duplicates already in the catalog; subsequent restarts are near-free thanks to the `seen_pairs` short-circuit.
- **`seen_pairs` pre-fetch is the optimization that survives admin decisions.** Detection (and backfill) pre-fetches every existing `(anime_a_id, anime_b_id)` pair into a set and skips them before any similarity work. Admin's `pending`/`dismissed`/`merged` decisions are status-agnostic for this filter — once a pair has a row, it never re-flags. Without this, dismissed pairs would re-appear on every restart.
- **Cascade FK kills the `merged` audit row.** `MergeCandidate.anime_b_id` has `ON DELETE CASCADE`, so when the merge service deletes anime B the candidate row is cascaded away in the same statement. Decision: don't try to set `status=merged` post-delete — accept the row is gone (cascade also handles other pending pairs that referenced B). Only `dismissed` candidates persist as audit history.
- **Fail-loud on shared `Media.mal_id` between A and B in merge.** Pre-flight SELECT each anime's mal_ids, intersect, raise `MergeMalIdConflictError` (409) if non-empty. `Media.mal_id` is globally unique so this is a should-never-happen invariant; quietly resolving it would mask a real data bug.
- **Bell retry gating via `PermanentPhsarError` marker class.** User raised that the bell was spamming retries on `AnimeNotFoundError` failures (which are deterministic — same query → same MAL → same empty result). Class hierarchy approach: `PermanentPhsarError(PhsarBaseError)` marker, with `AnimeNotFoundError` / `MainMediaNotFoundError` / `MalIdAlreadyExistsError` re-parented under it. `JobWorker` does `retryable = not isinstance(failure, PermanentPhsarError)` and stashes in `result_summary["retryable"]` (no schema migration). Bell hides the retry button when false; also disables the button across all rows while a retry is in flight (prevents fan-out).
- **Bell session-storage cleanup on logout.** sessionStorage survives a hard nav inside the same tab, so without explicit cleanup a re-login inherited the previous session's `bellLoginAt` timestamp + `bellSeenJobs` set — the user saw "old" jobs and the badge stayed cleared. `clearBellSession()` lives next to the keys in a small `bell-session.ts` module; the `auth.ts` token subscriber calls it whenever the token transitions to null. One spot covers every logout path (button, 401, maintenance redirect, account deletion).
- **Bell dropdown capped at 5 entries** with a "+N more in your library activity" tail link. Per-user active-job cap is 4 (`JOBS_PER_USER_LIMIT`) so this only kicks in once a session has accumulated finishes. Badge math still uses the full visible-jobs list — count stays honest.

### Commit 7a — Maintenance-window foundation

- **Sidecar tables, not inline columns on `anime` / `media`.** Initial sketch put `last_checked_at` and `stable_check_count` directly on the canonical rows. Pushed back: those tables are user-facing catalog data, freshness is *operational state*. Sidecar tables (`anime_freshness`, `media_freshness`) keep sweep cadence from leaking into Pydantic schemas via accidental `model_dump()` and stop the catalog from widening with every new sweep heuristic. Migration `cb6055f6a525` backfills every existing row to its parent's `created_at` so anime enter the rotation at honest age rather than "brand new".
- **`BaseModel` over bare `Base` for the sidecars.** Initial pass used `Base` directly (matching `user_visible_media`'s cache-table pattern). User pref'd `BaseModel`: created_at / modified_at help debugging when sweep behavior diverges, the `id` PK keeps rows addressable for ad-hoc admin queries, and the cost is two int columns nobody reads in the hot path.
- **Process-wide `_active` + `_scheduled_at` as module globals.** Single-worker assumption explicitly documented in `core/maintenance.py`. If we ever scale horizontally this becomes a file sentinel or a DB row, but not in v0.14.0. The two are paired — when the worker finishes a sweep it both clears `_active` and clears `_scheduled_at`, so the banner has nothing useful to say once the window passes.
- **`MaintenanceBanner` polls via raw `fetch`, not `api.ts`.** A 503 response with `{maintenance: true}` would otherwise trigger api.ts's global redirect to `/login?maintenance=1`, defeating the pre-warning point. The status endpoint is allowlisted in the maintenance gate so it returns truthful info during a window too. Authless because the endpoint reveals nothing sensitive — the banner is for the user's benefit, not a permission check.
- **Change-guards on `$state` writes in the banner.** Svelte 5 `$state` setters notify subscribers even when the value is unchanged — `if (active !== body.active) active = body.active` keeps a 60s poll from cascading through `$derived` chains every minute for no reason.

### Commit 7b — update_sweep dispatcher (data-refresh path)

- **Per-anime commit boundary, per-anime try/except.** The user explicitly asked for a "robust setup" — initial sketch had a single end-of-sweep commit. That meant a worker crash at anime #198 of 200 would lose every refresh. Rewritten so each anime is its own atomic refresh unit: a worker-process kill at anime N preserves N−1 successful refreshes, AND a single bad MAL response (404, malformed payload, asyncpg blip) fails *that* anime only — the loop continues, the job still succeeds with `anime_refreshed < total`. Pairs naturally with `JobDAO.reap_orphans` on lifespan startup: a `--reload` mid-sweep is a non-event.
- **Tier 3 narrowed from "any main media exists" to "latest main aired within 5 years".** Original plan would have weekly-probed every main-only anime forever, including 19-year-old franchises with no realistic sequel announce path. `SWEEP_RECENT_MAIN_YEARS = 5` is a module constant — promote to settings only if real data shows the cutoff is wrong. Older main-only franchises still get tier 4 (180-day long tail) and the always-on tier 2 (`stable < 3`).
- **`save_search_results` creates `AnimeFreshness` + `MediaFreshness` on insert.** Originally the migration backfilled existing rows but new `user_scrape` saves landed without sidecars, leaving the LEFT JOIN + COALESCE in the tier query as load-bearing rather than belt-and-braces. Now every catalog row is born with its sidecar (`last_checked_at=None`, honest: nothing has *checked* this row from MAL yet — the row IS the check). Defensive on-demand sidecar creation in the dispatcher's `_refresh_one_anime` covers any pre-7b row that escaped.
- **`refresh_anime` reads `/anime/{id}/full`, not `/anime/{id}` + `/anime/{id}/relations`.** Single-call replacement returning relations in the same response. 7b ignores the relations block; 7c will read it from the same payload — the dispatcher won't need to double the per-media MAL hit rate to add relation probing.
- **`AIRING_STATUS_CURRENTLY_AIRING` and `parse_mal_datetime` extracted module-level on `JikanScraper`.** Shared between the scraper's BFS, the DAO's tier query, and the dispatcher's diff. Dropped a duplicated `value.replace("Z", "+00:00")` parse + a stringly-typed `"Currently Airing"` literal that lived in three places.
- **Cron token allowlist for `/admin/jobs/schedule-sweep`.** Without it, a cron retry while a sweep was already running would hit the maintenance gate's 503 and tomorrow's sweep would never get scheduled. The endpoint is cheap (one row insert + flag set) and idempotent enough that a double-fire just enqueues a second sweep behind the live one.

### Commit 7b — Threshold for score+scored_by drift (`3c8aa28`, post-launch fix)

- **Bundle score and scored_by into a weighted-delta signal.** Without this, every popular anime (One Piece, BNHA, Naruto:Shippuuden — millions of votes each) would never stabilize: a single new vote per night moved `scored_by`, the diff loop flagged it as "changed", the counter reset to 0, the anime stayed in tier 2 forever. Verified in real catalog data — 200 media × ~24h drift produced 0 above-threshold changes once the threshold landed. The fix is `_weighted_score(score, scored_by) = score * log10(scored_by + 1)` (the same formula already used by search ranking in `anime_dao` / `media_dao` — keeps the formula's interpretation consistent across the codebase) and counts the pair as stability-resetting only when `abs(new − old) >= 0.05`.
- **Always write through, threshold gates the *signal* not freshness.** The new score / scored_by values land in the DB regardless of the threshold result. The threshold only decides whether the stability counter resets. Confused for the user the first time they read it — added a comment block to be unambiguous.
- **None↔value transitions bypass the threshold.** First votes coming in (score `None → 8.0`) or scored_by going `0 → N` is a structural transition, not noise. The XOR check `(old_weighted is None) != (new_weighted is None)` reads cryptic but covers all the boundary cases without listing them out.
- **Calibration math.** At >100k votes (catalog p95), +0.01 score = 0.05 weighted Δ — exactly at threshold (catches it). +50 vote drift at 100k = 0.002 weighted Δ — well under. So real-world weekly drift on popular anime is filtered as noise while small genuine score shifts still trip the signal.

### Commit 7b — CORS headers on maintenance 503s (`6e230ad`, post-launch fix) **🎯 LANDMINE**

- **`app.add_middleware()` *prepends* to `user_middleware`.** This is the trap. Reading code top-to-bottom, "register CORSMiddleware first, then maintenance gate second" *intuitively* puts CORS on the outside. **It does the opposite.** Starlette inserts at index 0, then iterates `reversed(user_middleware)` when building the stack — so the LAST registered middleware ends up OUTERMOST. CORS first → CORS innermost; CORS sees the response only AFTER the maintenance gate already short-circuited. No `Access-Control-Allow-Origin` header gets stamped on cross-origin 503s. Fix: register `MaintenanceGateMiddleware` FIRST in code, CORSMiddleware SECOND. The comment block in `main.py` documents this so the next refactor doesn't undo it.
- **`fetch()` rejects with TypeError, not a Response, on missing CORS headers.** Cross-origin 503 with no `Access-Control-Allow-Origin` → browser blocks the response → `fetch()` throws TypeError before producing a Response object. The frontend's catch only recognizes our own `ApiError`; the TypeError fell into the generic `else` branch, surfacing as "An unexpected error occurred." instead of the maintenance banner. Two compounding bugs that masked each other — the surface symptom looked like a frontend issue, the root was a backend middleware-order misconception.
- **Pure ASGI middleware over `@app.middleware("http")` (BaseHTTPMiddleware).** BaseHTTPMiddleware short-circuit responses interact awkwardly with CORSMiddleware's send wrapper — pure ASGI sends through the wrapped `send` cleanly, CORS intercepts on `http.response.start`, headers get stamped. The conversion was a ~15-line code change but eliminates the whole class of "middleware-stack ordering misbehaved" bugs for this gate.
- **Regression test pins the invariant.** `test_503_carries_cors_headers_for_cross_origin_requests` flips `set_maintenance(True)` and asserts the 503 from `/auth/login` carries `Access-Control-Allow-Origin: http://localhost:5173`. Cheap test, catches any future middleware reorder.

### Commit 7b — Maintenance window UX hardening (`9664494`, post-launch fix)

- **Translucent yellow tints only read on light surfaces.** Initial banner used `bg-yellow-500/10` (10% opaque yellow) with `text-yellow-900` (dark yellow). On the page's dark purple body gradient that rendered as dark-yellow text on a barely-visible yellow tint on dark purple — unreadable. Solid `bg-yellow-50` cream surface fixes it on any backdrop. Same palette already used by `BulkRateDialog`'s yellow notice — shared via the new `Notice.svelte` component so future yellow banners don't re-derive the right palette by trial and error.
- **Sticky wrapper around banner + navbar.** Originally the banner scrolled away with the page — users mid-scroll never saw the maintenance warning. Wrapping `<MaintenanceBanner />` and `<NavBar>` in a single `sticky top-0 z-50` container keeps both pinned. NavBar's own `sticky top-0 z-50` was dropped (parent handles it). Considered making each independently sticky with stacking z-index — they'd overlap rather than stack vertically because `position: sticky` resolves at top:0 independently per element.
- **`bumpMaintenanceRefresh` writable + api.ts hook.** `MaintenanceBanner` polls every 60s, plus subscribes to the auth `token` store so a login transition triggers a fresh fetch. But there's a gap: a user submits the login form during a sweep that just ended — `api.ts` calls `token.set(null)` after the 503 (via the maintenance redirect path), but `null → null` isn't a transition so the subscriber doesn't fire. The new `lib/stores/maintenance.ts` writable + `bumpMaintenanceRefresh()` helper is bumped by `api.ts` on every 503-with-maintenance response; the banner subscribes via `onBump` (skip-initial) and refetches in ms. Two subscriptions cover the orthogonal cases: token transitions (login *succeeds* post-window) and explicit bumps (login *fails* during window).
- **Drop /login's inline maintenance banner.** Originally /login had its own `<Notice>` for the 503 case ("Maintenance in progress. Login will be available again once it completes."). Once the global banner became sticky AND reactive (refresh-bump), the inline became redundant double-clutter. The login form catch is now silent on 503 (`if (err.status !== 503) error = err.detail`) — the global banner conveys the state.
- **Anime-themed copy + ~prefix on countdown.** "Save your work" reads corporate; "pause your current episode" matches the app's voice. The `~` softens the precision — "starts in 1 minute" set false expectations because of poll latency + the `Math.round` in the minutes-until calculation, "starts in ~1 minute" reads more honestly.



- **Picker UI for the chosen MAL anime.** Built it (form sent `mal_id`, dispatcher filtered `search_mal_api` results to just that one) before realizing connected candidates dedupe naturally. Net waste: ~1 hour of code that got reverted.
- **`join_transaction_mode='create_savepoint'` for worker tests.** Hit asyncpg's savepoint nesting limitation under the worker's per-job session pattern (`InvalidSavepointSpecificationError: savepoint "sa_savepoint_2" does not exist`). Switched to "explicit cleanup" via a `tracked_jobs` fixture that DELETEs at teardown.
- **Per-user dedupe (initial design).** The check is global because the catalog is shared. A per-user dedupe would just spam the same query from a second user and re-fail.
- **Comparing `finished_at` to `lastOpenedAt` (client clock).** Clock skew kept the badge stuck. UUID set replaced it.
- **Fixed 5s `setTimeout` to refresh recent-additions after submit.** Brittle: scrapes that take longer (most do — embedding generation alone is several seconds) refresh before the data lands. Replaced with `librarySaved` event-driven refresh.
- **Single `db.commit()` inside `JobDAO.reap_orphans`.** Combined with the conftest's connection-bound test session, the commit confused the rolled-back outer transaction. Made `reap_orphans` caller-commits instead — matches the rest of the codebase's DAO pattern.
- **Bell badge as just `activeJobs.length`.** Missed the case where the user submitted a job, walked away, and came back to find it succeeded — no signal that anything happened. Added the unseen-completed component to the count.
- **`title_studio` alone as the only detector.** Original commit-6 plan. Caught Naruto/Naruto Shippuuden but missed Dr. Stone/Dr. Stone: New World — `SequenceMatcher.ratio()` is length-symmetric and the "+ : New World" suffix dropped the score to ~0.62. Adding containment (one extra line: `find_longest_match.size / min(len)`) lifts that case to 1.0. Without containment the only knob is lowering the threshold, which would have flooded admin with false positives.
- **Pure-Python `_cosine_similarity` loop** (initial version). Pure list zip + accumulator over 384-dim vectors × O(catalog²) backfill = ~5–15s of pegged-CPU lifespan time at 500 anime. Replaced with `numpy.dot` / `numpy.linalg.norm`; numpy ships transitively via sentence-transformers so no new dependency.
- **Tuple-index passing through `_flag_if_similar`** (`a[0]`, `a[2]`, etc.). Worked but every reader had to remember the column order. Replaced with `AnimeForDetection` NamedTuple — DAO returns `list[AnimeForDetection]`, callers read `entry.studios` / `entry.description_embedding`. Same fix removed a clever `(existing, new)[bool].append(entry)` partition trick.
- **Setting `candidate.status = MergeCandidateStatus.merged` after `db.delete(anime_b)`** (initial merge service). The cascade FK deletes the candidate row when anime B is removed, so the post-delete update fired against zero rows and threw `StaleDataError`. Now the merge path doesn't try to update status — cascade has already done the right thing.
- **Comparing `finished_at`/`lastOpenedAt` for retry button visibility.** Considered using job-row metadata to decide; brittle because retry semantics belong to the *error*, not the row's lifecycle. The marker-class approach (`PermanentPhsarError`) puts the decision next to the throwing code.
- **`retryable` as a Boolean column on the jobs table.** Considered an Alembic migration. Rejected: the bell only needs the flag when displaying a failed row, `result_summary` JSONB already serializes to the frontend, no migration cost.
- **Single end-of-sweep commit in `update_sweep_dispatcher`.** Initial 7b sketch. Meant a worker crash at anime #198 lost every refresh. Switched to per-anime commit + per-anime try/except after user feedback. The original "robust setup" framing is now load-bearing — don't undo.
- **`@app.middleware("http")` for the maintenance gate.** Used in 7a and the initial 7b. Worked for same-origin, broke for cross-origin during a sweep (no CORS headers stamped on the 503 short-circuit). Switched to a pure ASGI `MaintenanceGateMiddleware` class. The class form also pairs with the registration-order swap that puts CORS outermost.

## Implementation Notes

- **`jobs.payload` is JSONB, not three tables.** `user_scrape` uses `{"query": ...}`, `update_sweep` will use `{"anime_ids": [...]}`, `seasonal_sweep` will use `{"season": "..."}`. Worth the JSONB cost (~no overhead on reads, no schema growth) over three near-identical tables.
- **Dedupe SQL: `func.lower(func.trim(Job.payload["query"].astext)) == normalized`.** Function-on-column, no functional index. Fine at our queue size; flag if dedupe scans become slow.
- **`search_title` + `save_search_results` accept `progress: ProgressReporter | None = None`.** TYPE_CHECKING import in `jikan_scraper.py` to avoid a runtime import cycle (jikan_scraper is the lowest layer). The reporter is duck-typed at runtime — `progress.update(items_done=X, items_total=Y)`.
- **Throttling uses `time.monotonic()`, not `datetime`.** Monotonic is the right primitive for elapsed-time checks (immune to wall-clock changes).
- **`ProgressReporter._last_at = 0.0` initially** so the first call always fires regardless of throttle. Subsequent calls within `min_interval_s` are silently dropped.
- **Bell stops polling on 401, not just any error.** Other errors (network blips, 5xx) get `console.error` and the next poll retries. 401 means the token is dead — keep polling would just produce noise.
- **Dispatcher's "Done" stage writes happen `force=True`** because they fall within the throttle window of the previous "Saving — N/N" update. Without force, the user would see the row stuck at "Saving — N/N" until the worker flips status.
- **`MergeCandidate` table stores ordered pairs (`anime_a_id < anime_b_id`)** with a unique constraint on the pair plus a CHECK on the ordering. Caller is responsible for sorting before insert; `upsert_pending` validates and uses `ON CONFLICT DO NOTHING` on the unique constraint. The ordering is also a UI convention — "A is the kept one" — which currently maps to lower-id-wins (older anime). Per-anime FK cascades on both sides so deleting either anime cleans up dangling candidate rows automatically. Partial index on `(created_at) WHERE status='pending'` keeps the admin-list scan tiny no matter how much resolved history accumulates.
- **`detected_by` is a String, not an Enum.** Adding a future detector ('description_overlap', 'shared_mal_id', ...) shouldn't need a migration. Three constants live module-level in `merge_detection_service.py`: `DETECTOR_TITLE_STUDIO`, `DETECTOR_TITLE_DESC`, `DETECTOR_RELATION_LINK`.
- **`refresh_spoiler_cache_for_all_users` extracted from save_service into spoiler_service.** Two callers now (save flow + merge flow) and commit 7 will need a third. Per-user try/commit so one poisoned user doesn't abort the rest. Caller-commits pattern matches the rest of the codebase.
- **`AnimeForDetection` NamedTuple lives in `merge_detection_service.py`, imported by the DAO via `TYPE_CHECKING`** + a deferred runtime import inside `get_anime_for_detection`. Avoids the circular: detection imports DAO at module-load, DAO needs the type at runtime to instantiate rows.
- **`result_summary["retryable"]` contract documented on `JobResponse`** Pydantic schema. Frontend reads it via the typed Job interface; defaults to `true` when missing so older rows / non-job paths still get a retry button.
- **JobBell retry button disable scope is global, not per-row.** While one retry is in flight, every retry button in the dropdown is disabled (the active row spins). Prevents fan-out from a frantic user who sees a bunch of failures and wants to nuke them all at once.
- **Update-sweep test pattern: real sessions + `tracked_anime` cleanup.** Diff-loop tests use the rolled-back `db_session` fixture. Full-dispatcher tests can't — `update_sweep_dispatcher` calls `session.commit()` per anime, which terminates the connection-bound outer transaction the fixture relies on. Those tests open `async_session_maker()` directly and clean up via a `tracked_anime: list[int]` fixture that DELETEs at teardown. Mirrors `test_job_worker.py`'s `tracked_jobs` pattern.
- **Patch `AnimeDAO.select_due_for_sweep` in dispatcher tests.** Without it, the dispatcher iterates over ALL due anime in the test DB (including leftovers from previous tests), calls the fake scraper with mal_ids it has no payload for, and KeyErrors. The patch returns just the test's own seeded anime ids, scoped per test.
- **`_FakeScraper` supports `error_for_mal_id`** so per-anime failure-isolation tests can inject a single bad payload mid-loop and assert that the surrounding anime still commit.
- **`_NoopProgressReporter` replaces `ProgressReporter` in dispatcher tests.** The real reporter writes via `async_session_maker()` which sidesteps the rolled-back fixture — fine in JobWorker tests with explicit cleanup, but here we want every write to live inside the test's session for assertion convenience.
- **Tier query LEFT JOIN + COALESCE is defensive belt-and-braces, not load-bearing.** In production every row has a sidecar (migration backfilled + `save_search_results` creates them on insert). The LEFT JOIN keeps the query honest if a future code path inserts without one.
- **`MaintenanceGateMiddleware._ALLOWED_PATHS` is a frozenset class attr.** Static membership check, no per-request set construction. Adding to it requires a code change — that's intentional. Each allowlisted path needs the rationale documented (in the docstring's allowlist section); it's a small invariant surface.
- **`onBump(store, fn)` reused for the maintenance refresh signal.** The same helper backs both `jobsRefresh`/`librarySaved` (commits 5–6) and `maintenanceRefresh` (commit 7b). One subscribe-and-skip-initial primitive, three call sites — extracts pulled their own weight.
- **`Notice.svelte` exists so future yellow banners share one source of truth.** Solid `bg-yellow-50` surface, dark `text-yellow-900` text, `AlertTriangle` icon. No variant prop yet (no second visual variant in v0.14.0); add when a third banner shape lands.
- **`bumpMaintenanceRefresh` covers the `null → null` transition gap.** `MaintenanceBanner` subscribes to the `token` store but Svelte stores fire only on actual *changes*. A user already on /login (token=null) submitting login during a sweep gets `token.set(null)` after the 503, which doesn't fire the subscriber. The bump signal in `api.ts`'s 503-handler is the explicit kick.
