# v0.14.6 — Backup retention refinement

Three independent retention pools (archival manual/cron, pre-restore relevance buffer,
uploads), admin naming-as-pin, restore-link surfacing, and nightly integrity re-verification.

## Key decisions

- **Naming IS the pinning mechanism — no separate pin toggle.** Forcing every pinned dump
  to carry a human-readable `name` means an admin can never accumulate anonymous pins and
  forget why a dump is protected. The name is the justification. Rejected: a boolean `pinned`
  flag + optional label (lets reasons go missing).
- **Pins kept ON TOP of a full rolling window, not inside it.** `apply_retention` counts the
  recent/Sunday/cap slots over the *un-named* subset only (`_auto`). Counting named dumps into
  the slots would let ~14 pinned dumps starve every fresh nightly backup. Subtle but load-bearing.
- **Pre-restores split into their own relevance pool.** A pre-restore's `created_at` is the
  restore moment, which can land on a Sunday and pollute the long-term weekly archive with
  throwaway rollback state. They get a 3-recent buffer + the snapshot tied to the current
  pointer; never compete for archival Sunday/recent slots.
- **Re-verify integrity before retention, not at create time only.** The create-time flag is a
  point-in-time snapshot; on-disk corruption (truncation, bad sector) would keep listing `ok`
  and the "most recent known-good" pin would trust it. `reverify_backups` re-runs `pg_restore
  --list` (TOC-only, sub-second/dump) across all dumps before retention. Closes the
  v0.13.0-flagged stale-integrity gap.

## Gotchas

- **`_latest_per_sunday` fixed a silent bug**: the old `[... Sunday ...][:8]` slice kept the 8
  most-recent *Sunday-created* dumps — several dumps from one busy Sunday could consume every
  weekly slot, collapsing 8 weeks of history to 1 day. Now: latest dump per *distinct* Sunday.
- **Restore-link dedup edge cases** (both intentional, documented in services/CLAUDE.md):
  1. When the live DB already matches an existing manual/cron dump, the pre-restore dedupes to
     *that* dump and `restored_to` lands on its sidecar — link is truthful but can dangle if the
     dump ages out of the archival window.
  2. **Self-reference**: when the matched dump IS the restore target itself, `restored_to` lands
     on the current row's own sidecar. The `previous_state` derivation guards with
     `m.filename != current_filename` so a row never points at itself.

## Future work / debt

- "Most recent known-good" pin is archival-scoped only; an upload-only or pre-restore-only
  catalog has no cross-source known-good guarantee (recency windows cover it in practice).
- All retention state (pointer, integrity flags, locks) assumes a single worker process. The
  documented scale-out path (file sentinel / DB row) must also cover `reverify_backups`' lock-free
  read-then-write loop, which is only safe today because no concurrent backup job can run.
