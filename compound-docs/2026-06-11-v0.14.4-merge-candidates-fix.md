## Summary

Made `media_relation_edges` sidecar the single source of truth for the `relation_link` merge-detection signal. All three trigger sites (save, `update_sweep`, backfill) now converge through one DAO query (`MergeCandidateDAO.get_cross_anime_relation_pairs`), so the signal fires identically regardless of which job created the anime rows or when. Fixes the Evangelion-shape miss: NGE TV and Eva Rebuild Movies as separate scrape jobs never surfaced as a merge candidate because the prior path relied on ephemeral BFS state.

## Failed approaches

- **Per-job BFS state as truth** (the pre-v0.14.4 design): `cross_link_mal_ids` was collected during a live scrape's BFS, resolved against the catalog inside `save_search_results`, and fed to the detector. Worked for new-against-existing pairs in the SAME scrape job but missed every pair where both sides predated the detector OR were scraped in separate jobs. Container restarts didn't help — backfill explicitly skipped the relation_link path because it had no live BFS to mine.

## Key decisions

- **Allowlist, not blocklist** for which MAL relations count as merge evidence. Restricted to `sequel`, `prequel`, `alternative_version` (the `ALT_CHAIN_EDGES` set the classifier already uses). Dump-tested against the prod catalog: filtering "anything non-crossover" produces dozens of cross-anime edges spanning ambiguous types (`spin-off`, `parent_story`, `side_story`, `summary`, `full_story`, `other`) — MAL uses those for "related but distinct," not "duplicates." Rejected: a blocklist starting from "all relations" because every false positive trains admin to dismiss reflexively.
- **Reuse `ALT_CHAIN_EDGES` (rename from `_ALT_CHAIN_EDGES`) across modules** instead of defining a separate `MERGE_LINK_RELATIONS` set. Both questions ("does MAL say these belong together inside an umbrella?" and "does MAL say these separately-scraped umbrellas belong together?") have the same answer — sharing the constant means a future allowlist change lands once. Rejected: per-purpose copies, would have drifted on the next addition.
- **`update_sweep` scope = `sidecar_touched_anime_ids`, not `probe_attached_anime_ids`**. Step 1 of the sweep rewrites sidecars from the `/full` payload even when no media is attached — a steady-state refresh on an existing anime can land a freshly-added MAL relation that points to another catalog row. Broader scope (≤ `JOBS_SWEEP_MAX_PER_RUN=200`) catches that; probe-attached alone (typically <20) would miss it. Title_studio/title_desc keep the narrower probe-attached scope because those signals only matter when new media land.
- **DAO owns the SQL, service owns the policy.** `find_cross_anime_relation_pairs` is a 3-line service wrapper that pins the `ALT_CHAIN_EDGES` allowlist; the DAO method is allowlist-agnostic. Matches the project convention that queries live in DAOs.

## Gotchas & learnings

- **Shared dev DB defeats absolute-count test assertions.** Pre-existing dev catalog rows trip the relation_link sidecar sweep during `backfill_merge_candidates`, so `assert inserted == 1` is non-deterministic. Tests now assert "our specific test pair exists with the expected `detected_by`" via an `_our_pair_exists(a_id, b_id)` helper. Idempotency tests survived (the SECOND call must return 0 regardless of what the first did).
- **`OR (a IN scope) OR (b IN scope)` is a planner-fragility hot zone** at large scope. Fine at ≤200; if `JOBS_SWEEP_MAX_PER_RUN` is ever raised significantly the form may flip to a seq-scan — rewrite as `UNION ALL` of two scope-anchored sub-selects + outer DISTINCT.
- **Race between concurrent saves is absorbed by `ON CONFLICT DO NOTHING`** on the `uq_merge_candidates_pair` unique constraint. Two HTTP saves can compute identical INSERT lists, both attempt, one wins, the other no-ops. No code change needed but worth knowing the invariant holds.
