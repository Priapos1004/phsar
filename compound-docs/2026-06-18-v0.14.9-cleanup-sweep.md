# v0.14.9 — Cleanup sweep (tooltips, curation history, sweep-tier polish)

**Date:** 2026-06-18 | **Branch:** v0.14.9-clean-up

## Summary

A grab-bag polish release shipped as ~20 small commits, each `/update-docs` → `/simplify` → lint/tests → commit. Themes, roughly in two waves:

- **Wave 1 (curation/sweep polish):** a single themed Tooltip component replacing ad-hoc native `title=`; admin curation gains a "dismissed decisions" history with username-gated delete-to-resurface; the sweep "Stabilizing" tier becomes a per-check pipeline; assorted fixes (related-media "eps ?", season banner, stabilize threshold 5→3).
- **Wave 2 (job-detail audit + detail-page polish):** `update_sweep` `result_summary` goes to **v6** — it now records *which* media the relations probe attached (not just a count), surfaced as an "Attached via probe" review card; the job-detail counters grid drops the redundant failure cells, softens labels, and gains tooltips; the Jobs Log gets a blue probe-attach row tint; failure/attach links carry a **"Back to job"** navigation origin; MAL score pills get a "this is the MyAnimeList community score" tooltip; and the themed tooltip surface's hue bug (red on green/blue themes) is fixed. A `seed_demo_sweep_job` dev script builds a realistic v6 job from real catalog rows for visual QA.

## Key Decisions

- **Jobs Log filter is store-only, NOT URL-mirrored.** First implemented as a URL param (`?tab=jobs&kind=…`) with a reconcile effect; reverted to a module-level `jobsFilter` store after weighing it. Rationale: it's an internal admin tool, so refresh-survival/shareable-links don't justify the reconcile complexity, and a URL copy *resurrects* a filter on browser-back after you've left /admin (fighting the "clean on return" intent). The store survives tab switches + the job-detail round-trip (in-SPA memory); `routes/admin/+layout.svelte` clears it `onDestroy` so re-entering /admin from elsewhere starts clean. Cleared-on-leave is the whole reason a module store beats local component state here.
- **The shared `Tooltip` renders its OWN `Tooltip.Provider` (self-contained).** bits-ui's Tooltip `Root` reads `delayDuration` from a Provider context; without one it throws `Cannot read properties of undefined (reading 'delayDuration')`. A single global Provider in `+layout` works in the app but breaks isolated component tests (which render a card with no layout ancestor). Putting the Provider inside the wrapper makes every `<Tooltip>` work anywhere — app, tests, storybook-less. Trade-off: loses cross-tooltip skip-delay; negligible.
- **Tooltip migration policy: themed component for intentional hints; keep native `title=` for text-overflow reveals.** Migrated ~30 `title=` (admin cards, status dots, sweep-tier rows, job-detail chips). Deliberately KEPT native `title=` on dense truncation cells (MediaChangeCard / AnimeUmbrellaCard / search-result titles) — turning every data cell into a bits-ui trigger makes each a keyboard tab stop (tab-stop spam). `aria-label` added ONLY to icon-only buttons; buttons with visible text ("Dismiss", "Merge") already have an accessible name (don't double-label). The attribute-distribution bar dropped its tooltip entirely — the legend below already shows label+count+color.
- **Tooltip theme tint via a CSS var.** Surface = `--tooltip-surface: color-mix(in oklab, var(--color-popover) 88%, var(--primary))` in `:root` (app.css). The inner `var(--primary)` resolves at use-site, so the portaled tooltip follows `.theme-*` overrides. 500ms delay (between native ~700ms and the snappy 300ms first tried). **Wave-2 correction:** this originally mixed `in oklch` and rendered *red* on the green/blue themes — see the hue gotcha below; the fix was the one-token switch to `oklab`.
- **Curation delete-to-resurface leans on each detector's existing skip-set — not a new flag.** Deleting a *dismissed* merge row drops the pair from `get_existing_pairs` (all-status skip-set) → re-detection re-flags it. Deleting a dismissed split row clears its cluster signature from `upsert_pending`'s sticky-dismissal history → same. Only `dismissed` rows are deletable: pending belongs to the live queue; merged merge-rows are already cascade-gone with anime B; deleting a `split`-status row wouldn't undo the executed split. Username-gated, mirroring backup restore (`CurationConfirmationMismatchError`).
- **Resurface re-runs the full backfill.** The "Resurface" action deletes the row then calls the card's existing Re-detect (full `backfill`) so the freed candidate re-flags as pending immediately + the pending list refreshes. A targeted per-pair re-detect would need the deleted row's anime IDs cached — not worth it for an admin-only, occasional action; the user explicitly wanted "re-run the candidate search and update the UI."
- **Stabilizing breakdown: anime grain bucketed by its least-settled member (MIN), dynamic in the threshold.** The 4-tier SweepTiersCard "Stabilizing" row expands into per-check sub-rows (`stabilizing_by_check`). Media grain = each media's own `stable_check_count`; anime grain = a correlated `MIN(stable_check_count)` over the anime's media (the member that keeps it in the tier). Sub-bucket count derives from the breakdown's key count (`Object.keys(...).length`), so retuning `SWEEP_STABILIZE_THRESHOLD` reshapes the card with zero frontend edits. No `%` column on sub-rows: two independently-rounded percentage sets can't be made to reconcile (±0.1 artifact), so the exact counts + bar carry it.
- **Stabilize threshold lowered 5 → 3** (single shared constant). MAL metadata mostly settles in the first few days; nights 4–5 re-fetched unchanged rows. Bonus: the relations-probe gate trips 2 sweeps earlier.

### Wave 2 — job-detail audit

- **Persist *which* media the probe attached, not just the count → `result_summary` v6.** The dispatcher already tracked `probe_attached_anime_ids` (for merge/cache scoping) but threw away the detail; the admin couldn't see *what* got attached or when. v6 adds `probe_attached_anime[]` (`{anime_uuid, title, name_eng, name_jap, media:[{media_uuid, title, name_eng, name_jap}]}`) + `counters.probe_attached_media_count`. The job's own timestamp is the "when". Net-new keys would normally not force a version bump, but we bumped (like the v4 `step1_failed` precedent) so the frontend can tell a genuinely-empty v6 list from a pre-v6 row that never tracked it — the "Attached via probe" card + Jobs Log blue tint gate on `version >= 6`.
- **Captured via a `saved_sink` out-param on `attach_search_result_to_anime`, not a changed return type.** The function returns an `int` count that the `handle_attach_to_existing` caller (+ tests) rely on. Adding an optional `saved_sink: list[dict] | None` (append `{media_uuid, title, name_eng, name_jap}` per saved media) mirrors the dispatcher's existing `diff_sink` pattern and keeps the count contract intact. `_probe_relations_for_anime` then returns its accumulated list (empty list stays falsy, so the caller's `if probe_added:` is unchanged) instead of a bool.
- **Drop the failure cells from the counters grid; the amber cards own that signal.** The grid had `step1_failed` ("Failed refresh") + `probe_failed` ("Probes failed") cells *and* dedicated amber "Failed refresh (N)" / "Failed probe (N)" detail cards below — duplicate. Removing the cells also let the version-gated amber-warn logic in `JobDetailCounters` go. (User raised the redundancy directly.) Labels also softened: "Anime reclassed" → "Anime changed" (it's often just a cover-image refresh, not a reclassification), "Anime w/ new attach" → "Anime w/ new media".
- **"Back to job" is a third `DetailOrigin`, consistent with `library`/`q`.** Failure + attach links navigate **same-tab** with `?from=job&job=<uuid>`; `BackLink` renders "Back to job" → `/admin/jobs/[uuid]`, and `buildDetailHref` propagates `job` across anime↔media↔sibling hops (incl. `RelatedMediaCarousel`). **Opposite choice for the Anime/Media *change* cards: new tab, no origin** — that list runs 300+ rows, so an admin verifying one row shouldn't lose their scroll; short failure/attach lists have no scroll worth preserving. Both link sites carry a comment explaining the split. (See tech debt for the discriminated-union refactor this defers.)
- **Failure/attach titles render in the admin's name language** via the existing `resolveTitle(romaji, eng, jap, lang)` (same as `MediaChangeCard`). Required adding `name_eng`/`name_jap` to `Step1Failure`/`ProbeFailure` + the probe-attach entries — pre-captured before the savepoint like the rest of the failure fields. `resolveTitle` falls back to romaji when eng/jap are null, so this was a net-new safe-default addition (no extra version bump).
- **MAL score pill gets a tooltip.** Users read the score + "N ratings" and assumed it was *Phsar* users' ratings. A hover tooltip ("Community score from MyAnimeList — not from Phsar users.") on both detail pages; cheap and unobtrusive vs. permanent label clutter.

## Gotchas & Learnings

- **A test that mocks `svelte`'s `setContext`/`getContext` to a no-op breaks bits-ui's Tooltip context.** `media-detail.test.ts` stubbed `getContext: () => mockGetUserRole` + `setContext: vi.fn()` (for the page's `userRole` context); the no-op `setContext` meant bits-ui's Provider never set its context → Root threw `delayDuration` on undefined. Fix: make the mock key-aware — override only `'userRole'`, delegate every other key to the real `getContext`, and don't stub `setContext`. Pattern to remember when adding a Tooltip (or any bits-ui context component) under a context-mocking test.
- **`bits-ui` `Tooltip.Root` + `Trigger` (via the `child` snippet) render NO wrapper DOM.** So migrating a `<Button class="ml-auto">` to `<Tooltip>{#snippet trigger(props)}<Button {...props} class="ml-auto"/>{/snippet}</Tooltip>` keeps the Button the direct flex child — `ml-auto`/layout preserved. Trigger-mode (merge props onto your element) is the safe default; children-mode (a `cursor-help` span wrapping content) is for non-interactive hints AND for disabled buttons (which swallow hover events, so the span must wrap them).
- **`grep "title="` over-counts.** Component `title` *props* (e.g. `<ScrollableCard title="…">`) look identical to the HTML `title=` attribute but must NOT be migrated. Check the element per-site.
- **`delete_decision` originally double-fetched.** It called `get_by_uuid` (for the status check) then a `delete_by_uuid` that fetched again. Caught in `/simplify`: reuse the already-fetched row with `BaseDAO.delete(db, candidate)` and drop the per-DAO `delete_by_uuid` entirely.
- **`_tier_bucket`'s `and_(still_stabilizing, stable_expr == n)` guard was redundant.** For `n < threshold`, `stable_expr == n` already IS the stabilizing condition, and `airing_now` precedes it in the cascade. Dropping the guard made `_SweepAtoms.still_stabilizing` dead → removed it + its correlated EXISTS from `_sweep_atoms` (one fewer subquery on the anime Overview count). The *media* grain's `still_stabilizing` atom stays — `select_due_media_for_sweep` uses it.
- **Symmetry over DRY for parallel merge/split services.** `delete_decision` / `list_dismissed` are near-identical across the two services; left parallel (not extracted) because the whole merge/split service pair is already an established parallel-but-separate convention here — extracting one function would make it inconsistent. The within-DAO list query (`_list_with_anime`) and the cluster-preview builder (`_build_list_item`, split only) WERE extracted, since those duplications are within a single file.
- **`dismissed_at` reuses the existing list-item DTO** (optional field, `None` for pending) rather than a new schema — the frontend `DismissedDecisionsSection` is generic over `{uuid, dismissed_at}`.
- **`color-mix(in oklch, …)` against white drags the result toward RED.** The themed tooltip surface looked red on the green/blue themes. Root cause: white is `oklch(1 0 0)`, whose hue is **0 (red)** — not "none". At 88% weight, oklch *hue interpolation* pulls the mix toward 0°: green (150°) lands ~18° (red-orange), blue (250°) wraps to ~347° (pink-red); default purple (302°) happens to land near pink and read OK, which is why it went unnoticed. Fix: mix `in oklab` — oklab has no hue channel, so white's neutral a/b let the primary's hue dominate → a correct pale-primary tint. Lesson: when `color-mix`-ing a chromatic color with white/black/grey, prefer `oklab` (or write the neutral's hue as `none`) to avoid hue-0 bleed.
- **A `saved_sink` test double must accept the kwarg *and* populate the sink.** The probe path stopped reading `attach_search_result_to_anime`'s return and reads the sink instead; the `fake_attach` doubles in `test_update_sweep.py` had to add `saved_sink=None` to their signature and append synthetic `{media_uuid, title}` dicts, or `probe_attached_anime`/the spoiler-recompute scope silently went empty and assertions broke. The seasonal-sweep double only needed the kwarg (it uses the count path).
- **Demo seed script: build from REAL rows so links resolve, and remember amber outranks blue.** `scripts/seed_demo_sweep_job.py` pulls live anime/media so every Failed-/Attached-card link opens a real page. The Jobs Log row tint is amber (unknown-genre) *or* blue (probe-attach), amber first — so a fully-loaded demo never shows blue. Added `--no-genre-tags` to drop the amber path and exercise the blue tint in isolation.

## Future Work / Technical Debt

- **`curationRefresh` cross-card chatter:** a merge dismiss bumps the shared `curationRefresh`, so the split card's dismissed section (if expanded) also re-fetches. Negligible (small admin-only fetches), but a typed/scoped bump (`{kind}`) would remove the redundant cross-domain fetch.
- **Resurface = full catalog backfill** per click. Fine for occasional admin use; a targeted re-detect scoped to the resurfaced anime would be cheaper if this gets heavy.
- **`DetailOrigin` is now three parallel URL flags (`q` / `from` / `library|job` + `job`).** Adding `job` followed the established convention, but a `/simplify` altitude pass flagged that a fourth origin will mean a fourth optional param threaded through `buildDetailHref` + `BackLink` + every detail page + the carousel. The deeper form is a discriminated-union origin (`{type:'job', uuid} | {type:'q', token} | …`) encoded in one opaque `?origin=` param, propagated once. Deferred: it would change the existing `?q=`/`?from=library` URL contracts (shareable links) and touch far more than this diff.
- **Probe-attach links use `buildDetailHref` but the Anime/Media *change* cards (`MediaChangeCard`/`AnimeUmbrellaCard`) intentionally don't carry the job origin** (new-tab, verify-without-losing-scroll). If a future need arises to back-link from those too, that's a deliberate behavior change, not an oversight.
- **The anime-grain SweepTiersCard stabilizing breakdown adds a correlated `MIN(stable_check_count)` to the Overview count query** (on top of the two existing airing/recent-main EXISTS atoms). Sub-ms at ~800 media; if the catalog grows large, collapse all three correlated subqueries into one pre-aggregated CTE (`GROUP BY anime_id` over `Media LEFT JOIN MediaFreshness`) and bucket from that.
