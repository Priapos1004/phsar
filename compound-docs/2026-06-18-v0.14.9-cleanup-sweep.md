## Key Decisions

- **Jobs Log filter is store-only, NOT URL-mirrored.** First implemented as a URL param (`?tab=jobs&kind=â€¦`) with a reconcile effect; reverted to a module-level `jobsFilter` store after weighing it. Rationale: it's an internal admin tool, so refresh-survival/shareable-links don't justify the reconcile complexity, and a URL copy *resurrects* a filter on browser-back after you've left /admin (fighting the "clean on return" intent). The store survives tab switches + the job-detail round-trip (in-SPA memory); `routes/admin/+layout.svelte` clears it `onDestroy` so re-entering /admin from elsewhere starts clean. Cleared-on-leave is the whole reason a module store beats local component state here.
- **The shared `Tooltip` renders its OWN `Tooltip.Provider` (self-contained).** bits-ui's Tooltip `Root` reads `delayDuration` from a Provider context; without one it throws `Cannot read properties of undefined (reading 'delayDuration')`. A single global Provider in `+layout` works in the app but breaks isolated component tests (which render a card with no layout ancestor). Putting the Provider inside the wrapper makes every `<Tooltip>` work anywhere â€” app, tests, storybook-less. Trade-off: loses cross-tooltip skip-delay; negligible.
- **Tooltip migration policy: themed component for intentional hints; keep native `title=` for text-overflow reveals.** Migrated ~30 `title=` (admin cards, status dots, sweep-tier rows, job-detail chips). Deliberately KEPT native `title=` on dense truncation cells (MediaChangeCard / AnimeUmbrellaCard / search-result titles) â€” turning every data cell into a bits-ui trigger makes each a keyboard tab stop (tab-stop spam). `aria-label` added ONLY to icon-only buttons; buttons with visible text ("Dismiss", "Merge") already have an accessible name (don't double-label). The attribute-distribution bar dropped its tooltip entirely â€” the legend below already shows label+count+color.
- **Tooltip theme tint via a CSS var.** Surface = `--tooltip-surface: color-mix(in oklch, var(--color-popover) 88%, var(--primary))` in `:root` (app.css). The inner `var(--primary)` resolves at use-site, so the portaled tooltip follows `.theme-*` overrides. 500ms delay (between native ~700ms and the snappy 300ms first tried).
- **Curation delete-to-resurface leans on each detector's existing skip-set â€” not a new flag.** Deleting a *dismissed* merge row drops the pair from `get_existing_pairs` (all-status skip-set) â†’ re-detection re-flags it. Deleting a dismissed split row clears its cluster signature from `upsert_pending`'s sticky-dismissal history â†’ same. Only `dismissed` rows are deletable: pending belongs to the live queue; merged merge-rows are already cascade-gone with anime B; deleting a `split`-status row wouldn't undo the executed split. Username-gated, mirroring backup restore (`CurationConfirmationMismatchError`).
- **Resurface re-runs the full backfill.** The "Resurface" action deletes the row then calls the card's existing Re-detect (full `backfill`) so the freed candidate re-flags as pending immediately + the pending list refreshes. A targeted per-pair re-detect would need the deleted row's anime IDs cached â€” not worth it for an admin-only, occasional action; the user explicitly wanted "re-run the candidate search and update the UI."
- **Stabilizing breakdown: anime grain bucketed by its least-settled member (MIN), dynamic in the threshold.** The 4-tier SweepTiersCard "Stabilizing" row expands into per-check sub-rows (`stabilizing_by_check`). Media grain = each media's own `stable_check_count`; anime grain = a correlated `MIN(stable_check_count)` over the anime's media (the member that keeps it in the tier). Sub-bucket count derives from the breakdown's key count (`Object.keys(...).length`), so retuning `SWEEP_STABILIZE_THRESHOLD` reshapes the card with zero frontend edits. No `%` column on sub-rows: two independently-rounded percentage sets can't be made to reconcile (Â±0.1 artifact), so the exact counts + bar carry it.
- **Stabilize threshold lowered 5 â†’ 3** (single shared constant). MAL metadata mostly settles in the first few days; nights 4â€“5 re-fetched unchanged rows. Bonus: the relations-probe gate trips 2 sweeps earlier.

## Gotchas & Learnings

- **A test that mocks `svelte`'s `setContext`/`getContext` to a no-op breaks bits-ui's Tooltip context.** `media-detail.test.ts` stubbed `getContext: () => mockGetUserRole` + `setContext: vi.fn()` (for the page's `userRole` context); the no-op `setContext` meant bits-ui's Provider never set its context â†’ Root threw `delayDuration` on undefined. Fix: make the mock key-aware â€” override only `'userRole'`, delegate every other key to the real `getContext`, and don't stub `setContext`. Pattern to remember when adding a Tooltip (or any bits-ui context component) under a context-mocking test.
- **`bits-ui` `Tooltip.Root` + `Trigger` (via the `child` snippet) render NO wrapper DOM.** So migrating a `<Button class="ml-auto">` to `<Tooltip>{#snippet trigger(props)}<Button {...props} class="ml-auto"/>{/snippet}</Tooltip>` keeps the Button the direct flex child â€” `ml-auto`/layout preserved. Trigger-mode (merge props onto your element) is the safe default; children-mode (a `cursor-help` span wrapping content) is for non-interactive hints AND for disabled buttons (which swallow hover events, so the span must wrap them).
- **`grep "title="` over-counts.** Component `title` *props* (e.g. `<ScrollableCard title="â€¦">`) look identical to the HTML `title=` attribute but must NOT be migrated. Check the element per-site.
- **`delete_decision` originally double-fetched.** It called `get_by_uuid` (for the status check) then a `delete_by_uuid` that fetched again. Caught in `/simplify`: reuse the already-fetched row with `BaseDAO.delete(db, candidate)` and drop the per-DAO `delete_by_uuid` entirely.
- **`_tier_bucket`'s `and_(still_stabilizing, stable_expr == n)` guard was redundant.** For `n < threshold`, `stable_expr == n` already IS the stabilizing condition, and `airing_now` precedes it in the cascade. Dropping the guard made `_SweepAtoms.still_stabilizing` dead â†’ removed it + its correlated EXISTS from `_sweep_atoms` (one fewer subquery on the anime Overview count). The *media* grain's `still_stabilizing` atom stays â€” `select_due_media_for_sweep` uses it.
- **Symmetry over DRY for parallel merge/split services.** `delete_decision` / `list_dismissed` are near-identical across the two services; left parallel (not extracted) because the whole merge/split service pair is already an established parallel-but-separate convention here â€” extracting one function would make it inconsistent. The within-DAO list query (`_list_with_anime`) and the cluster-preview builder (`_build_list_item`, split only) WERE extracted, since those duplications are within a single file.
- **`dismissed_at` reuses the existing list-item DTO** (optional field, `None` for pending) rather than a new schema â€” the frontend `DismissedDecisionsSection` is generic over `{uuid, dismissed_at}`.
