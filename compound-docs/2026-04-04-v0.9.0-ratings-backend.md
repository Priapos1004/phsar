## Summary

Full ratings backend: 11 attribute enums, upsert CRUD, bulk operations, and unified search that extends the existing media search infrastructure with rating-specific filters. DAO layer hardened with `lazy="raise"` on all relationships and batch query patterns.

## Failed Approaches

- **Single `Ending` enum** mixed structural type (open/cliffhanger) with quality (satisfying/great). Split into `EndingType` + `EndingQuality` as orthogonal dimensions.
- **Separate create/update endpoints** required frontend to track rating existence. Replaced with single upsert (PUT) that handles both.
- **`note_target` parameter** on bulk upsert exposed a design decision as config. Hardcoded to "last" since it's always the right answer.
- **Scattered eager loading comments** on mapping functions. Replaced with `lazy="raise"` on all ORM relationships, which enforces the contract at the ORM level.

## Key Decisions

- **Unified search over separate endpoint**: `GET /search/ratings` reuses all media filters via schema inheritance (`RatingSearchFilters extends MediaSearchFilters`) + shared `apply_media_filters`. Future watchlist search follows the same pattern.
- **`lazy="raise"` globally**: Prevents silent lazy loading and async MissingGreenlet errors. Every relationship access must go through explicit `selectinload` in the DAO. Surfaced one latent bug (new ratings accessing `rating_search`).
- **`get_all_by_field` on BaseDAO**: Batch-fetch complement to existing `delete_all_by_field`. Eliminated raw `select().where(.in_())` calls in services, restoring DAO layering.
- **Typed enum filters**: Rating filter fields use actual Python enums (`list[Pace]`) for API-level validation, unlike existing media filters (`list[str]`). Applied consistently to `MediaType`/`RelationType` too.

## Gotchas

- Alembic `op.add_column` with `sa.Enum(create_type=True)` does NOT create PostgreSQL enum types. Must call `sa.Enum(...).create(bind, checkfirst=True)` explicitly before `add_column`.
- New ORM objects after `flush()` raise on relationship access with `lazy="raise"` even though the relationship is None. Fixed by inlining the new-rating embedding creation instead of calling through `_upsert_note_embedding`.
- FastAPI `Depends()` with nested dependencies works for composing filter query params but each field must be listed explicitly (no Pydantic model auto-destructuring for query params with list types).
