# Frontend — design notes

SvelteKit + Svelte 5 runes + Tailwind 4 + shadcn-svelte. Loaded on top of root [CLAUDE.md](../../CLAUDE.md) when working in the frontend tree.

## routes/

Pages: home (`/`), login (`/login`), register (`/register`), search (`/search`), media detail (`/media`), anime detail (`/anime`), ratings (`/ratings`), settings (`/settings`), admin (`/admin`), add to library (`/library/add`).

- `GET /health` endpoint returning `{status, version}` for Coolify liveness — deliberately does not probe backend (liveness should only check what restarting this container can fix)
- `/library/add` — query-input + recent-additions panel; submitting POSTs to `/jobs/scrape` and the navbar bell takes over from there; restricted users see the page but the form is disabled
- **Tab title convention**: every route declares its own `<svelte:head><title>…</title></svelte:head>` in the format `<Page> — Phsar` (em dash, "Phsar" title case). Page name first so users with several PHSAR tabs open can distinguish them in the tab strip; the favicon already conveys "this is PHSAR". Detail pages (anime/media) bind the title reactively to the resolved name and fall back to a generic label while loading. `app.html` only sets the default `PHSAR` so the very first paint isn't blank — every route is expected to override
- **Admin tab navigation**: `/admin` uses a `?tab=` query param to switch between sections (`overview` default, `jobs`, `tokens`, `curation`, `completion`, `backups`). Tabs eager-render and stay mounted (visibility toggles via `class:hidden`), so first paint pays one parallel fetch per tab and subsequent switches are instant. The tab list, default, and validation live at the top of `routes/admin/+page.svelte`; the tab UI itself is `lib/components/admin/AdminTabNav.svelte`. Adding a tab requires updating three places: `AdminTabKey` in `lib/components/admin/types.ts`, the `TABS` array + content cascade in the page, and the rendering branch
- **Ratings page tab navigation** (v0.14.12): `/ratings` mirrors the admin `?tab=` switcher (`ratings` default, `stats`); tab list + default + validation at the top of `routes/ratings/+page.svelte`, UI in `RatingsTabNav`. The page owns **one** `GET /ratings/scores` fetch and both tabs derive entirely from it client-side (no per-user stats endpoint, no extra backend math). Two deviations from admin's eager-everything: the **stats tab is gated `{#if active === 'stats'}` so it remounts on each entry** (the ~6 ECharts replay their build-up animation rather than sitting idle), and `routes/ratings/+layout.svelte` calls `clearRatingsFilter()` `onDestroy` so leaving the section resets filters (same pattern as the admin layout + `adminJobsFilter`). Component composition — list-tab views, the five stats sections — is in the `lib/components/ratings/` bullet below; the pure stats math is in `lib/utils/ratingStats.ts`
- **Per-job detail page**: `/admin/jobs/[uuid]/+page.svelte` renders an update_sweep v2+ job's full audit trail — header card, counters grid, filterable media-change list (chips: all / dynamic-no-rating / rating / static / has drift; search across all title-language fields) sorted most-substantial-first via `sortMediaChanges` (group static > genre/studio > dynamic > rating-only, then by amount changed; rating-only by search-relevance impact), umbrella "Anime changes" cards (rendered **above** Media changes since v0.14.9 — far fewer rows, so they no longer get buried under 300+ media diffs). v4 (v0.14.7) renders a **Failed-refresh list card** (per-anime `step1_failures`); v5 (v0.14.8) adds a symmetric **Failed-probe list card** (`probe_failures` — title link + `error_category` + message) and a **progress-divergence warning** when `items_done < items_total` (anime selected but skipped); v6 (v0.14.9) adds an **"Attached via probe" card** (`probe_attached_anime` — anime title link + the new media titles listed below, each linked to its media detail). The three list cards (failed-refresh / failed-probe / attached) cap at ~4 rows and scroll (`max-h-72 overflow-y-auto`) so a long list doesn't bury the changes below. **Link-target design decision (v0.14.9):** the failure + attached links navigate in the **same tab** with a `from: 'job'` origin (→ "Back to job" button, propagated across anime↔media hops) — short lists, no scroll to lose; the **Anime/Media change** cards (`AnimeUmbrellaCard` / `MediaChangeCard`) instead open a **new tab** with no origin, because that list runs 300+ rows and an admin verifying one row shouldn't lose their scroll position. Failed jobs only show the error banner in the header; v1 sweeps render a "predates v0.14.5" notice. Click-through from `AdminJobsLogTab` is restricted to `update_sweep && version >= 2`; its `payloadSummary()` reads media-grained keys (`media_refreshed`, `media_with_dynamic_changes`) for v5 and the anime-grained keys for v2–v4
- **Settings page** (`/settings`): restricted (guest) users see **all controls disabled, not hidden** — Rating Step, Data Export buttons, and the Spoiler Protection select (pinned to `Off`, since guests can't rate and are excluded from the spoiler cache). Theme / name-language / search-view stay editable. The server also drops a guest's `spoiler_level` update defensively (`user_settings_service`)
- **Detail-page back button**: anime/media routes render `<BackLink searchToken fromParam jobUuid />` near the top. It decides between "Back to search" (when `?q=<token>` is present, from search results), "Back to library" (when `?from=library`, e.g. recent-additions panel), "Back to job" (when `?from=job&job=<uuid>`, from the admin job-detail page's Failed-refresh / Failed-probe / Attached-via-probe links → `/admin/jobs/[uuid]`), "Back to completion" (when `?from=completion`, from the admin Completion tab's anime links → `/admin?tab=completion`), "Back to curation" (when `?from=curation`, from the Merge/Split candidate cards' anime links → `/admin?tab=curation`), and "Back to statistics" (when `?from=ratings-stats`, from a You-vs-MAL scatter point → `/ratings?tab=stats`). New origins extend the `DetailOrigin` type union in `lib/utils/navigation.ts` AND the `target` switch in `BackLink.svelte`. `buildDetailHref()` propagates `q`/`from`/`job` on internal anime↔media jumps (incl. the RelatedMediaCarousel) so a deep dive (job → anime → media → sibling) stays linkable back to the origin

## lib/components/

App components using Svelte 5 `$props()`, `$state()`, `$derived()`, `$effect()`.

- **SearchBar, MediaInfo, NavBar, TagSelect, DoubleRangeSlider, RatingCard, BulkRateDialog, AttributeSelect, RatingNeighbors, DangerZone, RelatedMediaCarousel, SpoilerGuard, EChart** — core UI components
- **SearchBar layout + view toggle** (v0.14.13) — the filter (sliders) button overlays the input on the **left**, the search-trigger (magnifying glass, `type="submit"`) on the **right** so mobile users can tap to search instead of needing Enter. Both overlay buttons carry `z-10`: the input's `backdrop-blur` promotes it to a stacking context painted in tree order, so without it the input paints over the filter button that precedes it in source (the original right-only filter button worked only by sitting after the input). Switching anime↔media does a **partial clear** — `switchView` (`routes/search/+page.svelte`) carries only the directly-applicable filters via an allowlist (`CARRY_LIST_KEYS` = categorical lists, `CARRY_NUMBER_KEYS` = the 0–10 score range) into a fresh token and drops the view-relative ones (relation type + episodes/scored-by/duration/watch-time ranges). The `viewType` `$effect` closes the panel + refetches option bounds but does **not** `clearFilters()` — `syncFiltersFromParams` reconciles filter state from the new token. Relation Type is filtered out of the anime-view `activeFilterConfig` (per-media property, meaningless when aggregated)
- **SegmentedControl** (v0.14.12) — the on-card muted-track/solid-primary-thumb pill toggle, extracted from 4 inline copies (the ratings filter bar's any/all, the tag chart's genre/studio, the activity chart's MA-window + time-range). Generic over `T extends string | number`; presentational — the caller owns the source of truth via `value` + `onSelect`, so the highlighted `value` may be a **derived** value distinct from what `onSelect` writes (e.g. the moving-average window clamps to the dataset). Lives on the WHITE card surface, so inactive text is `card-foreground`, never the page `foreground`. `size` `sm` (next to a caption) / `md` (next to a heading). NOTE: the on-DARK grid/table view-pills in `RatingsFilterBar` are deliberately NOT this component — different surface (border-fill on dark, not track-thumb on card)
- **Tooltip** — the single app-wide hover/focus tooltip (wraps the shadcn-svelte/bits-ui `ui/tooltip` primitive). Themed to the active palette: a faint primary-tinted `popover` surface with a `--primary` border + soft primary glow (see `ui/tooltip/tooltip-content.svelte`), ~500ms delay. **Self-contained** — it renders its own `Tooltip.Provider`, so it works standalone (incl. isolated component tests) with no ancestor Provider. Two modes: pass `children` for non-interactive hints (renders a `cursor-help` span trigger — status dots, labels, chart bars) or a `trigger` snippet for interactive elements (spread `props` onto your own `<Button>`/`<button>` so keyboard focus reveals it and nothing gets double-wrapped; icon-only buttons must add an `aria-label`). Note: a test that mocks `svelte`'s `setContext`/`getContext` must delegate unknown keys to the real impl, or the Provider context breaks (see `tests/media-detail.test.ts`)
  - **The three tooltip types** (pick by what the cursor is over — never style raw text as a hint):
    1. **App `Tooltip` component** (`lib/components/Tooltip.svelte`) — the themed DOM hint, for every intentional UI hint on a DOM element. Examples: `GenreBadges` (description on a badge), `ScorePercentile` (the "Top N%" pill), `RatingsFilterBar` grid-order toggle (icon button → `trigger` snippet + `aria-label`), `JobDetailCounters` cells
    2. **ECharts chart tooltip** — the in-canvas hover for any `EChart`-based chart, configured on the chart's `option.tooltip` (the app `Tooltip` can't wrap a canvas region, so charts own their hover). Spread the shared **`chartTooltipStyle`** from `lib/utils/chartTheme.ts` into every chart's `tooltip` so the surface/border/glow match the app `Tooltip` (it themes via CSS custom props — `--tooltip-surface`, `--primary` — which resolve per active theme since the ECharts tooltip is a DOM node, and bundles `confine: true` so the popup stays in the card), then add the chart's own `trigger` / `formatter`. Examples: `RatingsOverviewTimeline.svelte` (anime page, per-media score bars) and `ratings/RatingsScoreHistogram.svelte` / `RatingsAlignmentChart.svelte` (ratings stats). Spreading `chartTooltipStyle` is the single source — don't re-set its keys per chart. **A chart `formatter` returns HTML** (ECharts renders it via `innerHTML`), and Svelte's auto-escaping does NOT reach inside these hand-built strings — so any catalog-sourced text spliced into one (anime/media titles, genre/studio names) MUST go through `escapeHtml` from `formatString.ts`. Enum-mapped labels (`formatRelationType`/`formatMediaType`), numbers, and ECharts' own `marker` spans are safe; only free-form catalog strings need it
    3. **Native `title=`** — raw text-overflow reveal on dense data cells ONLY (`MediaChangeCard` / `AnimeUmbrellaCard` / search-result titles / `RatedAnimeCard` title) so they don't each become a keyboard tab stop or spin up a Provider per row
- **GenreBadges / StudioLinks / ScorePercentile** (v0.14.11) — shared detail-page chips extracted from the anime+media `+page.svelte` (both rendered identical inline markup; extraction de-duplicates and makes them testable):
  - **GenreBadges** (`genres: string[]`) — renders each genre as a `cls.badgeGenre` Badge; wraps it in the app `Tooltip` with the genre's description when one exists (else a plain badge). Descriptions come from the `genres` store (lazy-fetched once); the tooltip is a progressive enhancement, so a missing description or failed fetch just drops the tooltip
  - **StudioLinks** (`studios: string[]`) — renders the "Studio" label + one clickable `<button>` badge per studio; clicking calls `navigateToSearch({ query:'', search_type:'title', view_type:'anime', studio_name:[studio] })` so the user lands on an anime-view search filtered to that studio ("other anime from this studio"). Tested in `tests/studio-links.test.ts`
  - **ScorePercentile** (`topPercent: number \| null`) — small always-visible "Top N%" pill placed **after** the score's rating-count text (raw MAL data first, then the derived standing), color-ramped by tier (≤15% emerald / ≤50% sky / else neutral). Always-visible so mobile doesn't depend on hover; the `Tooltip` only adds the full "among the top N% … weighted by vote count" explanation (phrased as the clamped chip rank, not an inverted "higher than (100-N)%" figure — the latter overstated the best title since `topPercent` is floored at 1). `null` renders nothing

The standing explanation lives in the component; callers pass only the count sentence (`detail`) so the copy can't drift
- **AttributeSelect** (v0.14.13) — the single attribute-dropdown used by **both** RatingCard and BulkRateDialog (was duplicated inline in each — the drift source). Props: `label`, full `options`, `value`, `onChange`, `disabled`. Adds a `✕` clear button (shown when set & not disabled) to deselect back to "Not set", and **filters the `not_applicable` sentinel out of the selectable list everywhere** — `not_applicable` is auto-set-only (never user-pickable), the single rule that keeps bulk + media from drifting (closes the old bulk-shows-N/A divergence). The trigger keeps no extra right padding: the built-in chevron is in-flow (extra `pr-*` shifts it inward), so the `✕` is an absolute overlay at `right-8`, just left of it. The auto-SET side stays in RatingCard: an `$effect` over `AUTO_NA_FIELDS` (`ending_type` + `ending_quality` — neither is knowable on an unfinished watch) sets both to `not_applicable` + `disabled` on on_hold/dropped and clears them on completed (one list so the effect + the `disabled` prop can't drift). v0.14.13 also rebucketed `has_3d_animation` to `none/rare/medium/heavy` (mirroring `fan_service`'s frequency scale) and gave `ending_quality` a `neutral` midpoint — both via an Alembic enum migration (`ALTER TYPE … ADD VALUE`/`RENAME VALUE`)
- **RatingNeighbors** (v0.14.11; extracted to a shared component in v0.14.13) — the collapsible "How you rated similar titles" panel, now mounted in **both** RatingCard and BulkRateDialog (bulk rating is anime-scoped, so it gets the same `animeUuid`/`genres`/`studios`/`ageRatingNumeric` exclude+tiebreak context, wired from the anime page). On first expand it fetches `GET /ratings/scores` once and caches it; a `$derived` over the live `score` prop runs `selectRatingNeighbors` (utils) to pick up to 4 ratings from **distinct other anime** straddling the score — up to 2 at the **same** score (capped), then the rest split evenly between strictly-lower and strictly-higher (odd slot → higher) so a dense library always shows a lower row for sanity-checking, not just equals + one step up — so moving the slider re-selects with no refetch. A failed fetch leaves `neighborItems` null (not `[]`) → "Couldn't load — Try again" instead of the misleading empty state. Rows render the score via `formatScore` (rounds to 2 dp to shed float noise like `7.890000000000001`) and expand to reveal that title's attributes (the shared `attributeBadges` from `utils/ratingAttributes.ts`). In the bulk **dialog**, the body needs `min-w-0` (Dialog.Content is a CSS grid, whose items won't shrink below a `nowrap` neighbor title) so the titles truncate instead of overflowing. v0.14.13 **color-codes** each neighbor attribute against the live `currentAttributes` prop via `compareAttribute`: green/red for the 5 quality attrs (higher/lower by ordinal position), blue for a categorical attr that differs, warm cream when it matches your pick, plain grey when you haven't set it — reactive to **both** the score and the user's own selections. Rows with attributes **default to expanded** (an explicit chevron toggle overrides) so the comparison is visible without per-row clicks

- **admin/CompletionStatusCard** — the **Completion** tab (v0.14.10). A manual curation surface (not a detection queue): a debounced anime search (reuses `fetchAnimeSearchResults` → `/search/anime`, with a clear-✕ button) to mark anime story-complete, plus the marked list (cover thumbnail + "marked {date} by {admin}" audit, client-side sort newest/oldest/title, unmark). Marking clears + closes the search so it reads as a committed selection. Same silent-refresh pattern as MergeCandidatesCard. Anime links navigate same-tab with `from=completion` (→ "Back to completion"). Unmark (X) is guarded by a **click-to-arm confirm** (v0.14.11): first click arms the row (X → red `Check`, tooltip "Click again to confirm removal", auto-disarms after 3s), the second click within the window fires the `DELETE` — mirrors the candidate cards' confirm-step so an accidental click can't silently un-mark. No `currentUsername` prop — the admin role gates the page
- **admin/DismissedDecisionsSection** — shared, generic (`<T extends {uuid, dismissed_at}>`) collapsible history of dismissed merge/split decisions, reused by both candidate cards. Lazy-fetches its `listUrl` on first expand; renders each row via a parent-supplied `row` snippet (merge shows A↔B; split shows the source + "would split off:" member titles per cluster). A username-gated confirm dialog (mirrors BackupsCard restore) POSTs `${basePath}/{uuid}/delete`, then calls `onResurfaced()` so the parent re-runs detection and the freed candidate pops back into the pending list. Subscribes to `curationRefresh` (`onBump`) so the counter/list re-fetch when the parent dismisses a new candidate — no stale `(N)`
- **MaintenanceBanner** — wraps a `Notice`, lives in a sticky container in `+layout.svelte` alongside the navbar
  - Polls `GET /maintenance/status` every 30s via raw `fetch` (bypasses `api.ts` so a 503 mid-window can't trigger maintenance redirect and defeat the pre-warning point — 30s instead of 60s because seasonal sweep's maintenance window can be only seconds long)
  - Subscribes to `token` store + `maintenanceRefresh` bump signal so relogin or any 503-with-maintenance response refreshes in milliseconds
  - Countdown wording: `"Scheduled maintenance starts in ~N minute(s) — pause your current episode."` within 30 min; `"Maintenance in progress. Please try again later."` when active; otherwise hidden
- **SessionTimeoutBanner** — the sliding-session driver + idle warning, mounted in the same sticky container as MaintenanceBanner (above it — a "you're about to be signed out" countdown outranks a future-window notice), guarded so it only runs while authenticated and off `/login`/`/register`. Self-contained like MaintenanceBanner: owns global activity listeners (`mousemove`/`keydown`/`scroll`/…) that just stamp `lastActivity`, a 1s tick, and a `visibilitychange` re-check (so a slept/backgrounded tab re-evaluates on resume instead of waiting for the next tick)
  - Each tick decodes the JWT `exp` and runs the pure `evaluateSession` (`lib/utils/sessionTimeout.ts`): **refresh** (recent activity + token in its back half → silent `POST /auth/refresh`, guarded by `refreshInFlight`, compare-and-swap on the `token` store so a response landing after logout/another-tab-change is dropped; a 401 from refresh is terminal), **warn** (idle + ≤3 min left → renders a `Notice` countdown `"You'll be signed out in m:ss … click your mouse to stay signed in."`), or **logout** (≤0 → calls the `onExpire` prop **once**, latched via `expired`; the layout shows its existing `TokenExpiryDialog`). `expired` resets when a fresh token arrives (login or cross-tab refresh) so the tab resurrects. Constants (`WARNING_LEAD_MS=3m`, `REFRESH_THRESHOLD_MS=5m`, `ACTIVITY_WINDOW_MS=60s`) live in the util with the invariant `REFRESH_THRESHOLD > WARNING_LEAD` so an active user never sees the banner. The layout's `token.subscribe` short-circuits its `/users/settings` refetch when the decoded `sub` is unchanged, so a silent refresh every ~5 min doesn't re-fetch settings
- **JobBell** — polls `/jobs/mine`
  - Caps dropdown at 5 entries (older spill to `/library/add`)
  - Hides retry button when `result_summary.retryable === false`
  - Disables retry across all rows while one is in flight
  - Progress bar uses `Math.min(100, Math.round(percentOf(items_done, items_total)))` — backends occasionally over-report a child batch as more items than the initial total estimate; the 100-cap keeps the bar from rendering 137%
  - **Clickable finished rows** (v0.14.13): a succeeded `user_scrape` row links to `/library/add` and a succeeded `backup` row to `/admin?tab=backups` (chevron affordance + dropdown closes on click). `rowHref(job)` gates on `status === 'succeeded'` and looks the destination up in `JOB_SUCCESS_HREF` (per-kind static metadata in `formatString.ts`, beside `JOB_KIND_LABELS`/`PARENTING_KINDS`); rows split into an `<a>` vs `<div>` wrapper (a shared `jobRow` snippet renders the body) rather than `<svelte:element>` so the anchor is a real interactive element with no a11y-role guesswork. Failed rows keep their retry button and stay non-linked (succeeded/failed are mutually exclusive)
  - **Dismissible-while-running badge** (v0.14.13): opening the dropdown now marks **every** visible job seen (active OR finished), not just finished ones — so the badge can clear while a job is still fetching. `badgeCount` counts `unseenActive + unseenFinished + unseenCuration` (active jobs filtered against `seenUuids` like finished ones); `pollDelay` still keys off the full `activeJobs` list so cadence stays fast after the badge is dismissed
  - **Completion toasts** (v0.14.13): on an active→finished **transition** (tracked via a non-reactive `prevStatusByUuid` snapshot rebuilt each fetch) the bell fires a green/red `pushToast` (`lib/stores/toast.ts`) for `user_scrape` + `backup` only — restore (admin is hard-navigated off-page) and sweeps (system jobs) return `null` from `toastForJob`. Empty snapshot on first fetch ⇒ already-finished jobs never toast; a `toastedUuids` Set guards a double-fire across overlapping poll/`jobsRefresh` fetches. This is independent of the existing `announcedSavedUuids` bump logic (which still drives `librarySaved`/`backupSaved` page refreshes)
- **Toast / ToastHost** — `Toast.svelte` is the `fixed top-20` fly-in banner; v0.14.13 gave it a `variant` prop (`info` → `bg-primary`, unchanged default; `success` → literal emerald — no `--success` theme token exists; `error` → themed `destructive`). `ToastHost.svelte` is mounted **once** in `+layout.svelte`, renders `{#if $activeToast}{#key $activeToast.id}<Toast …/>{/key}{/if}` — the `{#key}` on the store's monotonic `id` replays the fly-in when a new toast lands mid-display (old animates out, new in) instead of swapping text silently. **`pushToast()` is the single app-wide toast path** — the bell's completion toasts, `/library/add`'s enqueue toast, `/settings` ("Settings updated"), BackupsCard ("Backup queued…"), and RegistrationTokensCard ("Copied"/"Token deleted") all go through it (v0.14.13 removed the per-page local `Toast` copies + their divergent `setTimeout` durations). `Toast.svelte` is now consumed **only** by `ToastHost` — no component owns a local `<Toast>`
- **admin/AdminJobsLogTab** — paginated jobs table with kind/status filters, seasonal_sweep child expander
