# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Phsar is a full-stack anime search and rating web application. It combines a FastAPI backend with a SvelteKit frontend, using PostgreSQL with pgvector for semantic vector search over anime data sourced from the MyAnimeList (Jikan) API.

## Commands

All backend commands run from the `phsar/` subdirectory. The database Docker container must be running for tests and the app.

### Backend
```bash
cd phsar

# Run FastAPI (seeds genres + admin user on first start)
uvicorn app.main:app --reload

# Lint
ruff check .
ruff check . --fix    # auto-fix

# Tests (requires running PostgreSQL container; all DB changes are rolled back)
pytest
pytest tests/routers/test_auth.py           # single file
pytest tests/routers/test_auth.py::test_fn  # single test

# Database migrations
alembic revision --autogenerate -m "Describe change"
alembic upgrade head
```

### Frontend
```bash
cd phsar/frontend
bun install
bun run dev -- --open   # dev server at localhost:5173
bun run test            # vitest component tests
bun run check           # svelte-check type check
```

Backend and frontend must run simultaneously in separate terminals.

### Database (Docker)
```bash
# Start PostgreSQL with pgvector (use credentials matching your .env)
docker run --name anime-postgres \
  -e POSTGRES_USER=<DB_USER> -e POSTGRES_PASSWORD=<DB_PASSWORD> \
  -e POSTGRES_DB=<DB_NAME> -v pgdata:/var/lib/postgresql/data \
  -p 5432:5432 -d ankane/pgvector

# Reset database
rm phsar/alembic/versions/*.py
docker exec -it anime-postgres psql -U <DB_USER> -d <DB_NAME> \
  -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
```

### Docker (production parity)
```bash
# Build + run db + backend + frontend containers end-to-end (NOT the dev flow).
cp .env.example .env
docker compose up --build
```

## Architecture

### Backend (phsar/app/)

Layered architecture with strict dependency flow: **routers → services → DAOs → models**

- **routers/** — FastAPI endpoint definitions. Each router maps to an API prefix (`/admin`, `/auth`, `/search`, `/filters`, `/media`, `/save`, `/seed`, `/ratings`, `/users`, `/jobs`, `/library`, `/maintenance`). `/admin` handles registration token management (list, create, delete — admin-only), database backups (list, download, delete, restore, upload — admin-only; `POST /admin/backups` and `POST /admin/backups/auto` (cron-token authed) both enqueue a `backup` job and return 202 with the job uuid so the request thread doesn't block on `pg_dump`. Manual attributes to the admin user (visible only in their bell); cron is a system job (`requested_by_user_id=null`), invisible to every bell — the dump list is the audit log), merge-candidate review (list pending, merge, dismiss — admin-only; powered by the duplicate detector), and two cron-token-authed sweep schedulers — `POST /admin/jobs/schedule-sweep?delay_minutes=N` enqueues an `update_sweep` job (nightly catalog refresh + relations probe) and `POST /admin/jobs/schedule-seasonal?delay_minutes=N` enqueues a `seasonal_sweep` job (weekly `/seasons/now` scrape). Both set `core/maintenance.set_scheduled_at(...)` so the banner countdown shows, are bound to `Query(20, ge=0, le=1440)` on `delay_minutes`, and are allowlisted in the maintenance gate so a cron retry while a sweep is running can't 503 the cron itself. `/search` handles both `/search/media` (per-media) and `/search/anime` (aggregated by anime). `/media` handles both `/media/{uuid}` and `/media/anime/{uuid}`. `/users` handles settings CRUD, data export, and account deletion. `/jobs` handles user-triggered background work (`POST /jobs/scrape` enqueues a `user_scrape` job; `GET /jobs/mine` lists the current user's active + recently-finished jobs; `GET /jobs/{uuid}` single-job poll for owner or admin). `/library` exposes `GET /library/recent` for the `/library/add` page's recent-additions panel. `/maintenance` exposes a public `GET /maintenance/status` returning `{active, scheduled_at}` for the frontend's pre-warning banner; allowlisted in the maintenance HTTP middleware so it keeps returning truthful state mid-window.
- **services/** — Business logic as module-level async functions:
  - `jikan_scraper.py` — MAL API client with retry + client-side rate limiter. Tenacity tuned to `wait_exponential(multiplier=2, min=1, max=30)` × 5 attempts for better tail behavior on transient MAL outages. `retry_if_exception(_is_transient_mal_error)` gates retries to 5xx + timeouts + network errors — 4xx is deterministic and burns 31s of backoff before failing the same way. `reraise=True` surfaces the underlying `HTTPStatusError`/`TimeoutException` to the caller (and the bell's `result_summary["error"]`) instead of wrapping it in tenacity's `RetryError`. Class-level `_rate_lock` + `_MIN_REQUEST_INTERVAL_S=1.0s` gate spaces request *starts* at 1 req/s. Jikan documents 3 req/s burst AND 60 req/min sustained — the per-minute ceiling is what binds for continuous-request flows like `/seasons/now` pagination and bulk-refresh sweeps, so the spacing matches the sustained limit rather than the burst. 0.35s and 0.5s both observed 429s in practice. Jikan's docs also note MAL upstream can rate-limit independently. Retry policy is asymmetric: 429 IS retried but capped at 3 total attempts (2 retries — enough to bridge a misaligned per-minute window), while 5xx/timeout/network get the full 5 attempts. Other 4xx (most importantly 404) is deterministic-non-retryable. Cap on 429 is deliberate — retrying harder masks sustained throttling instead of fixing it; the right response then is to slow the source rate. BFS in `search_title` accepts an optional `ProgressReporter` so user-scrape jobs can stream progress to the bell. Also takes `seed_mal_id` + `seed_payload` so the sweep probe can BFS from a known mal_id instead of going through the fuzzy `q=<title>` top-3 lookup user_scrape uses (that lookup pulls unrelated franchises into the catalog if used for known-mal_id probes — see compound-doc). When `seed_payload` carries a `/full` response with relations bundled, the BFS reuses them instead of paying a second `/anime/{id}/relations` hit. Anime with `media_type=None` AND `airing_status="Not yet aired"` are skipped silently — they used to land in `media_unwanted` as `Unknown`, permanently blocking rediscovery once the show aired; now only true anomalies (null `media_type` with a non-Not-yet-aired status) get blacklisted. Anime with `title=None` are skipped silently too — MAL routinely leaves the romanization field null on freshly-announced donghua/Chinese shows and PV stubs (it usually fills in within hours); blacklisting them with a `<mal_id:NNNN>` placeholder would pollute `media_unwanted` and block rediscovery once MAL populates the title.
  - `job_worker.py` — single asyncio FIFO worker draining the `jobs` table. Per-user cap (`JOBS_PER_USER_LIMIT=4`); claims jobs via `with_for_update(skip_locked=True)`; `wakeup` Event for sub-second pickup with a 60s wall-clock fallback (shortens to 2s while `is_maintenance_active()` — restore lifts the gate without calling `notify()`, so the idle poll has to catch the transition itself). Dispatcher table maps `JobKind → handler` (registered from `main.py` lifespan). On failure, stamps `result_summary.retryable = not isinstance(failure, PermanentPhsarError)` so the bell knows whether to show a retry button. Also stamps `result_summary.error_category` via `_classify_error` for failure modes that have user-facing copy: `upstream_outage` for httpx 5xx + TimeoutException + NetworkError + `TransientUpstreamError` (MAL returning 200 with empty data on a known mal_id); `backup_disk_full` for `BackupDiskSpaceError`; `backup_corrupt` for `BackupIntegrityError` (pg_dump produced output that fails the `pg_restore --list` integrity check). The bell renders friendly text instead of the raw `Server error '504 Gateway Time-out' for url '...'` line or the pg_dump stderr tail. Anything not categorized falls through to the raw `error_message`, which is already friendly for the custom domain errors (`AnimeNotFoundError`, `MainMediaNotFoundError`, `MalIdAlreadyExistsError`, `AnimeFilteredOutError`). Sequential per-job sessions (claim-tx then work-tx in fresh sessions) avoid asyncpg pool deadlocks under nested-savepoint patterns. Sweep-kind jobs (`update_sweep`, `seasonal_sweep`) are bracketed in maintenance mode by the worker — `set_maintenance(True)` before dispatch, `set_maintenance(False)` + `set_scheduled_at(None)` in `finally` so a dispatcher crash can't leave the flag stuck behind a 503 until restart. `dispatch_one` also short-circuits via `is_maintenance_active()` BEFORE claiming any row, which prevents a queued `backup` job from slipping in during a restore — restore takes its pre-snapshot, releases `_BACKUP_WRITE_LOCK`, then flips maintenance + disposes the pool + `pg_terminate_backend`s every other session; without the early-return, the worker could claim the backup between the pre-snapshot's lock release and `pg_restore` starting, then either die mid-pg_dump on the terminated session (spurious `backup_corrupt`) or finish dumping against a half-dropped DB. The check is unconditional (all kinds) because any job during restore would have its session yanked out by `pg_terminate_backend` anyway. Sweep-induced maintenance is a no-op for the check: the worker is already running the sweep inside `dispatch_one`, so `_run` doesn't re-enter while the flag is set.
  - `scrape_dispatcher.py` — handlers for `user_scrape`, `update_sweep`, and `seasonal_sweep` jobs. `user_scrape` wraps the existing `handle_search_mal_api_results` + `save_search_results` pipeline behind a `ProgressReporter` so the bell sees mid-flight progress without waiting for the whole job tx; system-enqueued `user_scrape` children (from the seasonal sweep) carry `seed_mal_id` in their payload so each child's BFS skips the fuzzy `q=<title>` top-3 lookup and seeds directly off the known mal_id. `update_sweep` tier-selects due anime via `AnimeDAO.select_due_for_sweep`, then for each anime: **step 1** refreshes child media via `JikanScraper.refresh_anime` (`/anime/{id}/full`), diffs the volatile fields (`score`, `scored_by`, `episodes`, `airing_status`, `aired_to`) via `_apply_media_diff`, and updates `MediaFreshness`; **step 2** (only on tier 3 + tier 4 anime — `not currently_airing AND stable_check_count >= 3`) probes each Main media's relations via `JikanScraper.search_title(seed_mal_id=...)` and attaches discovered media to the existing parent anime via `save_service.attach_search_result_to_anime`, then advances `AnimeFreshness`. **Two commits per anime + per-anime try/except**: step 1 always commits durably (so a worker crash mid-sweep preserves the field-diff work, and a single bad MAL response fails *that* anime only without aborting the loop); step 2 only commits on probe success — a probe failure leaves `AnimeFreshness` untouched so the tier query re-selects the anime on the next sweep without losing the field-diff work. Per-Main-seed BFS (each main gets its own `visited_ids`) is what lets the probe reach disjoint sub-graphs — walking from BNHA's main alone wouldn't surface Vigilante S2 because Vigilante S1 is a side-story under BNHA and `search_title`'s BFS stops expanding side-stories once a main is found. Score and scored_by are bundled through `_weighted_score(score, scored_by) = score * log10(scored_by + 1)` (the same weighted formula used for search ranking) and only count as stability-resetting when `abs(new − old) >= _SCORE_STABILITY_THRESHOLD = 0.05` — without this, a single new vote per night on a million-vote anime would reset the counter forever and the row would never stabilize. None↔value transitions bypass the threshold (first votes coming in is structural). New values are written through to the DB regardless; the threshold gates only the stability *signal*. Score/scored_by writes are also gated on `payload["scored_by"]` being truthy — MAL returns scored_by as None or > 0 (never literally 0), and `extract_information` coerces None → 0 to satisfy the not-null insert column, so a 0 during a refresh means the field was omitted; clobbering a populated count with that would be silent data loss. Resets `stable_check_count` to 0 when any media is currently airing or any field changed (above threshold), else `min(prev + 1, 99)`. Coalesced `refresh_spoiler_cache_for_all_users` fires once at the end of the sweep if any new media landed — the probe path uses `save_service.attach_search_result_to_anime` (which never recomputes) instead of `save_search_results` (which always does), so a 200-anime sweep with 50 newly-discovered media triggers exactly one cache recompute instead of 50. `seasonal_sweep` paginates `JikanScraper.fetch_current_season` (`/seasons/now`), dedupes the page against `Anime.mal_id ∪ Media.mal_id ∪ MediaUnwanted.mal_id` (plus a per-run dedupe so MAL's occasional cross-page duplicate doesn't double-enqueue), and bulk-inserts one system `user_scrape` child per new mal_id with `requested_by_user_id=null` (system jobs skip the per-user cap). Bulk-insert instead of per-row commit because the enqueue loop does no MAL I/O — all the MAL work happens in the per-id children later — so the crash-safety argument that drives per-anime commits in `update_sweep` doesn't apply, and a shorter maintenance window matters more.
  - `backup_dispatcher.py` — handler for the `backup` JobKind. Wraps `backup_service.create_backup(source, label)`, reports progress through `Dumping` → `Applying retention` → `Done` stages, and stamps `result_summary = {filename, size_bytes, integrity, source, deduped_against}` so the bell can render `Backup ready (N MB)` or `Re-confirmed existing dump`. Catches `DuplicateBackupError` and surfaces it as a success outcome with `deduped_against` populated (the bell shows a softer "no new data" line rather than a 409-style fail). `apply_retention()` runs after every job — manual and cron share the same 14-recent + 8-Sunday + 1-known-good pool, so a manual-only install doesn't accumulate dumps indefinitely. **No maintenance bracket**: `pg_dump` runs on an MVCC snapshot, so user writes are safe during a dump window. `BackupDiskSpaceError` and `BackupIntegrityError` surface via `job_worker._classify_error` as `backup_disk_full` / `backup_corrupt` categories with friendly bell copy; both stay `retryable=True` so admin can free space or rerun.
  - `progress_reporter.py` — short-lived autocommit-tx writer that pushes `(stage, items_done, items_total)` updates to a job row. 0.5s throttle (with `force=True` bypass for stage transitions) keeps tight BFS loops from opening hundreds of sessions per second.
  - `search_service.py` — orchestrates the BFS output: takes one or more graphs from `JikanScraper.search_title`, runs DB-side dedupe against the catalog + `media_unwanted`, decides per-graph whether to save as a new anime, attach to an existing anime, or surface as a merge candidate, and packages the decision as `SaveAction | AttachToExistingAction` for the caller (save_service or the seasonal/sweep dispatcher) to execute. **`AttachToExistingAction`** routes orphan side-story graphs (single cross-link to an existing Media, no main of their own) through `save_service.attach_search_result_to_anime` — the same primitive the relations probe uses — so a tier-1/2 parent's new side-story gets attached under the right anime instead of becoming a duplicate Anime row. **`_pick_root_for_promotion`** is the seed-mode fallback when `get_first_main_relation` finds no main AND there's no single cross-link: it picks the most main-like graph entry by `(type_tier, aired_from)` where TV/Movie=0, ONA=1, OVA/TVSpecial=2, Special=3 (nulls last on aired_from) and promotes it to `relation_type="main"`. Necessary for donghua sub-universes whose canonical main is an ONA, and for the pilot-aired-before-the-real-show edge (tier > date). Cross-link signals are filtered to subtract `media_unwanted` mal_ids before the attach-vs-promote decision so filtered Music/PV entries don't get treated as franchise-overlap evidence. `alternative_setting` is treated as a graph boundary on par with `crossover` but is dropped entirely — alt-setting is MAL's explicit "different story, shared themes" label (Zhe Tian ↔ Wanmei Shijie, Madoka ↔ Magia Record) and conflating those into one Anime row triggered false-positive merge candidates on every sweep. All MAL relation strings are normalized once on entry via `_normalize_relation(rel) = rel.lower().replace(" ", "_")` so every downstream check works against the underscore form. Before v0.14.0 the comparisons used the underscore form but the lists held raw MAL strings (`"Parent story"`), so the `is_main_story` gate silently let every franchise movie classify as `main` instead of `side_story` — see [compound-docs/2026-05-11-jikan-scraper-quirks.md](compound-docs/2026-05-11-jikan-scraper-quirks.md). Fix applies to new scrapes only; pre-fix catalog rows keep stale `main` labels until the parent anime is re-scraped.
  - `vector_embedding_service.py` — sentence-transformers embeddings.
  - `media_search_service.py` / `anime_search_service.py` — filtered DB search. Anime variant uses a two-phase query (GROUP BY + HAVING for filter/order, then detail fetch) with majority-genre logic.
  - `filter_service.py` — filter option values; view-type-aware for anime vs media ranges.
  - `auth_service.py` — registration, authentication, token issuance, account deletion.
  - `user_settings_service.py` — user settings CRUD + default creation.
  - `token_service.py` — compressed JWT for shareable filter URLs.
  - `admin_service.py` — registration token list + delete.
  - `merge_detection_service.py` — duplicate-anime detector. Three signals feed the `merge_candidates` table: `title_studio` (SequenceMatcher ratio OR containment ≥ 0.85, gated by studio overlap; containment additionally requires a word-boundary on both sides of the match in the longer string AND a size floor of `MIN_CONTAINMENT_MATCH_CHARS=4` chars OR a full match where the shorter title is itself ≥ `MIN_FULL_MATCH_CHARS=3`), `title_desc` (weaker title match + description-embedding cosine ≥ 0.85), `relation_link` (BFS surfaced a non-crossover related media that lives under a different anime). Detection runs at the end of `save_search_results` (new × existing); a startup `backfill_merge_candidates` covers existing × existing. `seen_pairs` pre-fetch short-circuits flagged or admin-resolved pairs before any similarity computation.
  - `merge_candidate_service.py` — admin operations on merge candidates: list pending (ranked per-pair by `(earliest aired_from ASC nulls last, rating_count DESC, anime_id ASC)` so the recommended-keep side is surfaced as A; uses `MergeCandidateDAO.get_rating_counts_for_anime` for the tiebreak), dismiss (status flip), and merge (re-parents B's media onto A, deletes B via cascade, refreshes A's anime embedding, runs `detect_merge_candidates` against the survivor so freshly-valid pairs from the merged media get flagged, recomputes spoiler cache for all users). The merge accepts an optional `keep_uuid` to swap which side survives — the DB invariant `anime_a_id < anime_b_id` is unchanged, A/B is a presentation label only. Fail-loud on shared `Media.mal_id` between A and B since that's a global-unique-violation that needs human review.
  - `rating_service.py` — rating CRUD + note search; triggers a spoiler-visibility recompute on every change.
  - `spoiler_service.py` — frontier algorithm + precomputed visibility cache in `user_visible_media`; recomputed per-anime on rating changes, per-user on registration, and per-user after new scrapes land via `save_service`.
  - `export_service.py` — flat media-level export merging catalog + ratings + watchlist; respects user's `name_language` for localized title columns.
  - `backup_service.py` — pg_dump/pg_restore orchestration: atomic `.partial` writes, sidecar `.meta.json`, content-hash dedupe, retention (14 daily + 8 Sunday; never evicts the most recent known-good dump OR the `is_current` dump — the latter exemption prevents a stack of dedupe-hit re-confirms from pushing the pointed-at older dump out of the 14-recent window, which would delete the very file the bell labels as "matches live DB"; manual + cron share the pool, uploads + pre_restore are admin-managed and retention-exempt), pre-restore auto-snapshot. `.current_db.json` pointer tracks which dump matches the live DB: set by `restore_backup` (→ restored dump) and by every successful `create_backup` regardless of source — manual / cron / pre_restore, whether the dump was unique or a dedupe hit. Uploads NEVER move the pointer (`save_uploaded_backup` only stamps `is_current` on the returned metadata when the new filename happens to equal the unchanged pointer) — uploaded bytes are external and we can't verify they match live. Deleting the pointed dump clears the pointer. All write paths serialized via a module-level `asyncio.Lock` (single-worker assumption). Restore flips a maintenance flag so other requests 503 during the window; backup creation does NOT (MVCC-snapshot read-only). Subprocess password via `PGPASSWORD`, never on the CLI. Deeper design notes: [compound-docs/2026-04-19-v0.13.0-deployment.md](compound-docs/2026-04-19-v0.13.0-deployment.md).
- **daos/** — Data access layer. `BaseDAO` provides generic async CRUD; specialized DAOs (media, anime, genre, studio, user, user_settings, registration_token, rating, job, merge_candidate) add domain-specific queries with vector similarity, filtering, and aggregation. `AnimeDAO.search_anime_aggregated` uses two-phase query: SQL GROUP BY with HAVING for filtering/ordering, then detail fetch. `AnimeDAO.select_due_for_sweep` runs the four-tier sweep selection (airing now / still stabilizing / weekly recent main / 180-day long tail) backed by three indexes: `ix_anime_freshness_last_checked_at` (the ORDER BY), `ix_media_airing_now` (partial index on `media(anime_id) WHERE airing_status = 'Currently Airing'`), and `ix_media_main_aired_from` (composite `(anime_id, relation_type, aired_from)` for the recent-main EXISTS). `search_filters.py` provides shared filter/ordering helpers for media, anime pre-aggregation (WHERE), and anime post-aggregation (HAVING) filters. `JobDAO` provides claim-skip-locked dispatch + crash recovery: `reap_orphans` runs at startup and flips any `running` rows to `failed` so a mid-job process restart doesn't leave them stuck.
- **models/** — SQLAlchemy ORM models mapped to PostgreSQL tables. `media_search.py` stores pgvector embeddings for title and description; `anime_search.py` stores anime-level embeddings; `rating_search.py` stores note embeddings for rating note search. `user_settings.py` stores per-user preferences (one-to-one with Users) with enums for theme, name language, search view, rating step, and spoiler level. `user_visible_media.py` is a precomputed cache of which media are visible (not spoiler-protected) per user, updated on rating changes. `merge_candidate.py` stores admin-reviewable duplicate pairs with status enum (`pending`/`dismissed`/`merged`); FK cascades on both anime ids so merging A into B (or deleting either anime) cleans up dangling rows. Unique constraint + check on `(anime_a_id, anime_b_id)` with caller-enforced ascending order so `(A,B)` and `(B,A)` collapse. `job.py` stores the queue of background jobs with `JobKind` enum (`user_scrape`, `update_sweep`, `seasonal_sweep`, `backup`), `JobStatus` enum (`queued`/`running`/`succeeded`/`failed`), JSONB payload, progress fields (`stage`, `items_done`, `items_total`), `not_before_at` for scheduled-delay jobs, and `result_summary` JSONB (carries `retryable: bool` for the bell); partial composite index on `(created_at) WHERE status='queued'` keeps the worker's FIFO claim cheap regardless of finished-row volume. `anime_freshness.py` / `media_freshness.py` are 1:1 sidecars (`unique=True` on the FK column) for the nightly update sweep — they hold `last_checked_at` (and `stable_check_count` on the anime side) outside the canonical catalog rows so sweep cadence can't leak into Pydantic schemas via `model_dump()`. Existing rows are backfilled to the parent's `created_at` so the sweep enters them at honest age, not "never checked."
- **schemas/** — Pydantic request/response DTOs.
- **core/** — Config (`config.py` loads from `.env`), database engine (`db.py`), auth dependencies (`dependencies.py` — JWT user dep + `require_roles()` factory + a generic `require_bearer_token(settings_attr, error_factory)` factory used by both `BACKUP_CRON_TOKEN` and `JOBS_CRON_TOKEN` cron-authed endpoints), JWT/password security (`security.py`), process-wide maintenance state (`maintenance.py`: `_active` flag flipped by destructive ops + `_scheduled_at` future-window pointer set by the schedule-sweep cron endpoint when it enqueues a delayed sweep; both are module globals, single-process assumption).
- **seeders/** — Run at app startup via lifespan; seed genres, admin user, and optional guest user (restricted_user role). `backfill_user_settings` ensures all users have a UserSettings row. `backfill_spoiler_visibility` computes visible media for users with **zero** cache rows (new deployments, pre-feature users) — it does **not** mop up partial-cache drift on existing users. `embedding_backfiller.py` detects and regenerates any missing anime, media, or rating embeddings (enables seamless embedding model swaps via Alembic migration + restart). `merge_detection_service.backfill_merge_candidates` does a one-shot existing × existing pair sweep so duplicates that pre-date the detector get flagged on first restart after upgrade. `save_service.save_search_results` triggers a full-user spoiler-cache recompute after new animes land — load-bearing for existing `spoiler_level=hide` users, whose populated `user_visible_media` cache otherwise stays stale (the startup `backfill_spoiler_visibility` skips them since they have rows). The sweep dispatcher's relations probe goes through `save_service.attach_search_result_to_anime` instead, which intentionally does *not* recompute per-batch; the dispatcher fires one coalesced `refresh_spoiler_cache_for_all_users` at sweep end regardless of how many new media landed.
- **exceptions.py** — Custom exception hierarchy rooted at `PhsarBaseError` with `status_code` attribute. Single exception handler in `main.py` reads the status code from each exception class. `PermanentPhsarError` is a marker subclass for failures where retry won't help (`AnimeNotFoundError`, `MainMediaNotFoundError`, `MalIdAlreadyExistsError`, `AnimeFilteredOutError`); `JobWorker` reads the marker via `isinstance` and stamps `result_summary.retryable = False` on the failed job so the bell hides its retry button instead of letting users spam jobs that can't ever succeed. `TransientUpstreamError` sits outside the permanent branch — raised when MAL returns a 200 OK with empty data on an otherwise-valid mal_id, so the bell keeps the retry button enabled and `_classify_error` tags it as `upstream_outage` for friendly copy. `AnimeFilteredOutError` (sibling to `AnimeNotFoundError` under the permanent branch) surfaces honestly when a seeded BFS returns nothing AND the seed mal_id lives in `media_unwanted` — admin sees `"'X' was filtered out as Music"` instead of a misleading "not found".
- **main.py** — App factory + lifespan (seeders → embedding backfill → merge-candidate backfill → `JobDAO.reap_orphans` flips any `running` jobs to `failed` so a mid-job restart doesn't leave them stuck → `job_worker.register_dispatcher` is called for each of `user_scrape`, `update_sweep`, `seasonal_sweep`, and `backup` then `job_worker.start()`; on shutdown, `job_worker.stop()`). Also exposes `GET /` and `GET /health` directly (health returns `{status, version, db}` and pings the DB with a short 2s `asyncio.wait_for` so transient DB unavailability produces a fast 503 instead of riding asyncpg's 60s connect timeout into a Coolify liveness flap). Registers `MaintenanceGateMiddleware` (lives in `core/maintenance_middleware.py`; pure ASGI class, *not* `@app.middleware("http")` — the latter's BaseHTTPMiddleware short-circuits don't compose cleanly with `CORSMiddleware`'s send wrapper) that short-circuits every request not in the allowlist (`/`, `/health`, `/maintenance/status`, `/admin/jobs/schedule-sweep`, `/admin/jobs/schedule-seasonal`) to 503 `{maintenance: true}` while `core/maintenance.py`'s flag is set (flipped on by restore, or by the worker for the duration of an `update_sweep` / `seasonal_sweep` job). **Critical middleware ordering invariant:** `app.add_middleware()` *prepends* to `user_middleware` (`insert(0, ...)`), so the LAST registered middleware ends up OUTERMOST in the final stack. Register `MaintenanceGateMiddleware` FIRST, then `CORSMiddleware` — that way CORS wraps maintenance and the 503 short-circuit still gets stamped with `Access-Control-Allow-Origin`. Reversing the order silently drops CORS headers on the 503, browsers reject cross-origin responses without them, and `fetch()` rejects with TypeError instead of a Response — surfacing on the frontend as a generic "An unexpected error occurred." rather than the maintenance banner.

### Key Patterns

- **Dependency injection**: FastAPI `Depends()` for DB sessions, current user extraction, role-based access (`require_roles()` accepts `RoleType` enum).
- **Role-based access**: Three roles — `admin`, `user`, `restricted_user`.
- **Async throughout**: asyncpg driver, SQLAlchemy AsyncSession, async service/DAO methods. All ORM relationships use `lazy="raise"` to prevent implicit lazy loading — every relationship access must go through explicit `selectinload` in the DAO query.
- **Vector search**: `paraphrase-multilingual-MiniLM-L12-v2` model generates embeddings stored via pgvector; similarity search on title, description, and rating note vectors. `SearchType` enum (`title`, `description`, `rating_notes`) selects the target. Search filter schemas use inheritance: `MediaSearchFilters` (base) → `RatingSearchFilters` (adds rating-specific filters). Anime-level title search uses `AnimeSearch` embeddings directly; description search averages cosine distances across media. `ViewType` enum (`anime`, `media`) selects the search mode. `/filters/options?view_type=anime` returns anime-appropriate filter ranges (aggregated episodes/watch time, majority genres).
- **Domain exceptions**: All custom exceptions extend `PhsarBaseError` with a `status_code` class attribute. One handler in `main.py` serves all.
- **Theme system**: CSS custom properties with `@property` indirection — `@property --primary` / `--ring` hold the source values, `@theme inline` references them via `var()`, and `.theme-*` classes on `<html>` override them. This forces Tailwind to emit `var()` in utilities instead of inlining static values. Components use semantic tokens (`bg-primary`, `text-primary`, `ring-ring`). Centralized theme config in `lib/themes.ts` maps theme keys to CSS classes, character pics, and labels. FOUC prevention via inline localStorage script in `app.html`. Per-theme chart color palettes in `chartColors.ts` avoid hue clashes.
- **Spoiler protection**: Three levels (`off`/`blur`/`hide`) controlled by `SpoilerLevel` user setting. Uses a "frontier" algorithm: per anime, all media up to and including the next unwatched main-story entry are visible. Precomputed in `user_visible_media` table (updated per-anime on rating changes, per-user on registration, and per-user after new scrapes land via `save_service`). Backend `GET /ratings/spoiler-visibility` returns visible UUIDs; frontend stores as `Set` in `spoilerVisibility` store. `SpoilerGuard.svelte` wraps covers/descriptions with blur + click-to-reveal. Detail pages compute frontier locally for fresher data. `hide` mode in media search uses `WHERE media.id IN (...)` from the cache. Anime covers/descriptions are never spoiler-protected.
- **Background jobs**: A single asyncio FIFO worker (`job_worker.py`) drains the `jobs` table. JobKinds: `user_scrape` (admin or user-submitted scrape from `/library/add`), `update_sweep` + `seasonal_sweep` (system cron jobs that bracket maintenance), `backup` (admin or cron — does NOT bracket maintenance because `pg_dump` is MVCC-snapshot read-only). Per-user cap (`JOBS_PER_USER_LIMIT=4`) prevents one user from monopolizing the queue; system jobs submit with `requested_by_user_id=null` to skip the cap. Each handler accepts a `ProgressReporter` so `JobBell` sees mid-flight stage/items updates before the whole job tx commits (short-lived autocommit txes throttled to 0.5s). `PermanentPhsarError` marker class gates retry: deterministic failures (`AnimeNotFoundError`, `MainMediaNotFoundError`, `MalIdAlreadyExistsError`, `AnimeFilteredOutError`) stamp `result_summary.retryable = False` and the bell hides the retry button. `TransientUpstreamError` stays outside the marker so MAL's occasional empty-200 hiccup keeps a retryable bell entry. Per-query dedupe lives at submission time: re-running the same normalized query within `JOBS_DEDUPE_HOURS` returns 409 (failed jobs don't count, so a transient MAL outage doesn't lock a user out). Crash recovery: `JobDAO.reap_orphans` runs on lifespan startup and marks any `running` rows as `failed` (mirrors `embedding_backfiller`'s idempotency pattern). **Bell cadence**: 2s active poll while anything is queued/running, 30s idle. **Optimistic-stub pattern**: pages that enqueue a job push a `queued` stub keyed on the returned `job_uuid` into the `optimisticJobs` store so the bell renders it before the next `/jobs/mine` round-trip; the bell merges optimistic ∪ fetched (UUID-deduped) and calls `reconcileOptimisticJobs(fresh)` after every fetch to prune entries that have landed.
- **Maintenance mode**: Destructive operations flip `core/maintenance.py`'s in-memory `_active` flag — currently backup restore (synchronous, request-scoped) and nightly `update_sweep`/`seasonal_sweep` jobs (the worker brackets each sweep dispatch in a try/finally so a crash can't leave the flag stuck). `MaintenanceGateMiddleware` (a pure ASGI class in `core/maintenance_middleware.py`) returns 503 `{maintenance: true}` for every non-allowlisted request while `_active` is set; the allowlist is `/`, `/health`, `/maintenance/status`, `/admin/jobs/schedule-sweep`, `/admin/jobs/schedule-seasonal`. **Middleware ordering is load-bearing:** `MaintenanceGateMiddleware` must be registered BEFORE `CORSMiddleware` so that CORS ends up outermost in the final stack — `app.add_middleware()` prepends, so registration order reverses the apparent order. Get this wrong and the 503 short-circuit ships with no `Access-Control-Allow-Origin` header; the browser blocks the cross-origin response, `fetch()` throws TypeError instead of resolving, and the frontend's catch falls into the generic "An unexpected error occurred." branch instead of the maintenance banner. Pure ASGI (vs `@app.middleware("http")`/BaseHTTPMiddleware) is also load-bearing: BaseHTTPMiddleware short-circuits don't compose cleanly with CORSMiddleware's send wrapper. The frontend's `api.ts` detects 503-with-maintenance, clears the auth token, calls `bumpMaintenanceRefresh()` so the global banner refetches state in ms (covers the user-on-/login case where `token.set(null)` is a no-op transition), and hard-navigates to `/login`. Pre-warning protocol: the schedule-sweep cron endpoint sets `_scheduled_at` to a future timestamp when it enqueues a delayed sweep (e.g. 20 min ahead); `MaintenanceBanner` polls `GET /maintenance/status` every 30s and renders a yellow countdown when the schedule is within 30 min so users can save work before the window opens. Both `_active` and `_scheduled_at` are process-wide module globals — single-worker assumption, documented upgrade path is a file sentinel or DB row.
- **CORS**: Backend allows origins from `settings.CORS_ORIGINS` (defaults to `http://localhost:5173`) and exposes `Content-Disposition` so the frontend can read server-authored download filenames.

Frontend (runtime, read from `$env/dynamic/public`): `PUBLIC_API_BASE_URL` (backend URL; defaults to `http://localhost:8000` for local dev) and `PUBLIC_APP_VERSION` (shown in the footer). Both are set as container ENV in production — changing them does not require an image rebuild.

## Deployment

Self-hosted on a Coolify-managed VM. Images are built in GitHub Actions and pulled by Coolify — the VM is too small (2 vCPU / 4 GB) to survive a SvelteKit build.

Three services:

- `phsar/Dockerfile` — multi-stage backend. CPU-only torch from the pytorch CPU index; sentence-transformers model baked into `/opt/st-cache`. Runs as non-root `phsar` user (UID 1000). `/backups` is created at build time and chowned to `phsar:phsar` so a bind-mounted host dir matches. Entrypoint (`phsar/docker/entrypoint.sh`) applies Alembic migrations before exec'ing uvicorn.
- `phsar/frontend/Dockerfile` — bun build → `node:22-slim` runtime via SvelteKit `adapter-node`. Reuses the built-in `node` user.
- `docker-compose.yml` (repo root) — runs all three containers together; only for local parity smoke-testing, not day-to-day dev.

Deployment flow: tag any commit with `v*` (stable `v0.13.0` or preview `v0.13.0-rc1`) and push. The `build-images.yml` workflow builds both images in parallel and pushes to `ghcr.io/priapos1004/phsar-{backend,frontend}:<tag>`. In Coolify, point each service at the new image tag and redeploy.

### Backups

- Coolify mounts `/opt/phsar/backups` on the VM to `/backups` in the backend container. The host directory must be owned by UID 1000 (`chown 1000:1000 /opt/phsar/backups`) so the `phsar` user can write dumps.
- Manual backups: admin panel → Backups card → "Create backup". Both manual and cron POSTs are async — the endpoint enqueues a `backup` job (202 with `job_uuid`) and the worker runs `pg_dump` in the background. The bell tracks progress (`Dumping` → `Applying retention` → `Done`); on success the BackupsCard auto-refreshes the dump list. The "Create backup" button debounces for 5 s to absorb double-clicks. Download/delete/upload/restore from the same UI; restore stays synchronous (it's meant to be blocking).
- Scheduled backups: set `BACKUP_CRON_TOKEN` in Coolify backend env, then add a Coolify scheduled task that `curl -fsS -X POST -H "Authorization: Bearer $BACKUP_CRON_TOKEN" http://localhost:8000/admin/backups/auto` (task runs inside the backend container, which ships with curl, so loopback avoids TLS/DNS). The endpoint enqueues a `cron` backup job; the dispatcher creates the dump and applies retention. Retention now runs after **every** backup job (manual and cron share the 14-recent + 8-Sunday + 1-known-good pool), so a manual-only install no longer accumulates dumps indefinitely when cron is disabled.
- Off-host safety net: `scripts/pull-backups.sh user@vm` rsyncs `/opt/phsar/backups/` to the local machine. Run every ~2 months and before any restore.
- Restore workflow: UI prompts for the admin's username as the confirmation string; an automatic pre-restore snapshot is taken first, then `pg_restore --clean --if-exists --no-owner --no-privileges` runs against the target DB.

### Scheduled sweeps

Both sweep schedulers are cron-token-authed (`JOBS_CRON_TOKEN` in Coolify backend env — empty disables both endpoints and they fail closed) and enqueue a delayed job so `MaintenanceBanner`'s countdown can warn users before the maintenance window opens. `delay_minutes` is bound to `Query(20, ge=0, le=1440)`.

- **Nightly catalog refresh** — `POST /admin/jobs/schedule-sweep?delay_minutes=N` enqueues an `update_sweep` job (per-anime field diff + tier-3/4 relations probe). Recommended Coolify scheduled task (daily, ~02:40 UTC):
  ```
  curl -fsS -X POST -H "Authorization: Bearer $JOBS_CRON_TOKEN" \
    "http://localhost:8000/admin/jobs/schedule-sweep?delay_minutes=20"
  ```
- **Weekly seasonal pickup** — `POST /admin/jobs/schedule-seasonal?delay_minutes=N` enqueues a `seasonal_sweep` job (paginates `/seasons/now`, enqueues one system `user_scrape` child per new mal_id). Recommended Coolify scheduled task (weekly Sunday, ~02:30 UTC):
  ```
  curl -fsS -X POST -H "Authorization: Bearer $JOBS_CRON_TOKEN" \
    "http://localhost:8000/admin/jobs/schedule-seasonal?delay_minutes=20"
  ```

Both endpoints are allowlisted through the maintenance gate so a cron retry that lands mid-sweep still authenticates and gets rejected via the per-kind dedupe instead of 503'd.
